# Sprint 3 串行审计门禁执行模板

- 版本：V1.0
- 更新时间：2026-04-16
- 作者：技术架构师
- 状态：待审计
- 适用范围：Sprint 3 全部任务、补审任务、平台治理任务、生产联调任务

## 一、模板目标

本模板用于约束 Sprint 3 所有任务必须执行串行审计门禁，避免出现“设计未审计即实现、实现未审计即提交、提交未审计即进入下一任务”的流程缺口。

## 二、串行审计总流程

```text
任务单下发
  -> 设计冻结
  -> 设计审计
  -> 实现执行
  -> 实现审计
  -> 本地基线提交
  -> 提交审计
  -> 下一任务
```

任何任务不得跳过其中任一环节。若某任务为纯文档任务，可将“实现执行”定义为“文档输出”，但仍必须经过审计。

## 三、模板编号规则

### 3.1 三阶段编号

| 阶段 | 编号形式 | 示例 | 说明 |
|---|---|---|---|
| 设计冻结 | `TASK-XXXA` | `TASK-016A` | 输出设计文档与工程任务单 |
| 实现执行 | `TASK-XXXB` | `TASK-016B` | 按设计实现或输出执行产物 |
| 本地提交 | `TASK-XXXC` | `TASK-016C` | 白名单提交与基线锚点 |
| 封版审计 | `TASK-XXXD` | `TASK-016D` | 对 A~C 做全链路审计 |

### 3.2 整改编号

| 场景 | 编号形式 | 示例 |
|---|---|---|
| 设计审计不通过 | `TASK-XXXA1` | `TASK-016A1` |
| 实现审计不通过 | `TASK-XXXB1` | `TASK-016B1` |
| 提交审计不通过 | `TASK-XXXC1` | `TASK-016C1` |
| 多轮整改 | 继续递增 | `TASK-016B2`、`TASK-016B3` |

### 3.3 补审编号

Sprint 2 补审任务使用 `B-1` ~ `B-6`。若补审发现问题，整改编号采用：

```text
B-1A / B-1B / B-1C ...
```

其中：
- `B-1A` 表示第一次整改。
- `B-1B` 表示第二次整改。
- 补审整改不得混入新功能。

## 四、审计意见书格式

每份审计意见书必须包含以下字段：

```text
【审计意见书】第 N 份
模块：[任务编号 + 模块名]
审计类型：[设计审计 / 实现审计 / 提交审计 / 补审 / 复审]
审计时间：YYYY-MM-DD HH:MM
审计结论：[通过 / 有条件通过 / 不通过]

一、审计范围
- 文件/目录清单
- commit SHA（如适用）
- 前置依赖

二、Findings
- 高危 P1：数量与明细
- 中危 P2：数量与明细
- 低危 P3：数量与明细

三、验证命令
- 命令
- 结果

四、边界确认
- 是否修改前端
- 是否修改后端
- 是否修改 .github
- 是否修改 02_源码
- 是否 push / remote / PR
- 是否声明生产发布

五、结论
- 是否允许进入下一任务
- 若不允许，指定整改任务编号
```

## 五、Finding 格式

每个 finding 必须包含：

| 字段 | 要求 |
|---|---|
| 严重级别 | P1 / P2 / P3 |
| 标题 | 一句话说明问题 |
| 文件位置 | 精确到文件与行号 |
| 问题说明 | 为什么有风险 |
| 影响范围 | 会影响哪些业务或安全边界 |
| 修复建议 | 下一张整改任务必须做什么 |
| 复验要求 | 修复后必须如何验证 |

P1/P2 不允许只记录不整改。P3 可以记录为残余风险，但必须说明后续处理建议。

## 六、审计前置检查清单

### 6.1 设计审计前置

□ 设计文档已输出  
□ 工程任务单已输出  
□ 前置依赖明确  
□ 允许范围与禁止范围明确  
□ 权限动作明确  
□ 资源字段明确  
□ ERPNext fail-closed 口径明确  
□ Outbox 状态机口径明确（如涉及）  
□ 前端门禁口径明确（如涉及）  
□ 验收标准可检查  
□ 未修改业务代码  

### 6.2 实现审计前置

□ 实现文件清单完整  
□ 测试文件清单完整  
□ 交付证据已输出  
□ 定向测试已执行  
□ 全量或相关回归已执行  
□ `py_compile` / `typecheck` / `build` 已按任务要求执行  
□ 禁改路径扫描通过  
□ 暂存区为空  
□ 未 push / remote / PR  

### 6.3 提交审计前置

□ 实现审计已通过  
□ 白名单暂存  
□ `git diff --cached --name-only` 已回显  
□ `git diff --cached --check` 通过  
□ 提交文件与任务范围一致  
□ 未提交运行产物  
□ 未提交禁改路径  
□ commit SHA 已记录  
□ 暂存区为空  

## 七、整改闭环要求

1. P1 必须整改，未整改不得进入下一任务。
2. P2 原则上必须整改；若暂缓，必须由总调度确认并记录风险。
3. P3 可记录为风险，但不得影响生产安全边界。
4. 整改任务必须只修复审计指出的问题。
5. 整改任务不得夹带新功能。
6. 整改完成后必须复审。
7. 复审通过后才允许继续下一任务。

## 八、安全关键路径加严规则

以下任务必须执行设计审计、实现审计、提交审计三段完整门禁：

| 安全关键路径 | 示例 |
|---|---|
| 权限与资源隔离 | Role / User Permission / resource scope |
| ERPNext 写入 | Payment Entry / Stock Entry / Purchase Invoice |
| 财务影响 | GL / AR / AP / 对账 / 付款 |
| 库存影响 | Stock Ledger / Stock Entry / 盘点 |
| Outbox Worker | claim / lease / retry / dead |
| 前端写入口 | create / update / confirm / cancel / submit |
| 敏感审计 | Authorization / Cookie / Token / Secret 脱敏 |

## 九、禁止事项

- 禁止无设计审计直接实现。
- 禁止无实现审计直接提交。
- 禁止无提交审计直接进入下一任务。
- 禁止 `git add .`。
- 禁止 `git add -A`。
- 禁止 force push。
- 禁止把本地通过声明为平台通过。
- 禁止把本地封版声明为生产发布。

## 十、通过判定

任务通过必须同时满足：

1. 审计结论为通过。
2. 高危问题数为 0。
3. 中危问题数 ≤ 1，且已记录处理方式。
4. 禁改边界未破坏。
5. 未发生未经批准的 push / remote / PR。
6. 未声明生产发布完成。
