## 二、设计冻结任务模板

设计冻结任务用于输出架构设计、接口边界、数据模型、权限/审计/ERPNext/outbox/前端门禁口径。

必须包含：

1. 业务目标。
2. 系统边界。
3. 数据模型草案。
4. API 草案。
5. 权限动作。
6. 资源权限字段。
7. ERPNext fail-closed 策略。
8. Outbox 策略（如涉及）。
9. 前端门禁策略（如涉及）。
10. 审计验收标准。

禁止：

- 禁止写业务代码。
- 禁止修改前端/后端/.github/02_源码。
- 禁止声明生产发布完成。

## 三、工程实现任务模板

工程实现任务必须基于已审计通过的设计文档。

必须包含：

1. 允许修改文件白名单。
2. 禁止修改文件清单。
3. API / model / service / router / frontend / workflow 范围。
4. 测试要求。
5. 安全审计要求。
6. 操作审计要求。
7. 错误信封要求。
8. 禁改扫描要求。

禁止：

- 禁止超出设计范围。
- 禁止夹带新功能。
- 禁止无测试交付。
- 禁止未审计提交。

## 四、本地基线提交任务模板

本地提交任务必须在实现审计通过后下发。

必须包含：

1. 允许提交文件白名单。
2. 禁止提交清单。
3. `git diff --cached --name-only` 回显要求。
4. `git diff --cached --check` 要求。
5. 禁入路径 grep 检查。
6. commit message。
7. 提交后暂存区为空。

禁止：

- 禁止 `git add .`。
- 禁止 `git add -A`。
- 禁止提交运行产物。
- 禁止 push。
- 禁止 remote。
- 禁止 PR。

## 五、补审任务单格式

补审任务用于 Sprint 2 缺口补审或历史基线逆向审计。

```text
【补审任务卡】B-N [模块名] 补审
角色：工程师
优先级：[P1 / P2]
前置依赖：[上一补审任务审计通过]
补审范围：
- 设计文档
- 实现代码
- 测试
- 提交证据

审计要点：
1. [检查项 1]
2. [检查项 2]
3. [检查项 3]
4. [检查项 4]
5. [检查项 5]

通过标准：
- 高危 = 0
- 中危 ≤ 1
- P1/P2 必须整改复审

完成后回复：
B-N 执行完成。
检查项1：✅/❌（说明）
检查项2：✅/❌（说明）
检查项3：✅/❌（说明）
检查项4：✅/❌（说明）
检查项5：✅/❌（说明）
结论：提交审计 / 需整改
```

## 六、整改任务单格式

整改任务必须继承原编号并追加后缀。

示例：

| 原任务 | 整改任务 |
|---|---|
| TASK-016B | TASK-016B1 |
| TASK-016B1 | TASK-016B2 |
| B-1 | B-1A |
| B-1A | B-1B |

整改任务必须包含：

1. 原审计意见书编号。
2. 原 finding 摘要。
3. 允许修复范围。
4. 禁止夹带范围。
5. 复验命令。
6. 完成后交付证据。

## 七、审计任务单格式

```text
【审计任务卡】[任务编号] 审计
角色：审计官
前置依赖：[任务执行完成]
审计范围：
- [文件]
- [commit]
- [证据]

重点审计项：
1. [审计项 1]
2. [审计项 2]
3. [审计项 3]

必须复跑命令：
```bash
[命令]
```

通过标准：
□ 高危 0
□ 中危 ≤ 1
□ 禁改边界通过
□ 暂存区为空
□ 未 push / remote / PR

如通过，回复：
[任务编号] 审计通过。

如不通过，直接列 finding，并要求进入 [整改编号]。
```

## 八、任务卡强制字段

| 涉及文件 | 必填 |
| 审计前置要求 | 必填 |
| 验收标准 | 必填 |
| 验证命令 | 必填 |
| 完成后回复格式 | 必填 |

## 九、任务卡禁止语义

任务卡不得出现以下未完成态声明：

- 生产发布完成。
- GitHub required check 已闭环。
- Hosted Runner 已通过。
- ERPNext 生产联调完成。
- 已 push 到远端。
- PR 已创建。

除非对应平台证据已由审计官确认。

## 十、模板使用要求

1. 架构师下发任务必须使用本模板。
2. 工程师不得自行扩大任务范围。
3. 审计官不得以口头确认替代审计意见书。
4. 任何跳步都必须记录为流程缺口。
5. 所有任务完成后必须更新对应日志。
