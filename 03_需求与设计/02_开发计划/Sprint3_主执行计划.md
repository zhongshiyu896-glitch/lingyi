## 二、五轨道并行执行规则

```
轨道 A：Sprint 2 补审（B-1~B-6） ──────────────────────┐
轨道 B：ERPNext 联调（015B 完成后 015C） ─────────────┤
轨道 C：P1 设计冻结审计（021A/022A/023A） ────────────┤─→ 五轨道独立推进，互不阻塞
轨道 D：平台 CI（等待管理员） ─────────────────────────┤
轨道 E：P1 模块实现（P1 设计审计通过后解锁） ──────────┘
```

### 执行原则

1. **轨道 A / B / C / D 独立推进**，任何一个轨道完成不等待其他轨道。
2. **轨道 D（管理员依赖）**：不阻塞 A/B/C，随时就绪随时补齐。
3. **轨道 E（P1 实现）**：等待对应模块的设计审计通过后解锁。
4. **每个任务必须串行审计**：设计 → 审计通过 → 实现 → 审计通过 → 提交。
5. **发现高危问题**：立即停点 → 下发整改任务单 → 复审通过后继续。

## 九、CODEEX 三角色自主执行指南

### 架构师自主执行规则

```
IF 当前任务的前置依赖已满足 THEN
  执行当前任务，输出设计文档，提交总调度转审计官
ELSE
  等待依赖满足；如超过 3 天未满足，升级总调度
END
```

**可自主决定**：追加 P1 设计任务 / 拆分设计文档结构 / 调整禁止事项清单
**必须上报**：发现设计依赖缺失 / 发现新的安全关键路径未覆盖 / 发现与 ERPNext 集成策略冲突

### 工程师自主执行规则

```
IF 当前任务的前置审计已通过 AND 任务卡已收到 THEN
  执行任务，输出执行报告，提交总调度转审计官
ELSE
  等待前置条件满足
END
```

**可自主决定**：修复审计 finding 清单范围内的安全问题 / 补充测试用例 / 补充工程细节
**必须上报**：实现无法满足设计要求 / 需要新增文件超出任务范围 / ERPNext API 行为与设计不符

### 审计官自主执行规则

```
IF 待审材料齐全 THEN
  按审计要点逐项检查，输出审计意见书
  判断：通过 / 需整改 / 高危冻结，提交总调度
ELSE
  要求总调度补充材料，不出具意见书
END
```

**通过标准**：高危 = 0 → 通过；中危 ≤ 1 → 通过（记录）；高危 > 0 → 冻结；中危 > 1 → 整改
**必须上报**：发现系统性风险 / 发现 CODEEX 执行纪律问题

## 十三、禁止事项（全员遵守）

1. 禁止跳过审计流程。任何任务未审计不得进入实现阶段。
2. 禁止伪造证据。平台 CI 证据、审计结论、交付报告必须真实。
3. 禁止未接入公共基座。新模块实现必须引用 TASK-007/008/009/010。
4. 禁止 ERPNext 直连写接口。所有写入必须通过 adapter/outbox。
5. 禁止前端绕过 FastAPI。业务请求不得直连 ERPNext 或 PostgreSQL。
6. 禁止 200 + 空数据伪成功。所有 API 失败必须返回非 0 code。
7. 禁止 unknown config 静默跳过。未知配置必须 fail closed。
8. 禁止声明未完成的生产发布状态。
9. 禁止 force push / git reset --hard（除非总调度明确批准）。
10. 禁止 `git add .` 或 `git add -A`。必须白名单逐文件添加。
