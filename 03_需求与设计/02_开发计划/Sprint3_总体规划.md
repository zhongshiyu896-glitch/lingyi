## 一、Sprint 3 目标与边界

Sprint 3 的目标不是立即扩大业务功能，而是先修复 Sprint 2 暴露出的流程性风险：审计流程未闭环、平台 CI 未完成、生产 ERPNext 联调未验证、部分安全关键路径缺少生产级证据。

Sprint 3 采用“先治理、后扩展”的策略：先完成审计闭环、平台门禁、生产联调和安全关键路径前置规范，再逐步推进财务、采购、仓库、报表等 P1/P2 模块。

## 四、Sprint 3 与 Sprint 2 的依赖关系

Sprint 3 必须继承 Sprint 2 的四个公共基座：

1. TASK-007 权限与审计统一基座。
2. TASK-008 ERPNext Fail-Closed Adapter。
3. TASK-009 Outbox 公共状态机规范。
4. TASK-010 前端写入口门禁公共框架。

但 Sprint 2 状态已被标注为“本地封版完成 + 审计流程未闭环”。因此 Sprint 3 的任何生产发布、远端推送、ERPNext 生产联调和 required check 配置，均必须先完成 Sprint 2 缺口处理或明确标注风险。

## 五、Sprint 3 禁止事项清单

1. 禁止跳过设计审计直接写代码。
2. 禁止代码未审计直接提交。
3. 禁止安全关键路径未审计进入实现。
4. 禁止未完成 GitHub required check 就声明平台闭环。
5. 禁止未完成 ERPNext 生产联调就声明可发布。
6. 禁止使用 `git reset --hard` 回退历史，除非总调度明确批准。
7. 禁止 force push。
8. 禁止前端直连 ERPNext。
9. 禁止 200 + 空数据伪成功。
10. 禁止未接入权限、审计、错误信封的写接口。

## 六、Sprint 2 遗留问题处理方案

### 6.1 TASK-008~012 未审计代码

按 Sprint2 状态标注，TASK-008~012 存在审计流程未闭环问题。处理策略采用路径 C：标注现状，继续 Sprint 3，但生产发布前必须补审计。

### 6.2 生产发布前补审计路径

1. 审计官按 TASK-008~012 的真实代码和文档逐项补审。
2. 每个模块输出审计意见书。
3. 高危和中危问题必须整改并复审。
4. 补审通过后才允许进入生产联调。

### 6.3 GitHub required check 配置时机

GitHub required check 只能在 REL-004 平台管理员动作完成后配置。未提供 Hosted Runner artifact 与 branch protection 证据前，不得声称平台闭环。

### 6.4 ERPNext 生产联调前置条件

1. Sprint 2 补审完成。
2. GitHub required check 平台闭环完成。
3. ERPNext 凭据、权限源、主数据测试账号准备完成。
4. 生产联调任务单单独下发并审计通过。
