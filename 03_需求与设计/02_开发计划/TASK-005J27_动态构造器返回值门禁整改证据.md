## 2. 核心实现说明

### 2.1 constructor invocation descriptor 统一收敛

在 `check-style-profit-contracts.mjs` 将 Worker 构造入口统一映射为同一套构造描述与校验链路，覆盖：

- `new Worker(...)` / `new SharedWorker(...)`
- `new Alias(...)`
- `new (Worker.bind(...))(...)`
- `Reflect.construct(...)`（含别名、解构别名、括号成员）
- `new (getWorker())(...)`
- `new ((() => Worker)())(...)`

并统一归一到同一 Worker URL 校验逻辑，不再让 `new` 与 `Reflect.construct` 分叉。

### 2.2 函数返回构造器识别

新增函数返回构造器解析与别名绑定能力，支持：

- 零参数函数/箭头函数直接 `return Worker | SharedWorker`
- 返回 Worker 别名与 `globalThis.Worker/window.Worker`
- 条件返回两分支同构造器时继续解析

当返回值不可静态证明为安全构造器时，按 style-profit surface `fail closed`。

### 2.3 IIFE 与 Reflect.construct 动态构造目标收口

对 `new ((... )())` 与 `Reflect.construct(dynamicCtor, args)` 统一走构造目标解析；当构造目标不确定且 URL 为高危或不可静态证明时，统一阻断。

### 2.4 未知构造目标统一兜底

新增未知构造目标风险分级：

- `unknown ctor + data/blob/http/https/javascript URL`：阻断
- `unknown ctor + unresolved 标识符/表达式 URL`：阻断
- 已知安全构造器白名单（如 `Date`/`URL`/`Error` 等）维持不误杀

实现了 `new unknownCtor(...)` 与 `Reflect.construct(unknownCtor, ...)` 同口径。

### 2.5 本任务未新增依赖

继续复用现有 `typescript` 依赖进行 AST 分析，未引入新第三方解析库。

## 3. 新增/强化反向测试（节选）

`test-style-profit-contracts.mjs` 已加入并通过以下 J27 关键场景（均应失败）：

2. `function getWorker() { return Worker }; new (getWorker())('data:...')`
3. `const getSharedWorker = () => SharedWorker; new (getSharedWorker())('data:...')`
4. `const getWorker = () => globalThis.Worker; new (getWorker())('data:...')`
5. `const W = Worker; const getWorker = () => W; new (getWorker())('data:...')`
6. `new ((() => Worker)())('data:...')`
7. `new ((function () { return SharedWorker })())('data:...')`
8. `const getWorker = () => condition ? Worker : unknownCtor; new (getWorker())('data:...')`
9. `const getWorker = () => unknownCtor; new (getWorker())('data:...')`
10. `const getWorker = () => Worker; Reflect.construct(getWorker(), ['data:...'])`
11. `Reflect.construct((() => Worker)(), ['data:...'])`
12. `new unknownCtor('data:...')`
13. `new unknownCtor(workerUrl)`（`workerUrl` 不可静态证明）
14. `Reflect.construct(unknownCtor, [workerUrl])`（`workerUrl` 不可静态证明）
