# 模块设计：生产计划集成

- 模块名：生产计划集成
- 任务编号：TASK-004
- 版本：V1.15
- 更新时间：2026-04-13 21:49 CST
- 作者：技术架构师
- 责任模块：FastAPI `production` + Vue3 `production` 视图 + ERPNext `Sales Order / Work Order / Job Card / Warehouse / Item`
- 设计结论：生产计划本地保存计划、物料检查、ERPNext 工单映射和同步状态；ERPNext 继续作为 `Sales Order / Work Order / Job Card` 事实源；跨系统写入必须走 outbox/worker，禁止在本地事务内直接调用 ERPNext 创建 Work Order 或修改 Job Card。

## 1. 业务目标

生产计划集成负责把已审批销售订单转化为车间可执行生产任务，打通 `Sales Order -> BOM 展开 -> Production Plan -> Work Order -> Job Card -> Workshop Ticket` 主链路。

目标包括：

1. 读取 ERPNext `Sales Order` 作为订单事实源。
2. 按 `ly_apparel_bom` 默认或指定 BOM 生成本地生产计划。
3. 按 BOM 展开生成物料需求检查快照。
4. 通过 outbox 异步创建 ERPNext `Work Order`，并保存本地映射。
5. 同步 ERPNext `Job Card` 到本地计划工序映射，供工票模块关联。
6. 保证权限、幂等、审计、错误信封和跨系统一致性符合 TASK-001/TASK-003 已建立的标准。

## 2. 系统边界

### 2.1 ERPNext 负责

| ERPNext 能力 | 用途 | FastAPI 使用方式 |
| --- | --- | --- |
| `Sales Order` | 销售订单事实源 | REST API 读取订单、订单行、docstatus、status、company、customer |
| `Item` | 款式/成品主数据 | REST API 校验 item 有效性，FastAPI 不重复建款式主档 |
| `Warehouse` | 成品仓、WIP 仓 | REST API 校验仓库存在和权限 |
| `Work Order` | 标准生产工单 | FastAPI worker 通过 ERPNext API 创建/查询，不直接写 public 表 |
| `Job Card` | 工序卡 | FastAPI 读取并映射到本地；工票模块后续登记执行 |
| `Role / User Permission` | 权限事实源 | 动作权限和资源权限聚合，失败 fail closed |

### 2.2 FastAPI 自建负责

1. `ly_production_plan` 生产计划主表。
2. `ly_production_plan_material` 物料需求检查快照。
3. `ly_production_work_order_link` ERPNext Work Order 映射。
4. `ly_production_work_order_outbox` Work Order 创建异步任务。
5. `ly_production_job_card_link` Job Card 本地映射。
6. `ly_production_status_log` 状态流转日志。
7. SO/BOM/Item/Company 一致性校验、幂等、权限、审计、错误分类。

### 2.3 不在 TASK-004A 范围

1. 不做生产排程甘特图。
2. 不做产能负荷算法。
3. 不做生产入库 Stock Entry。
4. 不做外发单自动创建。
5. 不做 TASK-005 款式利润报表。
6. 不做 TASK-006 加工厂对账单。
7. 不修改工票计薪逻辑。

## 4. API 设计

统一前缀：`/api/production/`。

| 接口 | 方法 | 路径 | 权限动作 | 说明 |
| --- | --- | --- | --- | --- |
| 创建生产计划 | POST | `/api/production/plans` | `production:plan_create` | 校验 SO/BOM 后创建本地计划 |
| 查询生产计划 | GET | `/api/production/plans` | `production:read` | 按资源权限过滤列表 |
| 查询生产计划详情 | GET | `/api/production/plans/{plan_id}` | `production:read` | 返回计划、物料、Work Order、Job Card |
| 物料检查 | POST | `/api/production/plans/{plan_id}/material-check` | `production:material_check` | BOM 展开并快照库存可用量 |
| 创建 Work Order | POST | `/api/production/plans/{plan_id}/create-work-order` | `production:work_order_create` | 只创建 outbox，不在请求事务中调用 ERPNext |
| 同步 Job Card | POST | `/api/production/work-orders/{work_order}/sync-job-cards` | `production:job_card_sync` | 读取 ERPNext Job Card 并写本地映射 |
| 内部 Work Order Worker | POST | `/api/production/internal/work-order-sync/run-once` | `production:work_order_worker` | 服务账号限定，处理 outbox |

统一响应：`{code, message, data}`。

分页响应：`{items, total, page, page_size}`。

## 5. 权限与审计

### 5.1 权限动作

| 动作 | 用途 |
| --- | --- |
| `production:read` | 查询生产计划、物料检查结果、Work Order/Job Card 映射 |
| `production:plan_create` | 创建生产计划 |
| `production:material_check` | 执行物料检查 |
| `production:work_order_create` | 发起 Work Order 创建 outbox |
| `production:job_card_sync` | 手动同步 Job Card |
| `production:work_order_worker` | 内部 Worker 处理 Work Order outbox |

### 5.2 资源权限

1. `company` 和 `item_code` 必须从 ERPNext `Sales Order`、订单行和 BOM 派生，前端传入值不可信。
2. 列表查询必须在数据库层按 `company/item_code` 做过滤。
3. 详情、物料检查、Work Order 创建、Job Card 同步必须做资源级权限校验。
4. ERPNext 权限源不可用时 fail closed，返回 `PERMISSION_SOURCE_UNAVAILABLE`。
5. 越权访问返回 `AUTH_FORBIDDEN` 并写安全审计。

### 5.3 操作审计

必须写操作审计的动作：创建计划、物料检查、创建 Work Order outbox、Job Card 同步、内部 worker run-once、取消计划。

## 6. 关键业务规则

1. 只有 ERPNext `Sales Order.docstatus=1` 且非 `Cancelled/Closed` 才允许生成生产计划。
2. 同一个 SO 若存在多行相同 `item_code`，创建计划必须传 `sales_order_item`，否则返回 `PRODUCTION_SO_ITEM_AMBIGUOUS`。
3. `bom_id` 必须存在、已生效，并且 `bom.item_code == sales_order_item.item_code`。
4. `planned_qty` 必须大于 0，且不得超过该 SO 行剩余未计划数量。
5. 物料检查只生成快照，不锁库存、不改 ERPNext 库存。
6. Work Order 创建必须走 outbox；本地事务提交前不得调用 ERPNext。
7. Work Order outbox `event_key` 必须稳定，不包含 `outbox_id/request_id/created_at/operator` 等易变字段。
8. 重复创建 Work Order 时，如果已有成功映射，直接返回已有 `work_order`。
9. Worker 调用 ERPNext 前必须先短事务 claim outbox，提交后再调用 ERPNext，结果回写再开短事务。
10. ERPNext Work Order 查重必须优先按 `custom_ly_plan_id` 或 `custom_ly_plan_no`，不得只按 item/qty 模糊匹配。
11. Job Card 同步只读取 ERPNext `Job Card` 并更新本地 `ly_production_job_card_link`，不直接改 Job Card 完成数量；完成数量仍由 TASK-003 工票 outbox 同步。
12. BOM 中 `is_subcontract=true` 的工序只标记为外发工序，不在 TASK-004A 自动创建外发单。

## 9. TASK-004A1 审计整改口径

审计意见书第 62 份结论为 TASK-004A 不通过。V1.1 不改变 TASK-004A 的业务边界，只冻结以下整改口径，作为工程师 TASK-004A1 的唯一执行范围。

### 9.1 接口契约必须落地

1. `POST /api/production/plans/{plan_id}/material-check` 必须接收 `warehouse` 请求体字段。
2. `warehouse` 缺失或空白时返回 `PRODUCTION_WAREHOUSE_REQUIRED`。
3. 物料检查快照必须写入 `ly_production_plan_material.warehouse`。
4. `POST /api/production/plans/{plan_id}/create-work-order` 必须接收 `fg_warehouse/wip_warehouse/start_date/idempotency_key`。
5. `fg_warehouse/wip_warehouse/start_date` 必须进入 `ly_production_work_order_outbox.payload_json`。
6. Work Order outbox `payload_hash` 必须覆盖 `fg_warehouse/wip_warehouse/start_date`。
7. 同一 `plan_id + idempotency_key` 相同 payload 返回首次 outbox；不同 payload 返回 `PRODUCTION_IDEMPOTENCY_CONFLICT`。

### 9.2 Worker Claim 必须原子化

1. Work Order outbox worker 必须支持原子 claim。
2. PostgreSQL 下必须采用 `SELECT ... FOR UPDATE SKIP LOCKED` 或等价原子 `UPDATE ... RETURNING`。
3. due 条件必须覆盖 `pending/failed` 到期任务，以及 `processing` 且 lease 过期任务。
4. claim 成功后必须先提交本地事务，再调用 ERPNext。
5. ERPNext 调用不得发生在持有 outbox 行锁的事务内。
6. 并发 worker 不得重复调用 ERPNext 创建同一 `Work Order`。
7. worker 崩溃后，超过 lease 的 `processing` outbox 必须可恢复。

### 9.3 映射表字段必须补齐

`ly_production_work_order_link` 必须补齐：

1. `plan_id` 唯一约束。
2. `sync_status` 索引。
3. `last_synced_at`。
4. `created_by`。

`ly_production_job_card_link` 必须补齐：

1. `company`。
2. `item_code`。
3. `operation_sequence`。
4. `synced_at`。
5. `company,item_code` 联合索引。

### 9.4 测试门禁必须补齐

TASK-004A1 复审前，必须补齐以下测试类别：

1. 仓库字段必填与 payload 入库。
2. SO Closed / Cancelled。
3. BOM item mismatch。
4. 数据库写失败 `DATABASE_WRITE_FAILED`。
5. 操作审计失败 `AUDIT_WRITE_FAILED`。
6. create-work-order commit 失败时 ERPNext 调用次数为 0。
7. worker 原子 claim、并发不重复、lease 过期恢复。
8. work_order_link `plan_id` 唯一约束。
9. job_card_link `company/item_code/operation_sequence/synced_at` 写入。

## 11. TASK-004B1 前端契约整改口径

审计意见书第 64 份结论为 TASK-004B 有条件通过。V1.3 冻结以下契约整改决策。

### 11.1 `planned_start_date` 字段口径

1. `planned_start_date` 是本地生产计划事实字段。
2. `planned_start_date` 必须进入 `ly_production_plan`。
3. 创建生产计划时，前端允许录入并提交 `planned_start_date`。
4. 后端必须保存 `planned_start_date`，并在列表和详情返回。
5. 幂等请求 hash 必须包含 `planned_start_date`。
6. Work Order `start_date` 是创建 ERPNext Work Order outbox 的工单开工日期。
7. 前端创建 Work Order 时，可默认用 `planned_start_date` 填充 `start_date`，但两者是不同字段，不得混用。

### 11.2 内部 worker 权限口径
