## 1. 业务目标

款式利润报表用于按款式、销售订单和时间范围分析利润，输出标准成本、实际成本、差异、利润金额和利润率。该模块依赖 ERPNext 销售与库存事实，也依赖 FastAPI 自建的 BOM、外发、工票和生产计划事实。

## 2. 系统边界

### 2.1 ERPNext 负责

| ERPNext 能力 | 用途 | TASK-005 使用方式 |
| --- | --- | --- |
| `Sales Order` | 销售订单和订单金额候选来源 | 读取订单、客户、公司、订单行、销售单价、数量 |
| `Sales Invoice` | 实际确认收入候选来源 | TASK-005A 必须评估是否优先于 Sales Order |
| `Purchase Receipt` | 采购成本参考与异常排查来源 | 只作为参考来源写入 source map，不直接计入 `actual_material_cost` |
| `Stock Ledger Entry` | 库存实际成本候选来源 | 读取库存价值变动和实际成本 |
| `Item` | 款式和物料主数据 | 校验 item、物料名称、单位、公司范围 |
| `Role / User Permission` | 权限事实源 | 权限源不可用时 fail closed |

1. ERPNext `Sales Order / Sales Invoice` 提供收入候选事实。
2. BOM 展开结果提供标准材料成本和标准工序成本。

## 11. TASK-005B 利润口径设计冻结

审计意见书第 83 份确认 TASK-004C14 通过，TASK-005B 可从“等待 GitHub URL”的阻塞中释放，但只能进入“利润口径设计冻结 / ADR / 任务单”阶段，不得直接进入模型、迁移、API 或前端实现。V0.3 冻结以下 V1 利润口径。

### 11.1 收入口径

1. `estimated_revenue_amount` 来自 ERPNext `Sales Order` 已提交订单行，用于生产前或未开票场景。
2. `actual_revenue_amount` 来自 ERPNext `Sales Invoice` 已提交发票行，用于财务确认利润。
3. V1 默认展示 `actual_revenue_amount`；无已提交发票时展示 `estimated_revenue_amount`，并标记 `revenue_status=estimated`。
4. draft/cancelled Sales Invoice 不得计入实际收入。
5. Sales Order 与 Sales Invoice 不得重复计入收入。

### 11.2 标准成本口径

1. 标准材料成本公式：`standard_material_cost = sum(bom_exploded_required_qty * standard_unit_cost)`。
2. `standard_unit_cost` 来源优先级：ERPNext Item Price 有效采购价 -> ERPNext Item valuation_rate -> unresolved。
3. 标准工序成本公式：`standard_operation_cost = sum(bom_operation_rate * planned_qty)`。
4. 本厂工序使用 BOM 工序计件工价，外发工序使用 BOM `subcontract_cost_per_piece`。
5. 标准价或工价缺失时必须标记 unresolved，不得静默按 0 计算。

### 11.3 实际成本口径

1. 实际材料成本使用 ERPNext `Stock Ledger Entry` 口径：`actual_material_cost = sum(abs(stock_value_difference))`。
2. `Purchase Receipt` 仅作为采购成本参考和异常排查来源，不直接作为 V1 实际材料成本。
3. 实际工票成本公式：`actual_workshop_cost = sum(net_ticket_qty * wage_rate_snapshot)`，其中 `net_ticket_qty = register_qty - reversal_qty`。
4. 工票必须能归属到目标 Sales Order / Production Plan / Work Order / Job Card，无法归属时标记 `workshop_cost_unresolved`。
5. 实际外发成本采用结算优先、验货兜底：已结算/锁定用结算锁定净额，未结算但已验货用验货净额并标记 `subcontract_status=provisional`。
6. 扣款金额只作为明细展示；已使用 `net_amount` 时不得重复扣减。

### 11.4 费用分摊与利润公式

1. V1 默认不纳入制造费用/管理费用分摊：`allocated_overhead_amount = 0`。
2. 报表必须显示 `allocation_status=not_enabled`。
3. 若后续启用费用分摊，必须另行 ADR 冻结分摊基准。
4. `revenue_amount = actual_revenue_amount if revenue_status=actual else estimated_revenue_amount`。
5. `actual_total_cost = actual_material_cost + actual_workshop_cost + actual_subcontract_cost + allocated_overhead_amount`。
6. `standard_total_cost = standard_material_cost + standard_operation_cost`。
7. `profit_amount = revenue_amount - actual_total_cost`。
8. `profit_rate = profit_amount / revenue_amount`；收入为 0 时 `profit_rate = null`。

### 11.5 快照、幂等、权限与审计

1. 利润快照创建后不可修改；重算必须生成新 `snapshot_no`。
2. 快照必须保存公式版本、数据来源、来源单据号、金额和 unresolved 状态。
3. `POST /api/reports/style-profit/snapshot` 必须要求 `idempotency_key`。
4. 幂等唯一键为 `company + idempotency_key`。
5. `request_hash` 必须包含 `company / item_code / sales_order / from_date / to_date / revenue_mode / include_provisional_subcontract / formula_version`。
6. 权限动作冻结为 `style_profit:read`、`style_profit:snapshot_create`、`style_profit:export`。
7. 资源权限维度冻结为 `company + item_code + sales_order`。
8. 权限来源沿用 ERPNext Role / User Permission 聚合，权限源不可用必须 fail closed。
9. 401、403、503 必须写安全审计；快照生成、幂等 replay、幂等冲突、unresolved 快照必须写操作审计。
10. 日志不得写 Authorization/Cookie/token/password/secret 或原始 SQL 明文。

### 12.3 来源映射冻结

1. 收入来源必须同时支持 `Sales Order` 预计收入和 `Sales Invoice` 实际收入。
2. 已提交 `Sales Invoice` 优先作为实际收入；无已提交发票时才使用已提交 `Sales Order` 预计收入，并标记 `revenue_status=estimated`。
3. `Sales Invoice` 与 `Sales Order` 行级映射必须保存 `doctype/name/line_no/item_code/qty/rate/amount`，不得重复计入收入。
4. 实际材料成本只允许从 ERPNext `Stock Ledger Entry` 汇总，使用 `sum(abs(stock_value_difference))`。
5. `Stock Ledger Entry` 纳入规则：必须归属目标 `company + item_code + sales_order/work_order/production_plan`，必须为已提交库存事实，必须排除取消、冲销、非目标仓库或无法归属行。
6. `Purchase Receipt` 只允许作为参考来源写入 source map，不得直接进入 `actual_material_cost`。
7. 本厂工票成本必须来自 `YS Workshop Ticket` 净数量和工价快照。
8. 外发成本必须优先使用结算锁定净额，未结算但已验货时才使用 provisional 验货净额。
9. 生产计划映射只负责把 `Sales Order -> Production Plan -> Work Order -> Job Card` 关系提供给材料和工票归属，不在 TASK-005C 计算利润。

## 13. TASK-005C1 审计阻断整改

TASK-005C 审计结论为有条件通过，但明确要求先完成 TASK-005C1，之后才允许进入 TASK-005D。V0.5 冻结以下整改口径。

### 13.1 source_map 审计追溯字段

1. `ly_style_profit_source_map` 必须能追溯到 `snapshot_id` 和可选 `detail_id`。
2. `snapshot_id` 必须为非空外键，关联 `ly_style_profit_snapshot.id`。
3. `detail_id` 可空，关联 `ly_style_profit_detail.id`；excluded/unresolved 来源允许为空。
4. source_map 必须补齐 `source_system/source_doctype/source_status/qty/unit_rate/amount/currency/warehouse/posting_date/raw_ref`。
5. source_map 必须拆分 `style_item_code` 和 `source_item_code`，前者为利润报表款式，后者为来源行物料或款式。
6. 唯一约束必须改为 `snapshot_id + source_system + source_doctype + source_name + source_line_no`，避免新快照复用同一来源行时冲突。
7. `raw_ref` 必须脱敏，不得保存 Authorization、Cookie、token、password、secret 或原始 SQL。

### 13.2 来源状态 fail closed

1. `_is_submitted()` 不得在缺少 `docstatus/status` 时默认放行。
2. `docstatus == 1` 才可判定为已提交；`docstatus` 存在但非 1 必须排除。
3. `docstatus` 缺失时，只允许显式白名单 `status` 通过。
4. `status` 缺失、空、unknown、draft、cancelled、canceled、closed、void、return 必须排除。
5. `is_cancelled=true` 必须排除。
6. 被排除来源必须保留 `source_status_unknown` 或 `not_submitted_or_cancelled` 等原因，便于审计复核。

### 13.3 snapshot 复核字段

1. `ly_style_profit_snapshot` 必须补齐 `from_date`、`to_date`、`revenue_mode`、`unresolved_count`。
2. `revenue_mode` 默认 `actual_first`，限定 `actual_first/actual_only/estimated_only`。
3. `request_hash` 必须继续包含 `from_date/to_date/revenue_mode`，并继续排除 `created_at/operator/request_id`。
4. 增加 `company + item_code + from_date + to_date` 查询索引。

### 13.4 费用分摊默认状态

1. `ly_cost_allocation_rule.status` 默认值必须为 `disabled`。
2. V1 不允许启用费用分摊；`active` 仅作为未来状态保留。
3. 模型、迁移和测试必须一致。

### 13.5 SLE 款式编码与材料编码语义

1. 利润报表入参 `item_code` 表示款式成品编码，应在代码中明确为 `style_item_code`。
2. ERPNext `Stock Ledger Entry.item_code` 在材料成本场景下通常是材料编码，应映射为 `source_item_code` 或 `material_item_code`。
3. 不得用 `SLE.item_code != style_item_code` 直接排除真实材料成本。
4. SLE 行若能通过 `work_order / production_plan_id / sales_order` 归属到目标款式或订单，可纳入材料成本。
5. 若提供 BOM 物料集合，SLE 材料编码必须在集合内；不在集合内时排除并标记 `material_item_not_in_bom`。
6. 若既无工单/生产计划/销售订单桥，也无 BOM 物料集合验证，则不得纳入，标记 `unable_to_link_order_or_material_scope`。
7. `Purchase Receipt` 仍只允许作为参考来源，不得进入 `actual_material_cost`。

## 14. TASK-005C2 默认不纳入与状态白名单收口

TASK-005C1 审计确认 `_is_submitted()` fail closed、source_map 主体追溯字段、snapshot 复核字段、费用分摊默认 disabled、SLE 款式/材料编码语义已完成主体整改，但仍要求先完成 TASK-005C2，之后才可评估 TASK-005D。

### 14.1 include_in_profit 默认 false

1. `ly_style_profit_source_map.include_in_profit` 是利润来源纳入开关，默认必须为 false。
2. 模型、迁移和 DTO 默认值必须一致为 false。
3. 新来源初始必须为 false，只有通过提交状态、资源归属、材料范围和排除规则后，才允许显式设置 true。
4. `mapping_status` 默认建议为 `unresolved`，不得默认 `mapped`。
5. `mapping_status != 'mapped'` 时，`include_in_profit` 必须为 false。

### 14.2 source_map 字段契约

1. `snapshot_id` 非空，`detail_id` 可空。
2. `source_system` 长度 64，非空，限定 `erpnext/fastapi/manual`。
3. `source_doctype` 长度 140，非空。
4. `source_name` 长度 140，非空。
5. `source_line_no` 长度 140，非空，默认空字符串。
6. `source_status` 长度 64，非空，缺失时统一为 `unknown`；`unknown` 不允许纳入利润。
7. `style_item_code` 长度 140，非空；`source_item_code` 长度 140，可空。
8. `amount` 非空默认 0；`qty/unit_rate/currency/warehouse/posting_date` 可空。
9. `mapping_status` 非空默认 `unresolved`，限定 `mapped/unresolved/excluded`。
10. `raw_ref` 可空但必须脱敏。

### 14.3 索引与唯一约束

1. 唯一约束：`snapshot_id + source_system + source_doctype + source_name + source_line_no`。
2. 索引：`snapshot_id + mapping_status`。
3. 索引：`company + sales_order + style_item_code`。
4. 索引：`source_system + source_doctype + source_name`。
5. 索引：`detail_id`。

### 14.4 status-only 白名单

1. `docstatus` 是首选事实字段；`docstatus == 1` 才可判定 submitted。
2. `docstatus` 存在但不是 1 时，直接 false，不再看 status。
3. `is_cancelled=true` 直接 false。
4. `docstatus` 缺失时，必须识别 `source_doctype`；未知 doctype 一律 false。
5. `Stock Ledger Entry` 不允许 status-only 放行，必须有 `docstatus=1`。
6. `Purchase Receipt` 不允许通过 status-only 进入利润成本。
7. `Sales Invoice` status-only 白名单只允许 `paid/unpaid/overdue/partly paid`。
8. `Sales Order` status-only 白名单只允许 `to deliver and bill/to bill/to deliver/completed`。
9. `closed/draft/cancelled/canceled/void/return/unknown/空字符串` 一律 false。
