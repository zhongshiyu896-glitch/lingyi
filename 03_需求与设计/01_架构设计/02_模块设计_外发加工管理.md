## 1. 业务目标

外发加工管理负责服装工厂将洗水、印花、绣花、特殊加工等工序委托给加工厂的完整业务链路。

目标包括：

1. 建立外发单创建、发料、回料、验货、扣款、结算出口的闭环。
2. 外发业务事实沉淀在 `ly_schema`，库存事实落在 ERPNext `Stock Entry / Stock Ledger Entry`。
3. 外发单必须可追溯到 ERPNext `Supplier`、ERPNext `Item`、ERPNext `Warehouse` 和 FastAPI `ly_apparel_bom / ly_bom_operation`。
4. 发料、回料、验货写接口必须幂等，避免重复提交导致重复本地事实和重复库存落账。
5. 外发结算数据必须为 TASK-006 加工厂对账单提供稳定来源，结算后锁定金额。

## 2. 系统边界

### 2.1 ERPNext 负责

| ERPNext 能力 | 用途 | FastAPI 使用方式 |
| --- | --- | --- |
| `Supplier` | 加工厂主数据 | 校验 `supplier` 存在且属于加工厂类型 |
| `Item` | 款式和物料主数据 | 校验 `item_code/material_item_code` 有效 |
| `Warehouse` | 发料仓、回料仓 | 校验仓库存在和资源权限 |
| `Stock Entry` | 发料/回料库存单据 | FastAPI worker 通过 ERPNext API 创建 |
| `Stock Ledger Entry` | 库存台账事实源 | 只读查询库存落账结果 |
| `Role / User Permission` | 权限事实源 | 动作权限和资源权限聚合，失败 fail closed |

1. `ly_subcontract_order` 外发主单。
2. `ly_subcontract_material` 外发发料计划与事实。

6. `ly_subcontract_stock_outbox` ERPNext Stock Entry 异步同步任务。
7. `ly_subcontract_stock_sync_log` Stock Entry 同步尝试日志。
8. 外发状态机、幂等、金额计算、结算锁定、权限审计。

### 2.3 Vue3 前端负责

## 3. 数据模型

用途：外发加工主单。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `subcontract_no` | 外发单号，唯一 |
| `company` | 公司，来自 ERPNext Company |
| `supplier` | 加工厂，引用 ERPNext Supplier |
| `item_code` | 款式/成品，引用 ERPNext Item |
| `bom_id` | 引用 `ly_apparel_bom.id` |
| `process_name` | 外发工序名称，必须来自 `ly_bom_operation.is_subcontract=true` |
| `subcontract_rate` | 外发单件加工价 |
| `planned_qty` | 计划外发数量 |
| `issued_qty` | 已发料数量汇总 |
| `received_qty` | 已回料数量汇总 |
| `inspected_qty` | 已验货数量汇总 |
| `rejected_qty` | 不合格数量汇总 |
| `accepted_qty` | 合格数量汇总 |
| `gross_amount` | 加工费总额 |
| `deduction_amount` | 扣款总额 |
| `net_amount` | 实付金额 |
| `settlement_status` | `unsettled/partially_settled/settled/adjusted/cancelled` |
| `statement_id` | 加工厂对账单 ID，引用 TASK-006 |
| `settlement_no` | 对账单号，冗余快照 |
| `settled_by` | 结算确认人 |
| `settled_at` | 结算确认时间 |
| `status` | 外发状态 |
| `created_by/updated_by` | 操作者 |
| `created_at/updated_at` | 时间戳 |

索引与约束：

1. `uk_ly_subcontract_order_no`：`subcontract_no` 唯一。
2. `idx_ly_subcontract_supplier_status`：`supplier, status`。
3. `idx_ly_subcontract_item_status`：`item_code, status`。
4. `idx_ly_subcontract_company_status`：`company, status`。
5. `idx_ly_subcontract_statement`：`statement_id`。
6. `planned_qty > 0`。
7. `subcontract_rate >= 0`。
8. 金额和数量字段必须使用 `Decimal/Numeric`，禁止 `float`。

### 3.2 `ly_schema.ly_subcontract_material`

用途：外发发料计划和发料事实。

关键字段：`id, subcontract_id, company, issue_warehouse, material_item_code, color, size, batch_no, uom, required_qty, issued_qty, remaining_qty, stock_entry_name, stock_outbox_id, sync_status, sync_error_code, idempotency_key, created_by, created_at`。

规则：

1. `required_qty` 来自 BOM 展开或外发物料计划，不以前端传入值作为事实源。
2. `issued_qty <= remaining_required_qty`。
3. 同一发料幂等键不得生成多条发料事实或多个 Stock Entry outbox。

### 3.3 `ly_schema.ly_subcontract_receipt`

用途：外发回料事实。

关键字段：`id, subcontract_id, company, receipt_warehouse, item_code, received_qty, stock_entry_name, stock_outbox_id, sync_status, sync_error_code, idempotency_key, received_by, received_at`。

规则：

1. `received_qty > 0`。
2. 回料必须在允许状态下执行。
3. 同一回料幂等键不得生成多条回料事实或多个 Stock Entry outbox。

### 3.4 `ly_schema.ly_subcontract_inspection`

用途：外发验货、扣款和实付金额事实。

关键字段：`id, subcontract_id, inspected_qty, accepted_qty, rejected_qty, rejected_rate, deduction_rate, gross_amount, deduction_amount, net_amount, idempotency_key, inspected_by, inspected_at`。

规则：

1. `inspected_qty > 0`。
2. `0 <= rejected_qty <= inspected_qty`。
3. 同一验货幂等键不得重复生成验货事实或重复累计金额。

### 3.5 `ly_schema.ly_subcontract_status_log`

用途：外发状态流转审计。

关键字段：`id, subcontract_id, from_status, to_status, action, operator, request_id, remark, before_data, after_data, created_at`。

规则：状态变化必须写日志，日志不得记录敏感凭证。

### 3.6 `ly_schema.ly_subcontract_stock_outbox`

用途：ERPNext Stock Entry 异步同步任务。

关键字段：`id, event_key, subcontract_id, stock_action, company, warehouse, item_code, material_item_code, qty, uom, status, attempts, max_attempts, next_retry_at, locked_by, locked_at, stock_entry_name, last_error_code, last_error_message, request_id, created_by, created_at, updated_at`。

状态：`pending/processing/succeeded/failed/dead/blocked_scope`。

规则：

1. `event_key` 全局唯一。
2. `stock_action` 只能是 `issue/receipt`。
3. `status + next_retry_at` 建索引。
4. outbox 必须在本地事务内写入，ERPNext 调用必须在本地事务提交后执行。
5. 服务账号无资源权限的 outbox 不得被处理为成功。

### 3.7 `ly_schema.ly_subcontract_stock_sync_log`

用途：记录每次 ERPNext Stock Entry 同步尝试。

关键字段：`id, outbox_id, subcontract_id, stock_action, attempt_no, erpnext_status, stock_entry_name, error_code, error_message, request_id, created_at`。

迁移与回填规则：

2. 新建外发单必须写入非空 `company`；无法唯一解析时不得创建外发单。

6. 权限源不可用必须返回 `PERMISSION_SOURCE_UNAVAILABLE`，不得放开全资源。

### 3.9 TASK-002C1 迁移自洽与创建事实校验

TASK-002C1 是 TASK-002C 的阻断整改。迁移和创建单据必须满足以下硬约束：

5. 已有库升级必须只补缺失字段、索引、约束，不得破坏已有数据。
6. 创建外发单必须校验 `bom.item_code == payload.item_code`。
7. BOM item 与 payload item 不一致时必须返回 `SUBCONTRACT_BOM_ITEM_MISMATCH`，不得新增外发主单、状态日志、成功操作审计或子表事实。

12. 如果 ERPNext 明确配置 Item/Supplier/Warehouse 限制，则这些限制必须和 company 权限叠加校验，不得仅凭 company 放行。
13. TASK-002C1 阶段 `ly_subcontract_stock_outbox` 只验收资源 scope 骨架；完整 outbox 字段、幂等唯一约束和 worker 处理由 TASK-002D 接管。

### 3.10 TASK-002D 发料 Stock Entry Outbox 最终结构

| `ly_subcontract_stock_outbox` | `stock_action` | 本阶段只允许 `issue` |
| `ly_subcontract_stock_outbox` | `idempotency_key` | 发料写接口幂等键 |
| `ly_subcontract_stock_outbox` | `payload_hash` | 发料 payload 摘要，用于幂等冲突判断 |

索引与约束：

发料与 worker 规则：

1. 一次发料请求生成一个 `issue_batch_no`、一条 outbox、多条 `ly_subcontract_material`。
2. 发料接口本地事务内只写本地事实、状态日志、操作审计和 outbox，禁止调用 ERPNext。
3. 相同 `idempotency_key` + 相同 `payload_hash` 返回第一次结果。
4. 相同 `idempotency_key` + 不同 `payload_hash` 返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
5. Worker 主查询必须限制 `stock_action='issue'`，不得处理 `receipt`。
6. Worker 必须先按服务账号 `company/item_code/supplier/warehouse` scope 过滤，再 limit。
7. Worker 必须使用行级锁或租约字段避免并发重复处理同一 outbox。
8. ERPNext Stock Entry 必须创建并提交，提交成功后才能标记 outbox `succeeded`。
9. ERPNext payload 必须写入 `custom_ly_subcontract_no/custom_ly_subcontract_outbox_id/custom_ly_outbox_event_key/custom_ly_stock_action`。
10. 重试时必须先按 `custom_ly_outbox_event_key` 查询 ERPNext 是否已有 Stock Entry，存在则补本地成功回写，不得重复创建。
11. ERPNext 创建成功但本地回写失败时，下次 worker 必须通过 event_key 找回原单，避免重复库存落账。
12. `receive/inspect` 在 TASK-002D 完成后仍保持 fail closed。
13. 经 TASK-002D1 修订，`event_key` 必须以幂等事件为单位生成，输入为 `stock_action + subcontract_id + idempotency_key + stable_payload_hash`，不得包含 `issue_batch_no`。
14. `issue_batch_no` 只用于本地发料批次展示和 material 分组，不参与库存同步幂等事件键。

### 3.11 TASK-002D1 发料幂等与 Worker 库存一致性

TASK-002D1 是 TASK-002D 的阻断整改。本节冻结以下口径：

1. `event_key` 正式以幂等事件为单位生成，不包含易变 `issue_batch_no`。
2. `event_key` 推荐输入：`stock_action + subcontract_id + idempotency_key + stable_payload_hash`。
3. `issue_batch_no` 是本地发料批次号，只用于业务展示和 material 分组，不参与 event_key。
4. `ly_subcontract_stock_outbox` 是幂等权威表，保存 `idempotency_key/payload_hash/payload_json/event_key`。
5. `ly_subcontract_material` 以 `stock_outbox_id` 关联 outbox 查询幂等上下文；不强制在 material 行冗余 `idempotency_key/payload_hash`。
6. 发料幂等检查必须在剩余数量校验之前执行。
7. 相同 `subcontract_id + stock_action='issue' + idempotency_key` 且 `payload_hash` 一致时，必须直接返回第一次 outbox 结果。
8. 全量发料 `items=[]` 的 `stable_payload_hash` 必须基于请求语义计算，不得因第一次发料后剩余数量变化而改变。
9. `ERPNextStockEntryService.find_by_event_key()` 必须返回结构化结果，至少包含 `name/docstatus`。
10. 只有 `docstatus=1` 的 ERPNext Stock Entry 可直接补本地 `succeeded`。
11. `docstatus=0` 的 ERPNext draft 必须 submit 原单，submit 成功后才能本地 `succeeded`。
12. `docstatus=2` 或未知 docstatus 不得本地成功，必须 fail closed。
13. 相同 event_key 查询到多张 ERPNext Stock Entry 时必须 fail closed，禁止任选一张成功。
14. Worker 必须拆为 claim 短事务、ERPNext 网络调用、result 回写短事务。
15. Worker 调用 ERPNext 时不得持有数据库事务或行锁。
16. result 回写失败后，下一轮 worker 必须能通过 event_key 找回 ERPNext 已提交单据。
17. 并发 worker 不得重复处理同一 outbox；lease 到期后允许重新 claim。
18. TASK-002D1 不得恢复 `receive/inspect` 成功路径。

### 3.12 TASK-002E 回料 Stock Entry Outbox

TASK-002E 将 `POST /api/subcontract/{id}/receive` 从 fail closed 升级为正式回料 outbox 路径。本阶段只允许实现回料 `Material Receipt`，不得实现验货、对账或结算。

回料字段与 outbox 规则：

1. 一次回料请求生成一个 `receipt_batch_no`、一条 `stock_action='receipt'` outbox、一组 `ly_subcontract_receipt` 回料事实。
2. `event_key` 必须以幂等事件为单位生成，输入为 `stock_action + subcontract_id + idempotency_key + stable_payload_hash`。
3. `event_key` 不得包含易变 `receipt_batch_no/request_id/outbox_id/created_at/operator`。
4. `receipt_batch_no` 只用于本地回料批次展示和 receipt 分组，不参与库存同步幂等事件键。
5. 回料接口本地事务内只写回料事实、主单汇总、状态日志、操作审计和 receipt outbox，禁止调用 ERPNext。
6. 本地回料事实和 receipt outbox 必须同事务提交；commit 失败不得调用 ERPNext。
7. 回料 item 默认为 `ly_subcontract_order.item_code`，禁止前端提交任意 item 成为回料事实。
8. 默认禁止超收：`received_qty <= planned_qty - received_qty_already_recorded`。
9. 幂等检查必须前置到剩余可回数量校验之前。
10. `draft/completed/cancelled/settled` 状态不得回料。
11. 回料入列后，主单可推进到 `waiting_inspection`，但 `inspect` 仍必须 fail closed。
12. Worker 必须按 `stock_action` 明确分派 issue/receipt handler，禁止混用。
13. Worker 处理 receipt 时必须创建并提交 ERPNext `Stock Entry / Material Receipt`。
14. ERPNext receipt payload 必须使用 `t_warehouse`，不得使用 issue 的 `s_warehouse`。
15. Worker 继续遵守 TASK-002D1 的 docstatus 校验、event_key 查重和短事务规则。

## 4. 状态机

状态冻结：

1. `draft`：草稿。
2. `issued`：已发料，本地发料事实和 outbox 已入列。
3. `processing`：加工中。
4. `waiting_receive`：待回料。
5. `waiting_inspection`：待验货。
6. `completed`：已完成。
7. `cancelled`：已取消。

动作流转：

| 动作 | 前置状态 | 目标状态 | 说明 |
| --- | --- | --- | --- |
| 创建 | 无 | `draft` | 创建外发单 |
| 发料 | `draft` | `issued` | 写发料事实和 Stock Entry outbox |
| 加工确认 | `issued` | `processing` | 可由后续任务实现 |
| 回料 | `issued/processing/waiting_receive/waiting_inspection` | `waiting_inspection` | 写回料事实和 Stock Entry outbox，允许待验货状态继续分批回料 |
| 验货 | `waiting_inspection` | `completed` | 计算验货、扣款、实付金额 |
| 取消 | `draft/issued` | `cancelled` | 已产生不可逆库存落账的取消必须走反冲任务 |

禁止：

1. `draft` 直接回料。
2. `draft` 直接验货。
3. `completed/cancelled/settled` 后继续发料、回料、验货。
4. 结算锁定后直接修改金额。

## 6. 写接口幂等契约

外发写接口必须防重复提交。幂等范围覆盖：创建、发料、回料、验货、取消、库存同步重试。

### 6.1 幂等键来源

1. 首选请求头：`Idempotency-Key`。
2. 可兼容请求体字段：`idempotency_key`。
3. PDA/MES/外部系统推送可使用业务唯一键：`external_event_key`。
4. 后端必须对幂等键做长度限制和字符白名单，禁止记录 token、password、secret、cookie、authorization 等敏感语义。

### 6.2 幂等唯一约束

建议唯一维度：

| 动作 | 唯一键 |
| --- | --- |
| 创建外发单 | `company + supplier + item_code + bom_id + process_name + idempotency_key` |
| 发料 | `subcontract_id + stock_action='issue' + idempotency_key` |
| 回料 | `subcontract_id + stock_action='receipt' + idempotency_key` |
| 验货 | `subcontract_id + action='inspect' + idempotency_key` |
| 取消 | `subcontract_id + action='cancel' + idempotency_key` |
| 手动重试 | `subcontract_id + action='stock_sync_retry' + idempotency_key` |

### 6.3 重复提交处理

1. 幂等键相同且 payload 摘要一致：返回第一次成功结果，不新增本地事实，不新增 outbox，不新增 ERPNext Stock Entry。
2. 幂等键相同但 payload 摘要不一致：返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
3. 首次请求失败且事务回滚：允许用同一幂等键重试。
4. 首次请求已写本地事实但 outbox 待处理：重复提交返回原 outbox 状态。
5. 首次请求已同步 ERPNext 成功：重复提交返回原 `stock_entry_name`。

### 6.4 幂等验收

1. 连续两次相同发料请求，只生成 1 条 `ly_subcontract_material` 和 1 条 `ly_subcontract_stock_outbox`。
2. 连续两次相同回料请求，只生成 1 条 `ly_subcontract_receipt` 和 1 条 `ly_subcontract_stock_outbox`。
3. 连续两次相同验货请求，只生成 1 条 `ly_subcontract_inspection`，金额不重复累计。
4. 相同幂等键不同 payload 返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。

## 7. 公式与金额口径

1. `accepted_qty = inspected_qty - rejected_qty`。
2. `rejected_rate = rejected_qty / inspected_qty`。
3. `gross_amount = inspected_qty * subcontract_rate`。已废弃旧口径 `gross_amount = accepted_qty * subcontract_rate`，后续对账不得引用旧口径。
4. `deduction_amount = rejected_qty * deduction_rate`。
5. `net_amount = gross_amount - deduction_amount`。

要求：

1. 全部数量、单价、金额必须使用 `Decimal/Numeric`。
2. 禁止 `net_amount = inspected_qty - deduction_amount` 这类数量减金额公式。
3. 如果后续业务确认加工费按验货数量而非合格数量计算，必须出 ADR 修改公式，不得在代码中私自改口径。

## 8. 对账结算边界

外发模块只产生可结算明细，TASK-006 负责加工厂对账单生成、确认和应付草稿。

### 8.1 外发模块必须提供的结算字段

1. `settlement_status`：`unsettled/partially_settled/settled/adjusted/cancelled`。
2. `statement_id`：TASK-006 对账单主键。
3. `settlement_no`：TASK-006 对账单号快照。
4. `settled_by`：结算确认人。
5. `settled_at`：结算确认时间。
6. `gross_amount/deduction_amount/net_amount`：结算金额快照。

### 8.2 结算锁定规则

1. `settlement_status=settled` 后，外发单和验货明细金额不得直接修改。
2. 已结算外发单不得再次发料、回料、验货、取消。
3. 需要调整时，只能通过 TASK-006 生成调整单或反冲单。
4. 已结算记录必须保留原 `statement_id/settlement_no/settled_by/settled_at`。
5. 对账单取消后，外发单状态可按 TASK-006 规则退回 `unsettled` 或进入 `adjusted`，必须写状态日志和操作审计。

### 8.3 对账数据出口

TASK-002 提供给 TASK-006 的数据口径：

1. 只输出 `status=completed` 且 `settlement_status in ('unsettled','partially_settled')` 的外发单。
2. 明细金额使用验货完成时的金额快照。
3. 同一外发验货明细不得重复进入未取消对账单。
4. 对账查询必须按 `supplier/company/date range` 过滤，主体为空必须拦截。

## 9. ERPNext 集成与一致性策略

### 9.1 Stock Entry 类型

| 外发动作 | ERPNext 单据 | 说明 |
| --- | --- | --- |
| 发料 | `Stock Entry` / `Material Issue` | 从本厂仓库发给加工厂 |
| 回料 | `Stock Entry` / `Material Receipt` | 加工完成或半成品回入本厂仓库 |

### 9.2 一致性策略

1. 本地事务内写业务事实、状态日志、操作审计、`ly_subcontract_stock_outbox`。
2. 本地事务内禁止调用 ERPNext 写接口。
3. 本地 commit 成功后，由内部 worker 调用 ERPNext 创建 `Stock Entry`。
4. ERPNext 成功后，回写 `stock_entry_name`、outbox `succeeded`、sync_log success。
5. ERPNext 失败后，outbox 标记 `failed` 或 `dead`，sync_log failed，可重试。
6. 本地事务失败时不得调用 ERPNext。
7. 操作审计失败导致本地事务回滚时不得调用 ERPNext。

## 10. 内部库存同步 Worker 安全契约

`POST /api/subcontract/internal/stock-sync/run-once` 属于内部接口，不是普通前端业务能力。

### 10.1 生产开关

1. 生产环境默认关闭 HTTP 内部 worker 入口，除非 `ENABLE_SUBCONTRACT_INTERNAL_STOCK_WORKER_API=true`。
2. 未开启时返回 `INTERNAL_API_DISABLED`，写安全审计。
3. `dry_run=true` 生产默认禁用，除非 `SUBCONTRACT_ENABLE_STOCK_WORKER_DRY_RUN=true`。
4. dry-run 被禁用时返回 `SUBCONTRACT_DRY_RUN_DISABLED`，写安全审计，不查询 outbox。

### 10.2 服务账号最小权限

1. 调用主体必须已认证。
2. 调用主体必须具备 `subcontract:stock_sync_worker`。
3. 调用主体必须是服务账号、系统级集成账号或 `System Manager`。
4. 服务账号必须按 ERPNext `User Permission` 限定 `company/item_code/supplier/warehouse`。
5. 服务账号没有明确资源 scope 时 fail closed。
6. ERPNext 权限源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`，不得处理 outbox。
7. 服务账号不得使用 `*`、`all`、空数组表示全资源权限。

### 10.3 Worker 查询与越权处理

1. 主处理查询必须先按服务账号 `company/item_code/supplier/warehouse` scope 过滤，再 `order by/limit`。
2. 越权 outbox 不得进入主处理窗口。
3. 越权 outbox 不得被当前服务账号锁定、增加 attempts、更新 retry 或标记成功。
4. 缺少资源 scope 的 outbox 必须进入 `blocked_scope/dead`，并写安全审计。
5. 越权诊断必须显式启用，默认不扫描越权 outbox。
6. 越权诊断必须有扫描上限、冷却期、去重机制，避免安全审计放大。
7. dry-run 不得锁定 outbox、不得增加 attempts、不得更新状态、不得调用 ERPNext。
8. 授权 dry-run 成功必须写操作审计。

## 11. 权限与审计

### 11.1 权限动作

| 动作 | 权限码 |
| --- | --- |
| 外发读取 | `subcontract:read` |
| 外发创建 | `subcontract:create` |
| 外发发料 | `subcontract:issue_material` |
| 外发回料 | `subcontract:receive` |
| 外发验货 | `subcontract:inspect` |

| 内部库存同步 Worker | `subcontract:stock_sync_worker` |

### 11.2 资源权限

资源维度：`company/item_code/supplier/warehouse`。

规则：

1. 动作权限和资源权限必须同时通过。
2. 前端按钮权限不能替代后端权限校验。
3. 权限来源不可用必须 fail closed。
4. 401、403、503 必须写安全审计。
5. TASK-002C 完成后，`company` 必须优先从 `ly_subcontract_order.company` 读取。
6. 列表查询必须在数据库层按当前用户可访问 `company` 过滤，不得先查全量再在 Python 层过滤。
7. 详情和写接口必须先读取主单本地 `company`，再执行 `company/item_code/supplier/warehouse` 资源级校验。
8. `company` 缺失、空白、无法解析或 `resource_scope_status='blocked_scope'` 时，普通业务动作必须 fail closed。
9. Company-only 用户在权限源明确无 Item/Supplier/Warehouse 显式限制时，可以访问该 company 内外发单。
10. Company-only 用户不得访问其他 company 外发单。
11. Item/Supplier/Warehouse 限制一旦存在，必须与 company 权限叠加校验。
12. 权限源查询失败不得被解释为“无显式限制”。

### 11.3 操作审计

必须写操作审计的动作：创建、发料、回料、验货、取消、库存同步重试、内部 worker 成功执行、授权 dry-run。

审计必须记录：`module/action/operator/resource_type/resource_id/resource_no/before_data/after_data/request_id`。

禁止记录：`Authorization/Cookie/token/password/secret` 明文、SQL 原文、SQL 参数、ERPNext 凭证。

## 12. 校验规则

创建外发单必须校验：

1. `supplier` 存在于 ERPNext `Supplier`，且为加工厂类型。
2. `item_code` 存在于 ERPNext `Item`，且未禁用。
3. `bom_id` 存在于 `ly_apparel_bom`，且 `bom.item_code == payload.item_code`。
4. `process_name` 存在于 `ly_bom_operation`，且 `is_subcontract=true`。
5. `subcontract_rate` 必须来自 BOM 外发工序或明确输入，且 `>= 0`。
6. `payload.item_code` 必须与 `ly_apparel_bom.item_code` 一致。
7. `payload.item_code` 与 BOM item 不一致时返回 `SUBCONTRACT_BOM_ITEM_MISMATCH`，不得落库。
8. 显式传入 `company` 时，仍必须校验 Item/BOM 事实一致性。

发料必须校验：

1. 单据状态允许发料。
2. 发料仓存在且用户有权限。
3. 物料来自 BOM 展开或外发物料计划。
4. `issued_qty <= remaining_required_qty`。
5. 幂等键有效。

回料必须校验：

1. 单据状态允许回料。
2. 回料仓存在且用户有权限。
3. `received_qty > 0`。
4. 幂等键有效。

验货必须校验：

1. 单据状态为 `waiting_inspection`。
2. `0 <= rejected_qty <= inspected_qty`。
3. 幂等键有效。
4. 未结算。

## 13. TASK-002B 阶段临时门禁

TASK-002B 只允许交付权限、资源权限、安全审计、操作审计、统一错误信封和事务边界。回料 outbox、验货金额口径和正式业务事实写入分别属于 TASK-002E、TASK-002F，不得在 TASK-002B 中以旧演示逻辑继续成功。

### 13.1 回料接口门禁

`POST /api/subcontract/{id}/receive` 在 TASK-002B 阶段必须按以下顺序处理：

1. 解析并规范化 `request_id`。
2. 校验当前用户；未登录返回 `AUTH_UNAUTHORIZED` 并写安全审计。
3. 校验动作权限 `subcontract:receive`；无权限返回 `AUTH_FORBIDDEN` 并写安全审计。
4. 读取目标外发单并校验 `company/item_code/supplier` 资源权限。
5. 校验回料仓 `receipt_warehouse` 资源权限；当前 schema 若缺少回料仓字段，必须返回 `SUBCONTRACT_WAREHOUSE_REQUIRED` 或等价 fail closed 错误。
6. 权限全部通过后，因 TASK-002E 尚未实现回料 Stock Entry outbox，必须返回 `SUBCONTRACT_OUTBOX_REQUIRED` 或等价未实现错误。
7. 不得新增 `ly_subcontract_receipt`。
8. 不得推进外发单状态到 `waiting_inspection`。
9. 不得伪造 `stock_entry_name`。
10. 不得调用 ERPNext 写接口。

### 13.2 验货接口门禁

`POST /api/subcontract/{id}/inspect` 在 TASK-002B 阶段必须按以下顺序处理：

1. 解析并规范化 `request_id`。
2. 校验当前用户；未登录返回 `AUTH_UNAUTHORIZED` 并写安全审计。
3. 校验动作权限 `subcontract:inspect`；无权限返回 `AUTH_FORBIDDEN` 并写安全审计。
4. 读取目标外发单并校验 `company/item_code/supplier` 资源权限。
5. 权限全部通过后，因 TASK-002F 尚未实现验货扣款金额口径，必须返回 `SUBCONTRACT_INSPECTION_NOT_IMPLEMENTED` 或等价未实现错误。
6. 不得新增 `ly_subcontract_inspection` 或修改 `ly_subcontract_receipt`。
7. 不得推进外发单状态到 `completed`。
8. 不得计算或返回旧公式 `net_amount = inspected_qty - deduction_amount`。
9. 不得写入 `gross_amount/deduction_amount/net_amount`。

### 13.3 异常分类与顺序

1. 鉴权必须优先于业务字段校验，未登录或无权限时不得先返回字段校验错误。
2. 数据库读取外发单失败返回 `DATABASE_READ_FAILED`。
3. 数据库写入、flush、commit 失败返回 `DATABASE_WRITE_FAILED`。
4. 回料/验货在 TASK-002B 阶段被门禁拦截时，不属于数据库失败，不得写本地业务事实。
5. 旧 `draft` 直接回料通道必须删除；`draft` 状态调用回料不得成功。

## 14.10 TASK-002E1 回料同步重试精确定位整改

### 14.10.1 retry 契约

1. `/api/subcontract/{id}/stock-sync/retry` 必须接收 `outbox_id + stock_action + idempotency_key`。
2. retry 接口必须按 `outbox_id` 精确读取目标 outbox，不得按外发单取最新 outbox。
3. 目标 outbox 必须同时满足：`subcontract_id=路径 id`、`stock_action=请求 stock_action`、`idempotency_key=请求 idempotency_key`。
4. 只允许重试 `failed/dead` 状态。
5. `succeeded/processing/pending/blocked_scope` 禁止重置。
6. retry 接口只改本地 outbox 重试状态，不得调用 ERPNext。
7. retry 资源权限以目标 outbox 的 `company/item_code/supplier/warehouse/stock_action` 为准，权限源不可用 fail closed。
8. retry 成功写操作审计，401/403/503 写安全审计。

### 14.10.2 幂等 payload hash 归一化

1. `stable_payload_hash` 必须对数量字段做 Decimal 语义归一化。
2. `10`、`10.0`、`10.000000` 视为同一数量，必须生成相同 payload hash。
3. 数量归一化同时适用于发料和回料，至少覆盖 `issued_qty/required_qty/received_qty/qty`。
4. 相同幂等键但数量语义相同不得误报 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。

### 14.10.3 回料状态裁决

1. 正式允许 `waiting_inspection` 状态继续分批回料。
2. `waiting_inspection` 下只能继续写 receipt/outbox，不得执行验货成功路径。
3. `completed/cancelled/settled` 仍禁止回料。
4. 允许该状态的理由：服装外发存在分批回料、先到批次等待验货、后续批次继续回厂的业务场景。

### 14.10.4 ERPNext submit 后确认

1. Worker submit ERPNext draft Stock Entry 后必须确认最终 `docstatus=1`。
2. submit API 返回成功壳但最终 `docstatus!=1` 时，不得本地标记 `succeeded`。
3. 该规则同时适用于 issue 和 receipt outbox。

## 14.11 TASK-002F 验货扣款金额口径

### 14.11.1 验货边界

1. `POST /api/subcontract/{id}/inspect` 从 TASK-002F 开始允许写本地验货事实。
2. 验货必须绑定 `receipt_batch_no`。
3. 目标回料批次必须已完成 ERPNext Material Receipt 同步，即 `sync_status='succeeded'` 且 `stock_entry_name` 不为空。
4. 本任务不得调用 ERPNext 写接口，不得创建 `Stock Entry / Purchase Invoice / Payment Entry / GL Entry`。
5. 本任务不得实现加工厂对账单或结算锁定，只输出对账所需本地金额事实。

### 14.11.2 金额公式

1. `accepted_qty = inspected_qty - rejected_qty`。
2. `rejected_rate = rejected_qty / inspected_qty`，保留 6 位小数。
3. `gross_amount = inspected_qty × subcontract_rate`。
4. `deduction_amount = rejected_qty × deduction_amount_per_piece`。
5. `net_amount = gross_amount - deduction_amount`。
6. `deduction_amount` 不得大于 `gross_amount`。
7. 金额字段使用 Decimal，保留 2 位小数，舍入规则为 ROUND_HALF_UP。
8. 禁止旧错误公式：`net_amount = inspected_qty - deduction_amount`。

### 14.11.3 状态流转

1. 累计已验数量小于累计已回料数量：状态为 `waiting_inspection`。
2. 累计已验数量等于累计已回料数量，且累计已回料数量小于计划数量：状态为 `waiting_receive`。
3. 累计已验数量等于累计已回料数量，且累计已回料数量等于计划数量：状态为 `completed`。
4. `waiting_inspection` 允许继续分批回料，保持 TASK-002E1 已确认口径。

### 14.11.4 retry 回潮风险清理

1. 旧 `SubcontractStockOutboxService.get_retry_target()` 按最新 outbox 选择目标的逻辑必须删除或显式废弃。
2. `/api/subcontract/{id}/stock-sync/retry` 只能使用 `outbox_id + stock_action + idempotency_key` 精确定位。
3. 新增测试防止按订单最新 outbox 重试逻辑回潮。

## 14.12 TASK-002F1 验货幂等批次隔离与详情闭环整改

### 14.12.1 inspection 幂等 hash

1. inspection 幂等 hash 必须独立于 stock outbox event hash。
2. `receipt_batch_no` 是验货业务身份字段，必须参与 inspection payload hash。
3. stock outbox 中 `receipt_batch_no` 是库存同步 event_key 易变字段，不参与 receipt outbox event_key；该规则不得套用到 inspection。
4. 同一 `subcontract_id + idempotency_key + 同一 receipt_batch_no + 同一语义 payload` 返回第一次验货结果。
5. 同一 `subcontract_id + idempotency_key + 不同 receipt_batch_no` 必须返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
6. 同一 `subcontract_id + idempotency_key + 同一 receipt_batch_no + 不同数量/扣款/备注 payload` 必须返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
7. inspection hash 继续保留 Decimal 语义归一化。

### 14.12.2 详情接口闭环

1. `GET /api/subcontract/{id}` 必须返回 `inspections[]`。
2. `inspections[]` 必须展示验货批次、数量、扣款和金额快照。
3. 详情金额必须来自 `ly_subcontract_inspection` 或主单汇总字段。
4. `ly_subcontract_receipt` 的旧金额字段视为历史兼容字段，不作为 TASK-002F 后金额事实源。

### 14.12.3 迁移自洽

1. TASK-002F 验货字段必须通过独立 Alembic migration 补齐。
2. 禁止只修改已存在的 TASK-002C revision 作为验货字段唯一来源。
3. 已执行 TASK-002C revision 的数据库必须能通过 TASK-002F 独立迁移补齐验货字段和索引。
4. 空库升级到 head 必须包含验货字段和索引。

### 14.12.4 公式口径清理

1. 唯一有效加工费公式：`gross_amount = inspected_qty × subcontract_rate`。
2. `gross_amount = accepted_qty × subcontract_rate` 是历史旧口径，已废弃。
3. `accepted_qty` 只代表合格数量，不作为加工费总额基数。
4. TASK-006 加工厂对账必须引用 inspection 金额事实。

## 14.13 TASK-002F2 并发验货不超量与详情鉴权顺序

### 14.13.1 并发验货

1. 两个请求同时验同一 `receipt_batch_no` 时，最终累计验货数量不得超过该批次已同步回料数量。
2. 并发验货必须使用数据库事务锁或等效数据库并发控制，禁止用进程内全局锁替代。
3. 剩余可验数量必须在锁内重新计算。
4. 插入 inspection、更新主单汇总、写状态日志必须与剩余可验校验在同一事务内完成。
5. 并发冲突不得返回成功壳，不得写超量 inspection。
6. 必须补充两个独立 session/client 的并发回归测试。
7. SQLite 无法验证真实行锁时，必须增加 PostgreSQL 集成测试或明确 skip，并保留服务层防回归测试。

### 14.13.2 详情接口鉴权顺序

1. `GET /api/subcontract/{id}` 必须先完成订单级资源权限校验。
2. 权限通过后才允许读取 `inspections[]/receipts[]/status_logs[]/latest_outbox` 等子表明细。
3. 无权限或权限源不可用时不得读取子表明细。
4. 该规则作为外发详情接口统一安全口径，后续 TASK-002G 前端只能依赖鉴权后的详情响应。

### 14.13.3 迁移确认

1. TASK-002F 验货字段确认需要独立 Alembic migration。
2. 已执行 TASK-002C revision 的环境必须能通过 TASK-002F 独立迁移补齐验货字段与索引。
3. 空库升级到 head 必须包含 TASK-002F 验货字段与索引。

## 14.14 TASK-002F3 PostgreSQL 并发测试门禁

## 14.15 TASK-002G 前端状态与权限联动

### 14.15.1 前端权限来源

1. 外发前端按钮权限以 `/api/auth/actions?module=subcontract` 或资源级 actions 接口返回的 `button_permissions` 为准。

## 14.16 TASK-002H 对账数据出口

TASK-002H 负责把外发验货金额事实输出给 TASK-006 加工厂对账单。结算明细事实源以 `ly_subcontract_inspection` 为准，订单主表金额只作为汇总展示，不作为唯一结算行事实源。

必须补齐字段：

| 表名 | 字段 | 说明 |
| --- | --- | --- |
| `ly_subcontract_inspection` | `settlement_status` | `unsettled/statement_locked/settled/adjusted/cancelled` |
| `ly_subcontract_inspection` | `statement_id` | TASK-006 对账单 ID，当前阶段 nullable、无强制 FK |
| `ly_subcontract_inspection` | `statement_no` | TASK-006 对账单号快照 |
| `ly_subcontract_inspection` | `settlement_line_key` | 结算明细唯一键，建议 `subcontract_inspection:{inspection_id}` |
| `ly_subcontract_inspection` | `settlement_locked_by` | 锁定人 |
| `ly_subcontract_inspection` | `settlement_locked_at` | 锁定时间 |
| `ly_subcontract_inspection` | `settled_by` | 确认结算人，TASK-006 确认时写 |
| `ly_subcontract_inspection` | `settled_at` | 确认结算时间，TASK-006 确认时写 |
| `ly_subcontract_inspection` | `settlement_request_id` | 锁定/释放请求 ID |

对账候选规则：

1. 候选粒度为 inspection 行，不是 order 行。
2. inspection 必须 `status='inspected'` 且 `settlement_status='unsettled'`。
3. 对应回料批次必须已完成 ERPNext Material Receipt 同步，即 receipt `sync_status='succeeded'` 且 `stock_entry_name` 非空。
4. 外发单必须非取消、`resource_scope_status='ready'` 且 `company` 非空。
5. 候选金额读取 inspection 上的 `gross_amount/deduction_amount/net_amount`，不得重新计算或读取 receipt 历史金额字段。
6. 候选查询必须按当前用户 company/supplier/item 资源权限过滤，权限源不可用必须 fail closed。
7. 锁定以 inspection 行为粒度；同一 inspection 不能被两个 statement 锁定。
8. `statement_locked` 行只允许由匹配 statement release，`settled` 行不得 release。
9. TASK-002H 不创建 `factory_statement` 表，不创建 ERPNext Purchase Invoice，不写 GL。
10. 列表同步状态只展示最新发料/最新回料 outbox 摘要；多批次同步流水保留在详情页或后续独立同步流水页。


## 14.17 TASK-002H1 结算幂等重放漏洞整改

TASK-002H1 修复结算锁定/释放使用 inspection 可变字段作为幂等历史的问题。结算幂等历史必须进入 append-only 操作表，释放操作不得覆盖锁定操作的幂等凭据。

新增表：`ly_schema.ly_subcontract_settlement_operation`。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `operation_type` | `lock/release` |
| `idempotency_key` | 请求幂等键，长度 128 |
| `request_hash` | `operation_type + statement_id + statement_no + sorted(inspection_ids)` 的语义 hash |
| `statement_id/statement_no` | 对账单标识 |
| `inspection_ids_json` | 请求明细 ID 快照 |
| `result_status` | `succeeded/conflict/failed` |
| `affected_inspection_ids_json` | 实际影响明细 ID 快照 |
| `response_json` | 脱敏成功响应快照 |
| `operator/request_id/created_at` | 操作追踪字段 |

规则：

1. `operation_type + idempotency_key` 唯一。
2. 同一 key 同一 hash 返回首次响应，不再修改 inspection。
3. 同一 key 不同 hash 返回 `SUBCONTRACT_SETTLEMENT_IDEMPOTENCY_CONFLICT`。
4. `lock K -> release R -> retry lock K` 不得重新锁定已释放行。
5. `release R -> retry release R` 不得再次修改行。
6. 操作表写入与 inspection 状态变更必须同事务提交。
7. 操作表写入失败必须 rollback 状态变更。
8. `inspection.settlement_request_id` 不再作为幂等历史权威字段，只能作为最后操作 request id 或追踪字段。
9. `idempotency_key` 在 Schema、模型、迁移中统一长度 128。
10. `settlement-candidates.summary` 明确定义为筛选条件全量汇总；如保留分页汇总，必须命名为 `page_summary`，并另给 `filter_summary`。


## 14.18 TASK-002H2 结算并发幂等 Replay 整改

TASK-002H2 修复同一 `operation_type + idempotency_key` 并发请求在 operation 唯一约束处返回 `DATABASE_WRITE_FAILED` 的问题。

规则：

1. lock/release 在首次读取 operation 未命中后，拿到 inspection 行锁后必须再次读取 operation。
2. 第二次读取命中且 hash 相同，必须 replay 首次响应，不得修改 inspection。
3. 第二次读取命中但 hash 不同，返回 `SUBCONTRACT_SETTLEMENT_IDEMPOTENCY_CONFLICT`。
4. 写 operation 时如撞 `operation_type + idempotency_key` 唯一约束，必须 reload operation 并按 hash 决定 replay 或 conflict。
5. 同 key replay 不得追加第二条 operation，不得再次修改 inspection。
6. lock/release 响应必须包含 `operation_id/idempotency_key/idempotent_replay`。
7. 首次成功 `idempotent_replay=false`，幂等回放 `idempotent_replay=true`。
8. `settlement_request_id` 保留为兼容或最近请求标识，不是完整幂等历史。
9. PostgreSQL 有 `POSTGRES_TEST_DSN` 时必须补同 key 并发测试；无 DSN 时允许明确 skip。
