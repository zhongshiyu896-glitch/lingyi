# Sprint3 架构建议

- 文档版本：V1.0
- 输出时间：2026-04-16
- 输入依据：Sprint2 本地封版复盘、Sprint2 架构规范、任务链证据与审计记录
- 结论口径：本建议用于 Sprint3 任务设计与审计前置，不等同生产发布决策。

## 1. Sprint 3 总原则

1. 先平台闭环，再生产发布。
2. 先真实 ERPNext 联调，再开放业务写入口。
3. 先公共基座回迁，再新增复杂业务功能。
4. 所有模块继续坚持 fail-closed 默认策略。
5. 所有任务继续执行“任务单审计通过后再实现”的串行门禁。

## 2. 强制前置门禁（必须满足）

1. 每个任务必须声明权限动作（`module:action`）。
2. 每个任务必须声明资源字段与过滤语义。
3. 每个 ERPNext 依赖必须使用 TASK-008 fail-closed adapter。
4. 每个 outbox 必须使用 TASK-009 状态机规范（event_key/claim/lease/retry/dry-run/diagnostic）。
5. 每个前端模块必须接入 TASK-010 门禁框架并声明 `fixture.positive` + `fixture.negative`。
6. 每个写入口必须同时具备安全审计与操作审计。
7. 每个任务必须产出交付证据与本地基线锚点（HEAD/验证命令/禁改扫描）。
8. 每个任务必须明确“禁止范围”，并提供禁改扫描命令。

## 3. 架构禁止事项（Sprint 3 全局）

1. 禁止裸连 ERPNext。
2. 禁止 `200 + 空数据` 伪成功。
3. 禁止前端绕过 FastAPI 直连 ERPNext。
4. 禁止未审计写入口。
5. 禁止未审计 outbox worker。
6. 禁止未审计 dynamic import / Worker / runtime injection 绕过。
7. 禁止在本地封版阶段混入运行产物、测试缓存和非白名单提交。

## 4. 基座回迁优先级

### 4.1 最高优先

1. TASK-009 模板回迁到 TASK-002/003/004/006 的 outbox 实现。
2. TASK-008 adapter 在遗留 ERPNext 调用点全量收敛。
3. TASK-010 contract engine 与业务模块 contract 配置统一化（减少模块私有脚本漂移）。

### 4.2 次优先

1. TASK-007 权限资源字段声明自动校验（CI 阶段 fail-closed）。
2. 审计证据结构标准化（任务维度自动汇总轮次/阻断/闭环）。

## 5. Sprint 3 候选方向

1. REL-004 平台 CI / required check 闭环。
2. REL-005 ERPNext 生产联调。
3. TASK-013 质量管理增强。
4. TASK-014 销售库存联动增强。
5. TASK-015 Outbox 模板回迁。
6. TASK-016 权限矩阵生产化。
7. TASK-017 前端门禁 AST 深化。

## 6. 里程碑建议

### M1 平台与联调先行

- 目标：完成 REL-004 + REL-005。
- 出口：平台 required check 和生产联调证据可复验。

### M2 基座回迁

- 目标：完成 TASK-015 + TASK-016 + TASK-017（至少第一阶段）。
- 出口：旧模块重复风险显著下降。

### M3 业务增强

- 目标：完成 TASK-013 + TASK-014。
- 出口：质量管理与销售库存联动能力在受控写入口下扩展。

## 7. 风险控制要求

1. 所有生产相关任务必须区分“本地通过”与“生产通过”。
2. 所有外部依赖任务必须提供失败演练证据（timeout/5xx/权限源不可用）。
3. 所有并发语义任务必须提供 PostgreSQL 非 skip 证据。
4. 所有前端写入口任务必须提供反向 fixture 与场景数阈值。

## 8. 结论

Sprint 3 的核心不是扩功能，而是先完成“平台闭环 + 生产联调 + 基座回迁”。在此基础上推进质量与销售库存增强，才能避免 Sprint 1/2 中已暴露问题再次高频复发。
