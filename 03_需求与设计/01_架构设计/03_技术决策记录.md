## ADR-001：BOM 模块唯一契约冻结

- 决策时间：2026-04-11 21:55 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001 BOM 管理
- 审计来源：审计意见书第 3 份，BOM 模块契约不一致
- 结论：采用 `ly_apparel_* + /api/bom/` 作为 BOM 模块唯一契约

### 背景

审计发现 BOM 模块存在两套契约：

1. 模块设计旧版使用 `ly_style / ly_style_bom / ly_style_bom_item / ly_style_bom_version`，接口为 `/api/bom/styles`、`/api/bom/style-boms`。
2. 当前工程实现和任务卡使用 `ly_apparel_bom / ly_apparel_bom_item / ly_bom_operation`，接口为 `/api/bom/`。

该冲突会导致外发加工、生产计划、工票和款式利润模块无法稳定引用 BOM 地基。

### 决策

从本决策生效起，BOM 模块唯一契约如下：

| 类型 | 标准契约 |
| --- | --- |
| BOM 主表 | `ly_schema.ly_apparel_bom` |
| BOM 物料明细表 | `ly_schema.ly_apparel_bom_item` |
| BOM 工序明细表 | `ly_schema.ly_bom_operation` |
| 款式事实源 | ERPNext `public.tabItem` |
| API 前缀 | `/api/bom/` |
| 详情接口 | `GET /api/bom/{bom_id}` |
| 更新接口 | `PUT /api/bom/{bom_id}` |
| 默认接口 | `POST /api/bom/{bom_id}/set-default` |
| 发布接口 | `POST /api/bom/{bom_id}/activate` |
| 展开接口 | `POST /api/bom/{bom_id}/explode` |

废弃以下契约，不允许新开发继续引用：

1. `ly_schema.ly_style`
2. `ly_schema.ly_style_bom`
3. `ly_schema.ly_style_bom_item`
4. `ly_schema.ly_style_bom_version`
5. `/api/bom/styles`
6. `/api/bom/style-boms`

### 约束

1. 款式资料只引用 ERPNext `Item`，不在 FastAPI 自建款式事实表。
2. 外发加工、生产计划、工票和款式利润引用 BOM 时，只能使用 `ly_apparel_bom.id`。
3. 同一 `item_code` 只能有一个 `status='active' AND is_default=true` 的 BOM。
4. PostgreSQL 必须增加部分唯一索引：`UNIQUE(item_code) WHERE is_default = true AND status = 'active'`。
5. `set-default` 必须使用事务和行级锁，防止并发产生多个默认 BOM。

### 影响

1. `01_模块设计_BOM管理.md` 已更新到 V1.1。
2. `当前 sprint 任务清单.md` 与 `TASK-001_BOM管理_工程任务单.md` 保持 `ly_apparel_* + /api/bom/`。
3. 如工程代码中仍存在 `ly_style_*` 迁移或接口，工程师必须迁移或废弃。

### 验收

1. 设计目录内不再出现 `/api/bom/styles` 或 `/api/bom/style-boms` 作为有效新契约。
2. BOM 模块设计、任务单、代码实现均引用 `ly_apparel_bom / ly_apparel_bom_item / ly_bom_operation`。
3. 审计复查时，BOM 契约不一致问题应关闭。

## ADR-002：BOM 写接口鉴权、权限动作与敏感操作审计

- 决策时间：2026-04-12 00:00 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001A BOM 鉴权权限审计整改
- 审计来源：审计意见书第 4 份，BOM 写接口无鉴权、无真实操作者、无敏感操作审计
- 结论：BOM 写接口必须使用真实当前用户、权限动作校验和敏感操作审计日志

### 背景

审计发现 BOM 写接口仍存在以下问题：

1. `create_bom`、`update_bom_draft` 使用 `operator="system"`。
2. 发布、停用、设默认接口没有当前用户依赖。
3. BOM 写接口没有动作级权限校验。
4. 敏感写操作没有审计日志。

该问题会绕过 ERPNext/衣算云权限语义，不允许进入验收。

### 决策

1. BOM 写接口统一接入 `get_current_user()`。
2. 业务操作者必须来自真实登录用户或受控服务账号。
3. 禁止业务路径使用 `operator="system"` 代表用户。
4. BOM 写接口必须校验动作权限。
5. BOM 敏感写操作必须写入 `ly_schema.ly_operation_audit_log`。

### 权限动作

| 动作 | 权限码 |
| --- | --- |
| 创建 BOM | `bom:create` |
| 更新 BOM | `bom:update` |
| 发布 BOM | `bom:publish` |
| 停用 BOM | `bom:deactivate` |
| 设置默认 BOM | `bom:set_default` |

### 审计动作

以下动作必须审计：

1. `bom:create`
2. `bom:update`
3. `bom:publish`
4. `bom:deactivate`
5. `bom:set_default`

### 验收

1. 无登录态调用 BOM 写接口返回 `AUTH_UNAUTHORIZED`。
2. 无权限调用 BOM 写接口返回 `AUTH_FORBIDDEN`。
3. BOM 写接口不再出现 `operator="system"`。
4. 创建和更新写入真实 `created_by/updated_by`。
5. 发布、停用、设默认生成审计日志。

## ADR-003：权限来源临时方案与生产方案

- 决策时间：2026-04-12 09:49 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001B 权限来源升级 ERPNext
- 审计来源：审计意见书第 5 份，权限动作来源仍是 FastAPI 静态角色映射和环境变量
- 结论：Sprint 1 允许静态权限映射作为临时方案，生产前必须升级为 ERPNext 权限权威来源

### 背景

TASK-001A 已接入当前用户、权限动作和敏感操作审计，但审计指出权限动作来源仍是 FastAPI 静态角色映射和环境变量，不是 ERPNext `Role / User Permission` 或 `/api/auth/` 权限聚合结果。

### 决策

1. `permissions.py` 静态角色映射只允许作为 Sprint 1 临时方案。
2. 生产环境必须使用 ERPNext `Role / User Permission / Workflow` 作为权限权威来源。
3. FastAPI 必须提供 `/api/auth/me` 和 `/api/auth/actions` 聚合接口。
4. Vue3 前端按钮权限必须从 FastAPI 权限动作接口读取。
5. 后端写接口继续强制校验动作权限，不能只依赖前端按钮显示。

### 环境约束

| 环境 | 允许权限来源 |
| --- | --- |
| development | `static` 或 `erpnext` |
| sprint | `static` 或 `erpnext` |
| production | 仅 `erpnext` |

当 `APP_ENV=production` 且 `LINGYI_PERMISSION_SOURCE!=erpnext` 时，应用必须启动失败。

### 验收

1. `GET /api/auth/me` 返回当前用户和角色。
2. `GET /api/auth/actions?module=bom` 返回 BOM 动作权限。
3. 前端 BOM 按钮从权限接口读取。
4. 生产环境禁用静态权限来源。

## ADR-004：BOM 读接口和展开接口必须执行后端资源级鉴权

- 决策时间：2026-04-12 10:05 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001C BOM 读接口资源级鉴权
- 审计来源：审计意见书第 6 份，BOM 读接口和展开接口未做后端鉴权
- 结论：BOM 读接口、详情接口、展开接口必须执行 `bom:read` 后端资源级鉴权

### 背景

BOM 用料、工价、展开结果属于敏感经营数据。前端 `canRead` 只能控制显示，不能替代后端权限校验。

### 决策

1. `GET /api/bom/` 必须校验 `bom:read`。
2. `GET /api/bom/{bom_id}` 必须校验 `bom:read` 和目标 BOM 的 `item_code` 资源权限。
3. `POST /api/bom/{bom_id}/explode` 必须校验 `bom:read` 和目标 BOM 的 `item_code` 资源权限。
4. BOM 列表只能返回当前用户有权读取的 BOM。
5. 无登录态返回 `AUTH_UNAUTHORIZED`。
6. 无读权限返回 `AUTH_FORBIDDEN`。

### 验收

1. 未登录用户不能调用 BOM 读接口。
2. 无 `bom:read` 用户不能调用 BOM 读接口。
3. 有 `bom:read` 但无目标 `item_code` 权限的用户不能读取该 BOM。
4. 展开接口与详情接口使用同一资源级读权限。

## ADR-005：ERPNext User Permission 查询失败必须 Fail Closed

- 决策时间：2026-04-12 10:23 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001D ERPNext 权限 Fail Closed 整改
- 审计来源：审计意见书第 7 份，ERPNext User Permission 查询失败时 fail open
- 结论：ERPNext User Permission 查询失败不得返回空权限列表，必须 fail closed

### 背景

审计发现 `get_user_permissions()` 在 ERPNext 权限查询失败时返回 `[]`。后续资源级权限判断会把空列表理解成“用户没有 Item 限制”，导致有角色动作权限的用户可能访问全部 BOM。

该问题会泄露 BOM 用料、工价、损耗率和展开结果等敏感经营数据，必须阻断。

### 决策

1. ERPNext `User Permission` 查询失败、超时、数据库异常、REST API 异常或返回结构异常时，必须拒绝访问。
2. `get_user_permissions()` 禁止在异常时返回 `[]`。
3. 必须明确区分“ERPNext 查询成功但无 User Permission 限制”和“ERPNext 权限查询失败”。
4. 权限来源不可用时统一返回 `PERMISSION_SOURCE_UNAVAILABLE`。
5. BOM 列表、详情、展开接口必须继承 fail closed。
6. 生产环境 `/api/auth/actions` 不得在 ERPNext 权限失败时用静态权限兜底。

### 结果语义

| 场景 | 处理方式 |
| --- | --- |
| ERPNext 查询成功，返回 0 条 User Permission | 允许按“无资源限制”继续判断动作权限 |
| ERPNext 查询成功，返回 N 条 User Permission | 按返回记录限制 `item_code` / `company` |
| ERPNext 查询失败 | 返回 `PERMISSION_SOURCE_UNAVAILABLE`，不返回业务数据 |

### 验收

1. 模拟 ERPNext 权限查询失败时，`GET /api/bom/` 不返回 BOM 数据。
2. 模拟 ERPNext 权限查询失败时，`GET /api/bom/{bom_id}` 不返回 BOM 详情。
3. 模拟 ERPNext 权限查询失败时，`POST /api/bom/{bom_id}/explode` 不返回展开结果。
4. 代码中不存在用 `except Exception: return []` 掩盖权限查询失败的逻辑。

## ADR-006：BOM 权限拒绝必须形成安全审计闭环

- 决策时间：2026-04-12 10:35 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001E BOM 权限拒绝安全审计整改
- 审计来源：审计意见书第 8 份，401/403 权限拒绝未形成安全审计闭环
- 结论：BOM 模块认证失败、权限拒绝、资源级越权和权限来源不可用事件必须写入安全审计日志或持久化审计记录

### 背景

审计确认 TASK-001D 已修复 fail open 高危问题，但指出当前 401/403 权限拒绝只返回客户端，未形成审计闭环。

BOM 用料、工价、损耗率和展开结果属于敏感经营数据，越权尝试必须可追踪、可复查、可统计。

### 决策

1. BOM 模块所有 `AUTH_UNAUTHORIZED` 必须记录安全审计事件。
2. BOM 模块所有 `AUTH_FORBIDDEN` 必须记录安全审计事件。
3. `PERMISSION_SOURCE_UNAVAILABLE` 必须记录安全审计事件。
4. 资源级越权必须记录 `resource_type`、`resource_id`、`resource_no`。
5. 审计记录必须包含 `request_id`、`user_id`、`action`、`deny_reason`、`ip_address`、`user_agent`。
6. 审计日志禁止记录完整 Token、Cookie、密码、ERPNext API Secret。
7. 审计写入失败不得导致权限放行。

### 持久化方案

优先新增 `ly_schema.ly_security_audit_log` 独立记录安全审计事件，避免与业务操作审计 `ly_operation_audit_log` 混用。

如工程实现阶段决定复用现有审计表，必须保证字段可覆盖以下信息：

1. `event_type`
2. `module`
3. `action`
4. `resource_type`
5. `resource_id`
6. `resource_no`
7. `user_id`
8. `deny_reason`
9. `request_id`
10. `ip_address`
11. `user_agent`
12. `created_at`

### 验收

1. 未登录访问 `GET /api/bom/` 返回 401 且产生安全审计记录。
2. 无 `bom:read` 访问 `GET /api/bom/` 返回 403 且产生安全审计记录。
3. 无资源权限访问 `GET /api/bom/{bom_id}` 返回 403 且产生资源级安全审计记录。
4. ERPNext 权限来源不可用返回 503 且产生安全审计记录。
5. 安全审计日志不包含完整 Token、Cookie、密码、ERPNext API Secret。

## ADR-007：BOM 写接口异常必须分类，`AUDIT_WRITE_FAILED` 不得作为通用兜底

- 决策时间：2026-04-12 11:01 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001F BOM 写接口异常分类整改
- 审计来源：审计意见书第 9 份，BOM 写接口把所有未知异常统一返回为 `AUDIT_WRITE_FAILED`
- 结论：`AUDIT_WRITE_FAILED` 只能用于审计日志写入失败，业务异常、数据库异常、未知异常必须使用独立错误码

### 背景

审计发现 BOM 写接口仍把所有未知异常统一返回为 `AUDIT_WRITE_FAILED`，会误导排障和审计结论。

如果业务校验失败、数据库写入失败、未知程序异常都显示为审计失败，工程师无法判断真实故障点，审计官也无法确认审计链路是否真的失败。

### 决策

1. `AUDIT_WRITE_FAILED` 只能由审计服务写入失败产生。
2. 业务校验异常必须返回 BOM 业务错误码。
3. 主业务数据库写入异常返回 `DATABASE_WRITE_FAILED`。
4. 主业务数据库读取异常返回 `DATABASE_READ_FAILED`。
5. 未知程序异常返回 `BOM_INTERNAL_ERROR`。
6. 唯一索引冲突如能识别为默认 BOM 冲突，返回 `BOM_DEFAULT_CONFLICT`。
7. 客户端错误响应不得暴露 Python traceback、SQL 原文、数据库连接串、Token、Cookie。

### 错误码边界

| 错误码 | 使用边界 |
| --- | --- |
| `AUDIT_WRITE_FAILED` | 仅审计日志写入失败 |
| `DATABASE_WRITE_FAILED` | BOM 主业务数据库写入失败 |
| `DATABASE_READ_FAILED` | BOM 主业务数据库读取失败 |
| `BOM_INTERNAL_ERROR` | 未知程序异常 |
| `BOM_DEFAULT_CONFLICT` | 默认 BOM 唯一约束冲突 |

### 验收

1. 模拟审计写入失败时，BOM 写接口返回 `AUDIT_WRITE_FAILED`。
2. 模拟数据库写入失败时，BOM 写接口返回 `DATABASE_WRITE_FAILED`。
3. 模拟未知异常时，BOM 写接口返回 `BOM_INTERNAL_ERROR`。
4. 代码中不存在把所有 `Exception` 统一转换为 `AUDIT_WRITE_FAILED` 的逻辑。

## ADR-008：BOM 提交阶段数据库异常必须归类为 `DATABASE_WRITE_FAILED`

- 决策时间：2026-04-12 11:22 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001G BOM 提交阶段数据库异常分类整改
- 审计来源：审计意见书第 10 份，`session.commit()` 阶段数据库写入失败仍落到 `BOM_INTERNAL_ERROR`
- 结论：`flush/commit/事务退出` 阶段的数据库异常均属于主业务数据库写入失败，必须返回 `DATABASE_WRITE_FAILED`

### 背景

审计确认 `AUDIT_WRITE_FAILED` 已限定为审计写入失败专用错误码，但发现 `session.commit()` 阶段的数据库写入失败仍落到 `BOM_INTERNAL_ERROR`。

提交失败是主业务数据库写失败，不是未知异常。若归类为未知异常，会影响排障、告警分级和审计判断。

### 决策

1. `session.flush()`、`session.commit()`、事务上下文退出时触发的数据库异常，统一归类为 `DATABASE_WRITE_FAILED`。
2. `IntegrityError` 如能识别为默认 BOM 部分唯一索引冲突，返回 `BOM_DEFAULT_CONFLICT`。
3. 提交阶段数据库异常不得归类为 `BOM_INTERNAL_ERROR`。
4. 提交阶段数据库异常不得归类为 `AUDIT_WRITE_FAILED`。
5. 提交失败后必须执行 `rollback`，且 `rollback` 失败不得覆盖原始错误码。
6. `snapshot_resource()` 必须纳入统一异常处理，禁止绕过错误信封。
7. 测试环境变量必须在应用模块 import 前完成设置，避免 import-time 顺序脆弱性。

### 错误码边界

| 场景 | 错误码 |
| --- | --- |
| `session.commit()` 数据库写失败 | `DATABASE_WRITE_FAILED` |
| `session.flush()` 数据库写失败 | `DATABASE_WRITE_FAILED` |
| 默认 BOM 唯一索引冲突 | `BOM_DEFAULT_CONFLICT` |
| `snapshot_resource()` 数据库读取失败 | `DATABASE_READ_FAILED` |
| `snapshot_resource()` 未知异常 | `BOM_INTERNAL_ERROR` |

### 验收

1. 模拟 `session.commit()` 抛 `SQLAlchemyError` 时，BOM 写接口返回 `DATABASE_WRITE_FAILED`。
2. 模拟默认 BOM 唯一索引冲突时，返回 `BOM_DEFAULT_CONFLICT`。
3. `snapshot_resource()` 异常返回统一 `{code, message, data}` 错误信封。
4. 测试连续执行两次结果一致，不依赖 import-time 环境变量顺序。

## ADR-009：服务端普通日志必须脱敏，测试环境变量必须强制隔离

- 决策时间：2026-04-12 11:37 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001H 服务端日志脱敏与测试环境隔离整改
- 审计来源：审计意见书第 11 份，普通错误日志泄露 SQLAlchemy 异常原文，测试 `setdefault()` 对外部生产变量不免疫
- 结论：普通应用日志和审计日志不得记录 SQL 原文、SQL 参数、Token、Cookie、密码、Secret；测试环境变量必须强制覆盖并发生在应用 import 前

### 背景

审计复验时确认客户端响应未泄露敏感信息，但服务端普通错误日志仍直接记录 `str(exc)`，出现 `[SQL: UPDATE ...]` 和 SQLAlchemy 异常原文。

同时，测试环境变量使用 `setdefault()`，当外部已存在 `APP_ENV=production`、`LINGYI_PERMISSION_SOURCE=static` 时无法覆盖，导致测试导入阶段失败。

### 决策

1. 普通应用日志禁止直接记录 `str(exc)`。
2. SQLAlchemy 异常不得在普通日志中输出 SQL statement、SQL parameters、数据库连接串。
3. 认证、权限、审计相关日志不得记录完整 Token、Cookie、密码、ERPNext API Secret。
4. 普通日志只记录安全元数据：`error_code`、`exception_type`、`request_id`、`module`、`action`、资源标识和安全数据库错误码。
5. 如无法安全脱敏，错误明细统一写为 `internal error, detail redacted`。
6. 原始异常详情如需保留，只允许在非生产环境通过显式开关进入受限调试日志，生产环境强制禁止。
7. 测试环境变量必须强制覆盖，不允许用 `os.environ.setdefault()` 设置关键测试变量。
8. 测试环境变量设置必须发生在应用模块 import 前。
9. 生产保护逻辑不得删除：非测试入口下 `APP_ENV=production` 且 `LINGYI_PERMISSION_SOURCE=static` 必须启动失败。

### 验收

1. 数据库写失败、回滚失败、审计写入失败场景下，普通日志不包含 `[SQL:]`、`[parameters:]`、SQL statement、SQL parameters。
2. 普通日志和审计日志不包含完整 Token、Cookie、密码、Secret。
3. 外部注入 `APP_ENV=production LINGYI_PERMISSION_SOURCE=static` 时，测试入口仍能覆盖为 test 环境并通过。
4. 非测试入口的生产保护仍有效。

## ADR-010：`X-Request-ID/request_id` 必须白名单规范化

- 决策时间：2026-04-12 11:54 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001I Request ID 白名单校验与脱敏整改
- 审计来源：审计意见书第 12 份，`X-Request-ID/request_id` 未做统一白名单校验或脱敏
- 结论：外部传入的 `X-Request-ID` 不可信，进入日志、审计表和响应头前必须统一白名单规范化

### 背景

审计发现 `X-Request-ID/request_id` 会原样进入错误日志和审计上下文，位置包括 `main.py`、`logging.py`、`audit_service.py`。

如果请求头中包含换行、SQL 片段、Token、Cookie、密码或脚本片段，会造成日志注入、审计污染和敏感信息落库风险。

### 决策

1. 新增统一函数 `normalize_request_id(raw_request_id)`。
2. 白名单字符为 `A-Z`、`a-z`、`0-9`、`_`、`-`、`.`。
3. 长度限制为 1 到 64 个字符。
4. 缺失、超长或不符合白名单的 `request_id` 必须整体丢弃并重新生成。
5. 非法 `request_id` 禁止截断或部分清洗后继续使用。
6. `main.py` 请求中间件是唯一允许读取原始 `X-Request-ID` 的位置。
7. 后续日志、审计和业务模块只能使用 `request.state.request_id`。
8. 响应头 `X-Request-ID` 只返回规范化后的值。
9. 普通日志、`ly_security_audit_log`、`ly_operation_audit_log` 禁止记录非法 `request_id` 原文。

### 验收

1. 合法 `X-Request-ID` 被保留。
2. 缺失、超长、包含换行、SQL 片段、Token、Cookie、密码、脚本片段的 `X-Request-ID` 被替换为安全生成值。
3. 普通日志和审计表不包含非法 `request_id` 原文。
4. 除 `main.py` 请求中间件外，业务代码不得直接读取 `request.headers.get(\"X-Request-ID\")`。

## ADR-011：`request_id` 必须拦截语义敏感词

- 决策时间：2026-04-12 12:07 CST
- 决策类型：架构审计整改
- 关联模块：TASK-001J Request ID 语义敏感词拦截与单测隔离整改
- 审计来源：审计意见书第 13 份，`normalize_request_id()` 仅做字符白名单导致语义敏感 request_id 被放行
- 结论：`request_id` 必须同时满足字符白名单和语义安全校验，命中敏感关键词时整体丢弃并重新生成

### 背景

审计发现 `X-Request-ID: token-password-secret-cookie-authorization` 和 `X-Request-ID: Bearer.abc123_token_secret` 虽然字符合法，但语义上包含凭证、认证、Cookie、密码等敏感关键词，会原样进入响应头和安全审计表。

字符白名单只能防止格式注入，不能防止语义敏感数据落日志。因此必须增加语义敏感词拦截。

### 决策

1. `normalize_request_id()` 必须同时执行字符白名单、长度限制和语义敏感词校验。
2. 语义敏感词校验大小写不敏感。
3. 必须拦截 `authorization`、`bearer`、`token`、`cookie`、`set-cookie`、`password`、`passwd`、`secret`、`session`、`sessionid`、`api-key`、`api_key`、`access-token`、`access_token`、`refresh-token`、`refresh_token`。
4. 命中敏感词时必须整体丢弃原 request_id 并生成新值。
5. 禁止将敏感词 mask、截断或替换后继续作为 request_id。
6. 响应头、普通日志、`ly_security_audit_log`、`ly_operation_audit_log` 只能使用规范化后的 request_id。
7. 单文件 `unittest` 不得依赖 pytest `conftest.py` 自动加载；需要在业务模块 import 前强制配置测试环境。

### 验收

1. `token-password-secret-cookie-authorization` 被替换，不进入响应头、日志和审计表。
2. `Bearer.abc123_token_secret` 被替换，不进入响应头、日志和审计表。
3. 合法业务 request_id 未命中敏感词时可保留。
4. 外部生产变量注入下，`python -m unittest tests.test_request_id_sanitization` 可通过。
5. 非测试入口的生产保护仍有效。

## ADR-012：工票为本地工资事实源，Job Card 为 ERPNext 生产执行同步对象

- 决策时间：2026-04-12 12:21 CST
- 决策类型：模块架构决策
- 关联模块：TASK-003 工票/车间管理
- 结论：工票和工资结算事实保存在 `ly_schema`，ERPNext `Job Card` 仅作为生产工序卡和进度同步对象；工票创建时必须快照计件工价

### 背景

车间工票涉及计件工资、撤销、批量导入、PDA/MES 幂等和日薪汇总，属于服装工厂特有执行层业务。ERPNext `Job Card` 可承载生产工序卡，但不适合承载工票幂等、撤销和工资结算全部细节。

### 决策

1. `ly_schema.ys_workshop_ticket` 是工票事实源。
2. `ly_schema.ys_workshop_daily_wage` 是日薪汇总数据。
3. `ly_schema.ly_operation_wage_rate` 是本厂工序计件工价来源。
4. ERPNext `Job Card` 只作为工序执行对象和进度同步目标。
5. FastAPI 不直接写 ERPNext 数据库表，只通过 ERPNext API 同步 Job Card。
6. 工票创建时必须快照 `unit_wage`，历史工资不随工价档案后续调整自动变化。
7. Job Card 同步失败不得删除或回滚已成功落库的本地工票，应标记 `sync_status=failed` 并支持重试。

### 验收

1. 工票登记后 `ys_workshop_ticket.unit_wage` 有快照值。
2. 修改工价档案后，历史工票工资不变化。
3. Job Card 同步失败时，本地工票保留且有同步失败记录。
4. 工票撤销通过新增 reversal 记录完成，不物理删除原工票。

## ADR-013：工票写接口必须按 Job Card 派生资源做权限校验

- 决策时间：2026-04-12 13:17 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003A 工票写接口资源级权限整改
- 审计来源：审计意见书第 14 份，车间写接口缺少资源级 Item / Job Card 权限校验
- 结论：工票写接口必须同时校验动作权限和由 ERPNext `Job Card / Work Order` 派生出的 `item_code/company` 资源权限

### 背景

审计探针复现：仅授权 `ITEM-B` 的用户可以成功登记 `ITEM-A` 工票。该问题说明工票写接口只校验了 `workshop:*` 动作权限，未校验目标业务资源权限。

工票会影响员工工资、生产进度和 Job Card 同步，属于高敏业务写入。前端传入的 `item_code` 不可信，不能作为权限判断依据。

### 决策

1. 工票登记、撤销、批量导入、Job Card 同步必须先读取 ERPNext `Job Card`。
2. 必须由 `Job Card.work_order` 继续读取 ERPNext `Work Order`，并派生真实 `item_code/company`。
3. 前端传入的 `item_code` 只能做一致性校验，不能作为权限依据。
4. 用户必须同时具备对应 `workshop:*` 动作权限和目标 `item_code/company` 资源权限。
5. 仅授权 `ITEM-B` 的用户不得写入 `ITEM-A` 工票。
6. 资源权限不足时不得写工票、不得更新日薪、不得同步 Job Card。
7. 批量导入必须逐行校验资源权限；越权行进入 `failed_items` 并写安全审计。
8. 工价维护涉及具体 `item_code` 时必须校验该 `item_code` 资源权限；通用工价必须要求全局工价维护权限。
9. ERPNext 权限来源不可用时必须 fail closed，返回 `PERMISSION_SOURCE_UNAVAILABLE`。

### 验收

1. 仅授权 `ITEM-B` 的用户登记 `ITEM-A` 工票返回 `AUTH_FORBIDDEN`。
2. 越权登记不得新增 `ys_workshop_ticket`。
3. 越权登记不得更新 `ys_workshop_daily_wage`。
4. 越权登记不得调用 ERPNext Job Card 同步。
5. 批量导入中越权行失败，授权行可成功。
6. 所有资源级拒绝写入 `ly_security_audit_log`。

## ADR-014：工价读取和维护必须按 Item/Company 资源权限闭环

- 决策时间：2026-04-12 13:52 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003B 工价资源权限闭环整改
- 审计来源：审计意见书第 15 份，工价读取和创建资源权限未闭环
- 结论：工价属于工资/成本敏感数据，读取、创建、停用必须同时满足动作权限和 `item_code/company` 资源权限；Company-only 不得替代 Item 权限

### 背景

审计复现：仅授权 `ITEM-B` 的用户可以读取 `ITEM-A` 工价；只有 Company 权限、没有 Item 权限的用户也能创建指定 `ITEM-Z` 的工价。

工价直接影响员工工资和款式成本，属于敏感经营数据。只校验 Company 或动作权限不足以证明用户可访问某个款式工价。

### 决策

1. `GET /api/workshop/wage-rates` 必须在数据库查询层按 `item_code/company` 做资源过滤。
2. 显式查询未授权 `item_code` 时返回 `AUTH_FORBIDDEN`。
3. 创建或停用指定 `item_code` 工价，必须校验目标 Item 权限。
4. Company-only 权限不得替代 Item 权限。
5. `POST /api/workshop/wage-rates` 必须完成 `item_code -> company` 解析。
6. 无法解析 company 时返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`，禁止 fail open。
7. 通用工价必须使用独立全局权限：读取需 `workshop:wage_rate_read_all`，维护需 `workshop:wage_rate_manage_all`。
8. 权限来源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`。
9. 工价资源级拒绝必须写入 `ly_security_audit_log`。

### 验收

1. 仅授权 `ITEM-B` 的用户读取工价列表时，不返回 `ITEM-A` 工价。
2. 仅授权 `ITEM-B` 的用户显式查询 `ITEM-A` 工价时，返回 `AUTH_FORBIDDEN`。
3. 只有 Company 权限、没有 Item 权限的用户创建 `ITEM-Z` 工价时，返回 `AUTH_FORBIDDEN`。
4. 无法解析 `item_code -> company` 时，返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`。
5. 通用工价读取和维护必须具备全局工价权限。

## ADR-015：工票批量导入系统级异常不得包装为行级失败

- 决策时间：2026-04-12 14:22 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003C 工票批量导入系统级异常边界整改
- 审计来源：审计意见书第 16 份，批量导入将 `DatabaseWriteFailed` 包装成 `200/code=0` 的普通失败行
- 结论：批量导入必须区分行级业务失败和系统级失败；数据库、权限源、审计、ERPNext 服务、未知异常必须向上抛出标准错误响应

### 背景

审计指出 `POST /api/workshop/tickets/batch` 会把 `DatabaseWriteFailed` 这类系统级异常包装成 `failed_items`，并返回 `200/code=0`。这会让调用方误判为批次请求成功，也会让运维监控无法感知系统故障。

### 决策

1. 行级业务失败可进入 `failed_items`，批次可继续处理其他合法行。
2. 系统级失败必须中断批次，返回非 200 标准错误响应。
3. `DATABASE_WRITE_FAILED`、`DATABASE_READ_FAILED`、`PERMISSION_SOURCE_UNAVAILABLE`、`AUDIT_WRITE_FAILED`、`ERPNEXT_SERVICE_UNAVAILABLE`、`WORKSHOP_INTERNAL_ERROR` 禁止进入 `failed_items`。
4. 缺少批量导入动作权限时，整个请求返回 `AUTH_FORBIDDEN`，不进入行处理。
5. 未登录时，整个请求返回 `AUTH_UNAUTHORIZED`，不进入行处理。
6. 单行资源级 `AUTH_FORBIDDEN` 可进入 `failed_items`，但必须写安全审计。
7. 系统级异常响应不得包含 `success_count/failed_count`，避免调用方误判为业务行失败。

### 验收

1. 模拟数据库写失败时，批量导入返回 `DATABASE_WRITE_FAILED`，不是 `200/code=0`。
2. 模拟权限来源不可用时，批量导入返回 `PERMISSION_SOURCE_UNAVAILABLE`，不是 `failed_items`。
3. 模拟审计写入失败时，批量导入返回 `AUDIT_WRITE_FAILED`，不是 `failed_items`。
4. 行级业务错误仍可进入 `failed_items` 并允许合法行成功。
5. 行级资源越权进入 `failed_items` 且写安全审计。

## ADR-016：Job Card 同步必须采用 Outbox / After-Commit / 异步重试

- 决策时间：2026-04-12 14:38 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003D Job Card 同步 Outbox 整改
- 审计来源：审计意见书第 17 份，Job Card 同步发生在本地事务提交前，存在 ERPNext 已更新但本地工票/审计回滚风险
- 结论：工票写接口不得在本地事务提交前调用 ERPNext Job Card 更新，必须先持久化本地工票、日薪、审计和 outbox，再由 after-commit/worker 异步同步 ERPNext

### 背景

工票写入和 ERPNext Job Card 更新属于跨系统写入。如果在本地事务提交前先调用 ERPNext，可能出现 ERPNext 更新成功但本地工票事务回滚的情况，造成生产进度和工资事实不一致。

### 决策

1. `ys_workshop_ticket`、`ys_workshop_daily_wage`、操作审计、`ys_workshop_job_card_sync_outbox` 必须在同一本地事务内写入。
2. ERPNext Job Card 更新必须发生在本地事务提交之后。
3. 本地事务提交失败、操作审计失败、outbox 写入失败时，禁止调用 ERPNext。
4. outbox 是待同步任务事实，`ys_workshop_job_card_sync_log` 是每次尝试日志。
5. Worker 必须使用最终态覆盖方式同步 Job Card 完成数量，不得增量累加。
6. Worker 失败必须记录 attempts、next_retry_at、last_error_code，并支持重试和 dead 状态。
7. 服务账号必须最小权限，只允许读取 Job Card / Work Order 和更新 Job Card 完成数量。
8. 服务账号凭证不得进入日志、审计表或 sync_log。

### 验收

1. 工票登记、撤销、批量导入在本地事务提交前不调用 ERPNext。
2. 本地 commit 失败时不调用 ERPNext。
3. 审计写入失败时不调用 ERPNext。
4. 本地事务成功后产生 pending outbox。
5. Worker 成功后 outbox 变为 `succeeded` 并写 sync_log。
6. Worker 重试不会导致 ERPNext Job Card 完成数量重复累加。

## ADR-017：Outbox `event_key` 必须采用短前缀 + 完整 Hash，禁止截断 Digest

- 决策时间：2026-04-12 15:09 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003E Outbox Event Key 截断碰撞整改
- 审计来源：审计意见书第 18 份，`event_key` 先拼 `job_card:digest` 再截断导致长 Job Card 截掉 digest
- 结论：outbox `event_key` 必须采用固定短前缀 + 完整 SHA-256 摘要，禁止任何截断 digest 的实现

### 背景

审计发现 outbox `event_key` 当前先拼接 `job_card:digest` 再截断到 140 字符。当 `job_card` 长度达到 138/139/140 字符时，digest 可能被截掉，不同同步事件会生成相同 key，导致 outbox 误判任务已存在，最终漏同步 ERPNext Job Card 完成数量。

### 决策

1. `event_key` 禁止使用 `f\"{job_card}:{digest}\"[:140]` 或任何先拼接后截断模式。
2. `event_key` 必须采用固定短前缀 + 完整 SHA-256 摘要。
3. 推荐格式：`wjc:{sha256_hex}`，总长度 68。
4. hash 输入必须包含 `job_card`、`local_completed_qty`、`source_type`、排序后的 `source_ids`。
5. `source_ids` 必须排序，保证同一集合生成同一 key。
6. `local_completed_qty` 必须数值规范化，保证 `1`、`1.0`、`1.000000` 语义相同时 key 相同。
7. `event_key` 不保存完整 `job_card` 明文；排查使用 outbox 独立字段 `job_card`。
8. 旧 pending/failed outbox 不得因迁移被删除。

### 验收

1. 138/139/140/200 字符 Job Card 下，不同同步事件生成不同 `event_key`。
2. `event_key` 长度稳定小于等于 140。
3. `event_key` 不包含完整 `job_card` 明文。
4. 旧算法会碰撞的样例在新算法下不碰撞。
5. 重复同一业务事件仍返回同一 `event_key`。

## ADR-018：内部 Job Card 同步 Worker 接口必须服务账号化

- 决策时间：2026-04-12 15:21 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003F 内部 Job Card 同步 Worker 接口权限整改
- 审计来源：审计意见书第 19 份，`/api/workshop/internal/job-card-sync/run-once` 可能被普通车间管理角色触发
- 结论：内部 Worker 批量写同步接口必须与普通业务同步权限分离，只允许服务账号、系统级集成账号或 System Manager 调用

### 背景

TASK-003D 已将 ERPNext Job Card 写同步从工票登记事务内移到 outbox/worker 模式，TASK-003E 已修复 outbox `event_key` 截断碰撞。审计继续指出，内部 Worker 触发接口如果可由普通车间管理角色调用，会扩大批量写 ERPNext Job Card 的权限面，破坏内部调度和服务账号边界。

### 决策

1. `POST /api/workshop/internal/job-card-sync/run-once` 定义为内部接口，不是普通前端业务接口。
2. 内部 Worker 接口必须要求 `workshop:job_card_sync_worker`。
3. `workshop:job_card_sync_worker` 只能授予服务账号、系统级集成账号或 `System Manager`。
4. 普通 `Workshop Manager / Workshop User / Production Manager` 不得调用内部 Worker。
5. `workshop:job_card_sync` 仅用于用户手动重试单个 Job Card 同步，不得替代内部 Worker 权限。
6. 普通用户的 `/api/auth/actions?module=workshop` 不得返回 `workshop:job_card_sync_worker`。
7. 生产环境保留 HTTP 内部 Worker 接口时，必须支持 `ENABLE_INTERNAL_WORKER_API` 开关；未开启时返回 `INTERNAL_API_DISABLED`。
8. 未登录、权限不足、权限来源不可用、内部接口关闭时，不得锁定、更新或重试 outbox，也不得调用 ERPNext。
9. 权限拒绝必须写安全审计，成功执行 Worker 必须写操作审计。
10. 服务账号凭证、Authorization、Cookie、secret、password 不得进入日志或审计表。

### 验收

1. 普通 `Workshop Manager` 调用内部 Worker 返回 `AUTH_FORBIDDEN`。
2. 只有 `workshop:job_card_sync` 的用户调用内部 Worker 返回 `AUTH_FORBIDDEN`。
3. 未登录调用内部 Worker 返回 `AUTH_UNAUTHORIZED`。
4. 权限来源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`。
5. 生产环境内部接口未开启时返回 `INTERNAL_API_DISABLED`。
6. 未授权调用不得修改 outbox，不得写 sync_log，不得调用 ERPNext。
7. 服务账号具备 `workshop:job_card_sync_worker` 时可以处理 pending outbox。
8. 普通用户动作权限列表不包含 `workshop:job_card_sync_worker`。

## ADR-019：服务账号必须按 ERPNext User Permission 执行最小资源权限

- 决策时间：2026-04-12 15:40 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003G 服务账号最小权限策略整改
- 审计来源：审计意见书第 21 份，服务账号仍是“全模块全资源”硬编码授权并跳过 item/company 资源校验
- 结论：服务账号不得默认全模块全资源；内部 Worker 每处理一条 outbox 前，必须按 ERPNext User Permission 校验 `company/item_code`

### 背景

TASK-003F 已将内部 Worker API 限定为服务账号或系统级集成账号调用。但如果服务账号本身被硬编码为全模块全资源，仍会绕过最小权限原则。服务账号一旦被误用或凭证泄露，就可能批量处理全部公司、全部款式的 Job Card 同步。

### 决策

1. 删除或废弃服务账号默认全模块全资源授权。
2. 服务账号身份必须来自 ERPNext `User + Role`，不得由请求参数声明。
3. 服务账号动作权限必须具备 `workshop:job_card_sync_worker`。
4. 服务账号资源权限必须来自 ERPNext `User Permission`。
5. 服务账号必须明确具备目标 `Company` 和 `Item` 权限，才能处理对应 outbox。
6. 没有明确 Company 或 Item 授权时，默认 fail closed。
7. 禁止用 `*`、`all`、空数组表示全资源。
8. ERPNext 权限来源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`，不得处理 outbox。
9. outbox 缺少 `company/item_code` 时，不得调用 ERPNext。
10. 资源越权 outbox 不得标记为 `succeeded`，不得写成功 sync_log。
11. 服务账号资源越权必须写安全审计。
12. 服务账号成功处理 outbox 必须写操作审计，并记录 `principal/outbox_id/job_card/item_code/company/request_id`。

### 验收

1. 代码中不存在服务账号默认 `all modules/all resources/*` 硬编码授权。
2. 服务账号只有 `workshop:job_card_sync_worker` 但没有 Company/Item 权限时，不处理 outbox。
3. 服务账号授权 `COMPANY-A + ITEM-B` 时，不能处理 `COMPANY-A + ITEM-A`。
4. 服务账号授权 `COMPANY-A + ITEM-B` 时，不能处理 `COMPANY-B + ITEM-B`。
5. 服务账号授权 `COMPANY-A + ITEM-B` 时，可以处理匹配 outbox。
6. 资源越权 outbox 不调用 ERPNext，不写成功 sync_log，不标记 `succeeded`。
7. 权限来源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`。
8. 安全审计和操作审计均不泄露服务账号凭证。

## ADR-020：Outbox Worker 必须先按服务账号 Scope 过滤再 Limit

- 决策时间：2026-04-12 16:02 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003H Outbox 跳过行队头阻塞整改
- 审计来源：审计意见书第 22 份，越权 pending outbox 配合 `list_due(limit)` 固定窗口导致后续有权任务被长期阻塞
- 结论：内部 Worker 主处理查询必须先按服务账号 `company/item_code` scope 做数据库层过滤，再执行排序和 limit；缺 scope 行必须脱离 due pending 队列

### 背景

审计复现：第 1 条 outbox 越权、第 2 条 outbox 有权，`limit=1` 连续两轮都只跳过第 1 条越权行，第 2 条有权行始终未处理，ERPNext 调用 0 次。根因是 Worker 先按 `id asc limit` 取 due pending 窗口，再在 Python 中逐行跳过越权行，导致越权行长期占据队头。

### 决策

1. 新增或改造主处理查询为 `list_due_for_service_account(policy, limit)`。
2. 主处理查询必须先限定 `status/next_retry_at`。
3. 主处理查询必须在数据库层追加 `company in allowed_companies` 和 `item_code in allowed_items`。
4. `order by id asc` 和 `limit batch_size` 必须发生在资源过滤之后。
5. 禁止先 `limit` 再逐行权限过滤。
6. 当前服务账号越权 outbox 不得被锁定、增加 attempts、更新 `locked_by/locked_at` 或标记成功。
7. 缺少 `company/item_code` 的 outbox 必须转为 `dead` 或 `blocked_scope`，并写 `SERVICE_ACCOUNT_RESOURCE_SCOPE_REQUIRED` 安全审计。
8. 服务账号空 scope 不得查询全量 outbox。
9. 权限来源不可用时不得执行 outbox 查询锁定或 ERPNext 调用。
10. 如果保留 `skipped_forbidden_count`，只能来自独立诊断扫描，不得消费主处理 limit。

### 验收

1. 第 1 条 outbox 越权、第 2 条 outbox 有权、`batch_size=1` 时，第一轮 Worker 必须处理第 2 条有权 outbox。
2. 连续两轮 Worker 不得重复只跳过同一条越权队头行。
3. 主处理查询在 SQLAlchemy 查询层包含 `company/item_code` scope 过滤。
4. 越权 outbox 不被当前服务账号锁定或增加 attempts。
5. 缺 scope outbox 不会长期停留在 due pending 队头。
6. 空 scope 服务账号不触发全量扫描。
7. 权限来源不可用时不锁定 outbox、不调用 ERPNext。

## ADR-021：越权 Outbox 诊断必须节流并对安全审计去重

- 决策时间：2026-04-12 16:15 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003I Outbox 越权诊断节流与审计去重整改
- 审计来源：审计意见书第 23 份，越权 outbox 每轮 run-once 重复枚举并写安全审计，存在审计表放大和 Worker 事务变重风险
- 结论：默认 Worker 不扫描越权 outbox；越权诊断必须显式启用、限量扫描，并按 `outbox_id + principal + reason_code + scope_hash` 做冷却期去重

### 背景

TASK-003H 已关闭队头阻塞高危问题，但越权 outbox 仍可能在每轮 run-once 中被重复枚举并写安全审计。多公司、多服务账号场景下，这会造成安全审计表快速膨胀，也会让同步事务承担与实际同步无关的重复诊断负担。

### 决策

1. 默认 `run-once` 只处理当前服务账号有权的 outbox，不扫描越权 outbox。
2. 越权诊断必须显式启用，不能作为主处理默认路径。
3. 越权诊断必须设置单次扫描上限，默认不超过 50 行。
4. 同一 `outbox_id + principal + reason_code + scope_hash` 在冷却期内只能写 1 条安全审计。
5. 推荐冷却期为 6 小时，通过 `WORKSHOP_OUTBOX_DENIAL_AUDIT_COOLDOWN_SECONDS` 配置。
6. 冷却期内重复发现只更新 denial 记录的 `seen_count/last_seen_at`。
7. 冷却期到期、scope 变化或 reason 变化时，才允许再次写安全审计。
8. 越权诊断不得锁定 outbox，不得增加 attempts，不得更新 `next_retry_at`，不得调用 ERPNext。
9. 缺 scope outbox 已转为 `dead/blocked_scope` 后，不得继续进入越权诊断扫描。
10. 去重 key 和审计上下文不得包含 Authorization、Cookie、token、secret、password 明文。

### 验收

1. 连续 3 次默认 `run-once` 不新增同一越权 outbox 的安全审计。
2. 显式诊断首次发现越权 outbox 时写 1 条安全审计。
3. 冷却期内重复诊断同一拒绝事件不重复写安全审计。
4. 冷却期内重复诊断只更新 `seen_count/last_seen_at`。
5. 冷却期到期后再次诊断同一拒绝事件最多新增 1 条安全审计。
6. 越权诊断不锁定 outbox、不增加 attempts、不调用 ERPNext。
7. TASK-003H 的队头阻塞回归用例继续通过。

## ADR-022：内部 Worker Dry-Run 必须可追溯且生产默认禁用

- 决策时间：2026-04-12 16:37 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003J 内部 Worker Dry-Run 操作审计整改
- 审计来源：TASK-003I 审计意见，内部 Worker `dry_run=true` 路径不写操作审计，敏感内部诊断接口存在不可追溯调用
- 结论：dry-run 属于敏感内部诊断行为，授权成功必须写操作审计；生产环境默认禁用，除非显式开启

### 背景

`dry_run=true` 虽不修改 outbox、不调用 ERPNext，但会读取内部同步候选、服务账号 scope、诊断结果和生产同步状态。该能力如果没有操作审计，服务账号或运维调用将不可追溯。

### 决策

1. 生产环境默认禁用 dry-run，除非 `WORKSHOP_ENABLE_WORKER_DRY_RUN=true`。
2. dry-run 被禁用时返回 `WORKSHOP_DRY_RUN_DISABLED`，写安全审计，不查询 outbox。
3. dry-run 授权成功必须写操作审计。
4. dry-run 操作审计失败时返回 `AUDIT_WRITE_FAILED`，不得静默成功。
5. dry-run 不得锁定 outbox，不得增加 attempts，不得更新 `next_retry_at`，不得调用 ERPNext，不得写成功 sync_log。
6. dry-run 操作审计必须记录 `principal/dry_run/batch_size/include_forbidden_diagnostics/would_process_count/request_id`。
7. dry-run 操作审计不得包含 Authorization、Cookie、token、secret、password 明文。
8. dry-run 与 forbidden diagnostics 联用时，仍必须遵守 TASK-003I 的诊断节流和安全审计去重规则。

### 验收

1. 生产环境未开启 dry-run 时返回 `WORKSHOP_DRY_RUN_DISABLED`。
2. dry-run 被禁用时不查询 outbox、不调用 ERPNext、不写操作审计，但写安全审计。
3. 授权 dry-run 成功时新增 1 条操作审计。
4. dry-run 操作审计写入失败时返回 `AUDIT_WRITE_FAILED`。
5. dry-run 不修改 outbox 状态、attempts、锁字段、重试时间。
6. dry-run 不新增成功 sync_log，不调用 ERPNext。
7. dry-run + forbidden diagnostics 仍遵守冷却期去重。

## ADR-023：生产 Dry-Run 禁用判断必须早于 ERPNext 资源权限读取

- 决策时间：2026-04-12 16:53 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003K 生产 Dry-Run 禁用判断前置整改
- 审计来源：TASK-003J 审计意见，生产 dry-run 禁用且 ERPNext 权限源超时时返回 `PERMISSION_SOURCE_UNAVAILABLE`，说明禁用判断晚于权限源读取
- 结论：生产 dry-run 禁用是本地配置门禁；完成身份、动作权限、服务账号主体校验后，必须立即返回 `WORKSHOP_DRY_RUN_DISABLED`，不得读取 ERPNext User Permission 或 outbox

### 背景

生产环境默认禁用 dry-run。该禁用状态应由本地配置决定，不应依赖 ERPNext 权限源是否可用。若禁用判断晚于 ERPNext User Permission 查询，权限源故障会导致禁用 dry-run 返回 `PERMISSION_SOURCE_UNAVAILABLE`，既暴露外部依赖状态，也让禁用门禁不稳定。

### 决策

1. `dry_run=true` 请求先完成 request_id 规范化和当前用户解析。
2. 未登录返回 `AUTH_UNAUTHORIZED`。
3. 缺少 `workshop:job_card_sync_worker` 返回 `AUTH_FORBIDDEN`。
4. 非服务账号、非系统级集成账号、非 `System Manager` 返回 `AUTH_FORBIDDEN`。
5. 通过身份和主体校验后，若生产 dry-run 未开启，立即返回 `WORKSHOP_DRY_RUN_DISABLED`。
6. 返回 `WORKSHOP_DRY_RUN_DISABLED` 前不得读取 ERPNext User Permission。
7. 返回 `WORKSHOP_DRY_RUN_DISABLED` 前不得解析服务账号资源策略。
8. 返回 `WORKSHOP_DRY_RUN_DISABLED` 前不得查询 outbox 或 forbidden diagnostics。
9. dry-run 开启后仍必须读取 ERPNext User Permission，并按服务账号资源策略 fail closed。
10. dry-run disabled 不得掩盖未登录、无动作权限或非服务账号主体错误。

### 验收

1. 生产 dry-run 禁用且 ERPNext User Permission 超时时，返回 `WORKSHOP_DRY_RUN_DISABLED`。
2. 上述场景 ERPNext User Permission 查询调用次数为 0。
3. 上述场景 outbox 查询调用次数为 0。
4. 上述场景写安全审计，不写操作审计。
5. 未登录、无动作权限、非服务账号主体仍返回对应鉴权错误。
6. dry-run 开启且 ERPNext 权限源不可用时，仍返回 `PERMISSION_SOURCE_UNAVAILABLE`。

## ADR-024：历史 Item 工价必须补齐 Company，计薪匹配必须与工价权限过滤同口径

- 决策时间：2026-04-12 17:07 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003L 历史工价 Company 补数与计薪一致性整改
- 审计来源：TASK-003K 审计意见，历史 item 工价 `company=NULL` 迁移策略未闭环，存在页面权限过滤不可见但登记计薪仍命中的风险
- 结论：`item_code IS NOT NULL AND company IS NULL` 的 active 工价属于历史脏数据，必须补数或阻断；工票计薪匹配必须带 `company`

### 背景

工价列表已按 `item_code/company` 做资源过滤。如果历史款式工价缺少 company，页面可能因 company 过滤看不到该工价；但登记计薪若只按 `item_code/process_name/effective_date` 匹配，则仍可能命中该历史工价，造成权限可见性和工资计算口径不一致。

### 决策

1. 款式专属工价必须绑定明确 `company`。
2. `item_code IS NOT NULL AND company IS NULL` 的 active 工价不得参与计薪。
3. `company=NULL` 只允许用于明确的全局通用工价，且必须满足 `item_code IS NULL` 和全局工价权限。
4. 历史 item 工价必须通过迁移或补数脚本补齐 company。
5. 可唯一解析 company 的历史工价允许自动补数。
6. 多 company 或无法解析 company 的历史工价必须停用或标记 `blocked_scope`，不得保持 active 可用。
7. 工票登记计薪必须使用 ERPNext `Job Card -> Work Order` 派生 company 匹配工价。
8. 工价列表、工价维护、工票计薪必须使用同一套 `item_code/company` scope 口径。
9. 创建款式专属工价时 company 为空必须返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`。
10. 补数、阻断、冲突和失败必须写补数日志或等价审计记录。

### 验收

1. 迁移后不存在 `item_code IS NOT NULL AND company IS NULL AND status='active'` 的工价。
2. 可唯一解析 company 的历史 item 工价被补齐 company。
3. 无法解析或多 company 冲突的历史 item 工价被停用或标记 `blocked_scope`。
4. 工票登记不会命中历史 `company=NULL` 款式工价。
5. 工票登记使用 Job Card / Work Order 派生 company 匹配同 company 工价。
6. 工价列表可见集合和计薪可命中集合在 `item_code/company` 口径上一致。
7. 补数脚本可重复执行，且不会重复放大日志。

## ADR-025：工价 Company 缺失口径统一为 NULL / 空字符串 / 空白字符串

- 决策时间：2026-04-12 17:31 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003M 工价 Company 空字符串口径统一整改
- 审计来源：TASK-003L 审计意见，`company=''` active item 工价仍出现在列表中，计薪错误码与缺 scope 场景不一致
- 结论：`company IS NULL OR trim(company) = ''` 必须统一视为缺失 company；迁移、列表过滤、补数扫描、计薪阻断和写入校验必须共用同一口径

### 背景

TASK-003L 解决了 `company=NULL` 的历史工价问题，但空字符串和空白字符串仍可能作为脏数据存在。若列表过滤、计薪匹配、补数扫描只判断 SQL NULL，会出现空字符串工价页面可见、计薪错误码不一致等问题。

### 决策

1. `NULL`、`''`、`'   '` 在工价 company 语义上全部等价为“缺失 company”。
2. 必须提供或统一使用 `is_missing_company(company)` / `normalize_company(company)` 等价逻辑。
3. 迁移补数扫描必须覆盖 `company IS NULL OR btrim(company) = ''`。
4. 空字符串和空白字符串必须先 normalize 为 NULL，再进入补数或阻断流程。
5. 工价列表不得返回 active `item_code IS NOT NULL AND is_missing_company(company)` 记录。
6. 工票登记计薪不得命中 active `item_code IS NOT NULL AND is_missing_company(company)` 记录。
7. 创建或更新款式专属工价时，normalize 后 company 为空必须返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`。
8. 计薪发现唯一候选是缺失 company 历史 item 工价时，必须返回 `WORKSHOP_WAGE_RATE_SCOPE_REQUIRED`。
9. `WORKSHOP_WAGE_RATE_NOT_FOUND` 只用于确实不存在任何有效工价候选的场景。
10. 迁移后不得存在 active `item_code IS NOT NULL AND (company IS NULL OR btrim(company) = '')` 工价。

### 验收

1. `company=NULL`、`company=''`、`company='   '` 全部被判定为缺失。
2. 创建 item 工价时 company 为 `NULL/''/'   '` 均返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`。
3. 列表不返回 active 空字符串或空白字符串 company 的 item 工价。
4. 计薪不命中空字符串或空白字符串 company 的 item 工价。
5. 唯一候选为缺失 company 历史工价时返回 `WORKSHOP_WAGE_RATE_SCOPE_REQUIRED`。
6. 迁移后不存在 active `item_code IS NOT NULL AND (company IS NULL OR btrim(company) = '')` 工价。

## ADR-026：工价 Company 补数 Dry-Run 必须零写入且不挂载 ORM 对象

- 决策时间：2026-04-12 17:59 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003N 工价补数 Dry-Run 只读整改
- 审计来源：TASK-003M 审计意见，`backfill_wage_rate_company_scope(dry_run=True)` 会把补数日志对象挂入 session，调用方后续 commit 仍会持久化日志
- 结论：工价 company 补数 dry-run 必须是真正只读预演；不得向 session 挂载 ORM 对象，调用方 commit 后也不得产生补数日志、工价变更或审计记录

### 背景

补数 dry-run 用于迁移前预演和管理入口预览。如果 dry-run 路径仍将补数日志 ORM 对象加入当前 session，即使函数本身不 commit，调用方后续 commit 也会产生持久化日志，破坏 dry-run 的只读语义。

### 决策

1. `backfill_wage_rate_company_scope(dry_run=True)` 不得调用 `session.add/add_all/merge/delete/flush/commit`。
2. dry-run 不得修改已加载 ORM 实例字段。
3. dry-run 不得创建并挂载补数日志 ORM 对象。
4. dry-run 报告必须使用 plain dict、dataclass、Pydantic schema 或不可持久化 DTO。
5. dry-run 结束后 `session.new/session.dirty/session.deleted` 必须为空。
6. dry-run 后调用方即使执行 `session.commit()`，也不得新增补数日志、不得修改工价、不得新增审计。
7. 推荐将补数流程拆分为 `build_plan` 与 `apply_plan`；dry-run 只 build plan，execute 才 apply plan。
8. `dry_run=False` 正式执行仍必须写补数日志并更新工价 company/status。

### 验收

1. dry-run 后 `session.new/session.dirty/session.deleted` 均为空。
2. dry-run 后调用 `session.commit()`，补数日志新增 0 条。
3. dry-run 后调用 `session.commit()`，工价 company/status 无变化。
4. dry-run 后调用 `session.commit()`，审计表无 dry-run 副作用记录。
5. dry-run 报告行不是 SQLAlchemy ORM 持久化对象。
6. execute 正式执行仍能写补数日志并更新工价。

## ADR-027：工价补数解析数据库读取失败必须 Fail Closed

- 决策时间：2026-04-12 18:17 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003O 工价补数读取失败 Fail Closed 整改
- 审计来源：TASK-003N 审计意见，`_resolve_backfill_companies()` 吞掉 `SQLAlchemyError` 并继续生成补数计划
- 结论：补数链路中数据库读取失败是系统级失败，必须返回 `DATABASE_READ_FAILED` 并中断计划生成；不得降级为业务多义或无法解析

### 背景

工价 company 补数依赖数据库中的 scoped 工价、历史引用和 ERPNext 派生关系。如果某一步数据库读取失败，继续生成补数计划会基于不完整事实做数据修复判断，可能错误停用、错误补数或漏补数。

### 决策

1. `_resolve_backfill_companies()` 捕获 `SQLAlchemyError` 时必须转换为 `DATABASE_READ_FAILED`。
2. 数据库读取失败不得返回空候选、`ambiguous_company` 或 `company_unresolved`。
3. `build_wage_rate_company_backfill_plan()` 遇到 `DATABASE_READ_FAILED` 必须立即中断。
4. dry-run 读取失败不得返回成功报告，不得挂载 ORM 对象。
5. execute 读取失败必须 rollback，不得更新工价、不得写补数日志。
6. `ambiguous_company/company_unresolved` 只允许在数据库读取成功后作为业务判断结果出现。
7. 错误日志必须脱敏，不得记录 SQL 原文、参数、连接串或敏感凭证。

### 验收

1. 模拟 scoped 工价候选查询抛出 `SQLAlchemyError` 时返回 `DATABASE_READ_FAILED`。
2. 读取失败不返回 `ambiguous_company` 或 `company_unresolved`。
3. 读取失败不返回成功 dry-run 报告。
4. dry-run 读取失败后 session 无 new/dirty/deleted。
5. execute 读取失败后无工价变更、无补数日志。
6. 查询成功但业务多义时仍返回 `WORKSHOP_WAGE_RATE_COMPANY_AMBIGUOUS`。
7. 查询成功但业务无法解析时仍返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。

## ADR-028：ERPNext Item 候选查询不可用必须 Fail Closed

- 决策时间：2026-04-12 18:30 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003P 工价补数 ERPNext Item 候选查询 Fail Closed 整改
- 审计来源：TASK-003O 审计意见，ERPNext Item 候选查询不可用时仍静默当成“无候选”，execute 会把历史工价停用为 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`
- 结论：ERPNext Item 候选查询不可用是外部主数据源不可用，必须 fail closed；不得降级为无候选、不得停用历史工价、不得写补数日志

### 背景

工价 company 补数需要从 ERPNext Item 主数据解析候选 company。如果 ERPNext Item 候选查询失败，系统无法确认“确实没有候选”还是“主数据源暂时不可用”。继续执行会把系统故障误判为业务无法解析，导致 active 历史工价被错误停用。

### 决策

1. ERPNext Item 候选查询超时、连接失败、HTTP 非成功、响应解析失败或权限源不可用时，必须 fail closed。
2. 如果候选查询通过 ERPNext REST API 执行，不可用时返回 `ERPNEXT_SERVICE_UNAVAILABLE` 或等价主数据源不可用错误。
3. 如果候选查询通过共享 PostgreSQL `public.tabItem` 执行，SQLAlchemy 读取失败时返回 `DATABASE_READ_FAILED`。
4. 不可用场景不得返回空候选集合。
5. 不可用场景不得返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
6. 不可用场景不得停用历史工价、不得标记 `blocked_scope`、不得写补数日志。
7. dry-run 不得返回成功预演报告。
8. execute 必须 rollback，不得更新 `ly_operation_wage_rate.company/status`。
9. 只有 ERPNext Item 查询成功且确认无候选时，才允许返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
10. 错误日志必须脱敏，不得记录 ERPNext token、Cookie、Authorization、响应原文中的敏感字段。

### 验收

1. 模拟 ERPNext Item 候选查询超时时返回 `ERPNEXT_SERVICE_UNAVAILABLE` 或 `DATABASE_READ_FAILED`。
2. ERPNext Item 候选查询不可用时不返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
3. ERPNext Item 候选查询不可用时不返回成功 dry-run 报告。
4. execute 遇到 ERPNext Item 候选查询不可用时不修改工价 company/status。
5. execute 遇到 ERPNext Item 候选查询不可用时不新增补数日志。
6. ERPNext Item 查询成功且确认无候选时，仍按业务规则返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。

## ADR-029：内部 Worker 响应中 skipped_forbidden 字段必须废弃或标注诊断语义

- 决策时间：2026-04-12 19:15 CST
- 决策类型：架构审计整改
- 关联模块：TASK-003Q 内部 Worker skipped_forbidden 兼容字段语义整改
- 审计来源：TASK-003P 审计意见，内部 Worker 响应仍保留 `skipped_forbidden_count/skipped_forbidden` 兼容字段，且字段值等于 `forbidden_diagnostic`，可能被误读为实际跳过的越权 outbox 数
- 结论：`forbidden_diagnostic_count` 是越权诊断唯一推荐字段；`skipped_forbidden_count/skipped_forbidden` 必须废弃或明确标注为 diagnostics-only 兼容别名

### 背景

TASK-003H 到 TASK-003I 已把越权 outbox 从主处理窗口中移除，并改为显式诊断扫描。此后内部 Worker 主处理不再“跳过越权 outbox”，而是默认只查询当前服务账号有权处理的 outbox。继续暴露 `skipped_forbidden_count/skipped_forbidden` 会让前端或运维误以为主处理流程跳过了越权行，影响排障判断。

### 决策

1. 主处理指标只使用 `would_process_count/processed_count/succeeded_count/failed_count/blocked_scope_count/dead_count`。
2. 越权诊断指标只推荐使用 `forbidden_diagnostic_count`。
3. `skipped_forbidden_count/skipped_forbidden` 是历史兼容字段，不得作为新契约继续推荐。
4. 如立即删除会破坏兼容，本阶段允许保留，但必须在 Pydantic Field、OpenAPI description、接口文档中标注 `deprecated`、`diagnostics-only`、`alias of forbidden_diagnostic_count`。
5. 默认主处理模式未启用诊断时，`forbidden_diagnostic_count/skipped_forbidden_count/skipped_forbidden` 必须均为 0。
6. 显式启用诊断时，兼容字段如保留只能等于 `forbidden_diagnostic_count`，不得描述为主处理跳过数。
7. 推荐新增 `forbidden_diagnostics_enabled` 或等价字段，明确本次响应是否执行了越权诊断扫描。
8. 操作审计 `after_data` 的判读以 `forbidden_diagnostic_count` 为准。
9. 下一次破坏性 API 版本升级时删除 `skipped_forbidden_count/skipped_forbidden`。

### 验收

1. OpenAPI schema 中 `skipped_forbidden_count/skipped_forbidden` 标注 deprecated 和 diagnostics-only。
2. 接口文档不再把 `skipped_forbidden_count/skipped_forbidden` 描述为主处理跳过数。
3. 默认 `run-once` 不启用诊断时三个相关字段均为 0。
4. `include_forbidden_diagnostics=true` 时兼容字段与 `forbidden_diagnostic_count` 相等，并有清晰诊断说明。
5. 前端和测试只断言推荐字段 `forbidden_diagnostic_count`，不新增对兼容字段的业务依赖。

## ADR-030：外发模块 V1.1 生产级契约冻结

- 决策时间：2026-04-12 20:08 CST
- 决策类型：架构审计整改 / 开发前门禁
- 关联模块：TASK-002A 外发模块设计契约冻结
- 审计来源：TASK-002A 任务单/架构审阅意见，要求补齐写接口幂等契约、对账结算边界、内部库存同步 worker 安全契约
- 结论：外发模块进入工程实现前，必须以 V1.1 契约为唯一基线；未满足幂等、结算锁定和 worker 安全契约，不得进入 TASK-002B

### 背景

外发加工链路会同时写本地业务事实和 ERPNext 库存单据。若写接口无幂等，重复提交会产生重复发料、重复回料、重复验货和重复 Stock Entry outbox。若对账边界不清，TASK-006 加工厂对账会无法锁定结算金额。若内部库存同步 worker 缺少安全契约，普通账号或越权服务账号可能触发库存落账。

### 决策

1. 外发写接口必须支持 `idempotency_key` 或业务唯一键。
2. 相同幂等键 + 相同 payload 必须返回第一次结果，不新增本地事实、不新增 outbox、不新增 ERPNext Stock Entry。
3. 相同幂等键 + 不同 payload 必须返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
4. 外发主单必须包含 `statement_id/settlement_no/settled_by/settled_at`。

9. dry-run 必须可审计且生产默认禁用。
10. 越权 outbox 不得进入主处理窗口，诊断必须显式启用、限流、去重。

### 验收

1. TASK-002 V1.1 设计文档包含写接口幂等契约。
2. TASK-002 V1.1 设计文档包含对账结算字段和锁定规则。
3. TASK-002 V1.1 设计文档包含内部库存同步 worker 安全契约。
4. TASK-002A 任务单包含上述三类契约的可检查验收标准。
5. 当前 sprint 任务清单标注 TASK-002A 为 TASK-002B~TASK-002H 的开发前门禁。

## ADR-031：TASK-002B 阶段回料与验货旧演示成功路径必须 Fail Closed

- 决策时间：2026-04-12 21:25 CST
- 决策类型：架构审计整改 / 阶段门禁
- 关联模块：TASK-002B1 回料验货旧演示路径 Fail Closed 整改
- 审计来源：TASK-002B 审计意见，回料和验货接口仍保留旧演示成功路径，越过 TASK-002B 边界
- 结论：TASK-002B 阶段只做权限、审计、统一错误和事务边界；回料和验货不得继续旧演示成功路径，必须在鉴权和资源权限通过后 fail closed

### 背景

TASK-002B 不实现回料 Stock Entry outbox，也不实现验货扣款正式金额口径。若继续允许 `receive()` 写本地回料事实并推进状态，或允许 `inspect()` 使用旧公式完成验货，会在后续 outbox 和金额口径未落地前制造错误事实，甚至为 TASK-006 对账和库存同步留下脏数据。

### 决策

1. `POST /api/subcontract/{id}/receive` 在 TASK-002B 阶段必须 fail closed。
2. 回料接口必须先完成认证、动作权限、资源权限校验，再返回阶段门禁错误。
3. 回料接口不得新增 `ly_subcontract_receipt`，不得推进状态，不得伪造 `stock_entry_name`。
4. `POST /api/subcontract/{id}/inspect` 在 TASK-002B 阶段必须 fail closed。
5. 验货接口不得新增验货事实，不得推进 `completed`，不得写金额字段。
6. 验货接口不得继续使用旧公式 `net_amount = inspected_qty - deduction_amount`。
7. `draft` 直接回料通道必须删除。
8. 回料接口必须鉴权优先于 payload 字段校验。
9. 数据库写入、flush、commit 失败必须归类为 `DATABASE_WRITE_FAILED`。
10. TASK-002C/D/E/F 未审计通过前，不得恢复回料/验货成功路径。

### 验收

1. 回料鉴权通过后返回 fail closed 错误，且不新增回料事实。
2. 回料不修改外发状态，不新增成功状态日志，不返回伪 `stock_entry_name`。
3. 验货鉴权通过后返回 fail closed 错误，且不新增验货事实。
4. 验货不修改外发状态，不写金额，不使用旧金额公式。
5. 未登录、无权限、权限源不可用优先于 payload 字段校验。
6. DB 写失败返回 `DATABASE_WRITE_FAILED`。
7. 全量测试通过，且 `rg "STE-ISS|net_amount = .*inspected_qty"` 不命中外发正向实现。

## ADR-032：外发资源权限必须以本地 company 事实字段为准

- 决策时间：2026-04-12 21:51 CST
- 决策类型：架构审计整改 / 数据模型门禁
- 关联模块：TASK-002C 外发数据模型与迁移
- 审计来源：TASK-002B1 审计通过后进入 TASK-002C 的门禁意见，要求补齐外发单本地 `company` 字段、迁移、回填和索引，让资源权限从“推断”升级为“本地事实字段校验”
- 结论：外发主单 `ly_subcontract_order.company` 是外发资源权限的本地权威字段；TASK-002C 完成后，列表、详情、写接口和后续 outbox 查询必须以本地 `company` 做 fail closed 权限校验

### 背景

外发加工横跨公司、加工厂、款式、仓库和 ERPNext 库存单据。若外发单没有本地 `company` 事实字段，后端只能在每次请求时通过 Item、Supplier、BOM 或请求上下文推断公司归属。该方式在多公司、多候选、ERPNext 主数据不可用或历史脏数据场景下容易误判资源范围，导致用户访问或操作不属于自己的外发单。

### 决策

1. `ly_subcontract_order.company` 是外发资源权限的本地权威字段。
2. TASK-002C 必须新增或补齐 `ly_subcontract_order.company`，并建立 `company/status`、`company/supplier/status`、`company/item_code/status` 索引。
3. `company` 缺失口径统一为 `NULL/空字符串/空白字符串`。
4. 新建外发单必须写入非空 `company`；无法唯一解析时不得创建。
5. 历史外发单回填只接受唯一公司候选；候选多义、候选为空、ERPNext 主数据不可用、数据库读取失败必须 fail closed。
6. 不得在公司无法解析时写入默认公司、放开全公司权限或降级为“无资源限制”。
7. `ly_subcontract_material`、`ly_subcontract_receipt`、`ly_subcontract_inspection`、`ly_subcontract_status_log`、`ly_subcontract_stock_outbox`、`ly_subcontract_stock_sync_log` 的 `company` 必须从主单派生。
8. 外发列表必须在数据库层按本地 `company` 做资源过滤。
9. 外发详情和写接口必须先读取主单本地 `company`，再做 `company/item_code/supplier/warehouse` 资源校验。
10. `resource_scope_status='blocked_scope'` 或 company 缺失的历史外发单，普通业务动作必须禁止。
11. dry-run 回填必须真正只读，不得挂载 ORM 对象、不得产生迁移日志、不得写审计、不得修改业务数据。
12. TASK-002C 不得恢复发料、回料、验货成功路径，不得创建 ERPNext `Stock Entry`。

### 验收

1. 新建外发单返回体包含后端解析并校验后的 `company`。
2. `company=NULL/''/'   '` 使用同一缺失判断。
3. 历史外发单唯一候选能正确回填 company。
4. 历史外发单候选多义返回 `SUBCONTRACT_COMPANY_AMBIGUOUS`，不改数据。
5. 历史外发单候选为空返回 `SUBCONTRACT_COMPANY_UNRESOLVED`，不写默认公司。
6. ERPNext 主数据不可用返回 `ERPNEXT_SERVICE_UNAVAILABLE`，不改数据。
7. 数据库读取失败返回 `DATABASE_READ_FAILED`，不改数据。
8. dry-run 后 `session.new/session.dirty/session.deleted` 均为 0，后续 commit 不产生副作用。
9. 授权 `Company B` 的用户无法通过列表、详情、回料、验货访问 `Company A` 外发单。
10. `blocked_scope` 外发单不能发料、回料、验货、取消、库存同步重试。
11. `receive/inspect` 在 TASK-002C 完成后仍保持 TASK-002B1 fail closed。

## ADR-033：外发迁移必须自洽且创建单据必须校验 BOM 款式一致

- 决策时间：2026-04-12 22:21 CST
- 决策类型：架构审计整改 / 阻断修复
- 关联模块：TASK-002C1 外发迁移自洽与 BOM 一致性整改
- 审计来源：TASK-002C 审计意见书第 37 份，迁移链缺少外发基础表创建、创建外发单未校验 BOM item 与 payload item 一致
- 结论：外发迁移必须能从空库自洽升级；创建外发单必须以 BOM item 作为款式事实校验边界，禁止 `payload.item_code` 与 `bom_id` 脱钩

### 背景

TASK-002C 已补本地 `company` 事实字段，但迁移脚本只对外发基础表执行 `add_column`，当前迁移目录没有创建 `ly_subcontract_order/material/receipt/status_log` 的迁移。新库从迁移链升级会直接失败，说明迁移不可独立落地。

同时，审计复现 `item_code=ITEM-A` 但 `bom_id` 指向 `ITEM-B` BOM 时，接口仍能创建外发单。这会污染外发主单事实，绕过 BOM item 一致性和潜在资源权限边界，后续发料 outbox 会使用错误 BOM 物料计划。

### 决策

1. TASK-002C1 必须修复迁移链自洽性。
2. 新库从空库执行 Alembic 迁移链，必须能创建 `ly_schema` 和外发基础表。
3. 迁移链必须创建或补齐 `ly_subcontract_order/material/receipt/status_log`，不得只对不存在的表执行 `add_column`。
4. 迁移验收不得使用 ORM `metadata.create_all()` 替代 Alembic 迁移。
5. 已有库升级必须只补缺失字段、索引和约束，不得破坏已有数据。
6. 创建外发单必须校验 `bom.item_code == payload.item_code`。
7. BOM item 与 payload item 不一致时必须返回 `SUBCONTRACT_BOM_ITEM_MISMATCH`。
8. BOM item 不一致时不得新增外发主单、状态日志、成功操作审计或子表事实。
9. 显式传入 company 时，仍必须校验 Item/BOM 事实一致性。
10. company 解析、权限读取、资源校验、service 调用必须纳入统一异常处理范围。
11. `company=null/''/'   '`、候选多义、候选为空、ERPNext 不可用、权限源不可用必须返回统一错误信封，不得裸 500。
12. 回填 execute 写失败必须 rollback，调用方后续 `commit()` 不得落半更新。
13. Company-only 权限允许作为公司内范围权限，但必须由权限源明确返回 Item/Supplier/Warehouse 无显式限制；权限源不可用不得解释为无显式限制。
14. TASK-002C1 不得恢复发料、回料、验货成功路径，不得创建 ERPNext `Stock Entry`。
15. TASK-002D 不得启动，直到 TASK-002C1 审计通过。

### 验收

1. 空库迁移链能创建 `ly_schema`。
2. 空库迁移链能创建 `ly_subcontract_order/material/receipt/status_log`。
3. 空库迁移链能完成 TASK-002C 的 company 字段、索引和约束。
4. 迁移测试不依赖 `metadata.create_all()`。
5. `payload.item_code` 与 BOM item 不一致时返回 `SUBCONTRACT_BOM_ITEM_MISMATCH`。
6. BOM item mismatch 不新增外发主单、不写成功操作审计。
7. `company=null/''/'   '` 返回 `SUBCONTRACT_COMPANY_REQUIRED`，不是裸 500。
8. 公司候选多义、候选为空、ERPNext 不可用、权限源不可用均返回统一错误信封。
9. 回填子表传播写失败或 flush 写失败时 rollback，后续 commit 不落半更新。
10. Company-only 用户只能在权限源明确无 Item/Supplier/Warehouse 限制时访问该 company 外发单。
11. `receive/inspect` 仍保持 TASK-002B1 fail closed。

## ADR-034：外发发料必须通过 outbox 异步创建 ERPNext Material Issue

- 决策时间：2026-04-13 09:24 CST
- 修订时间：2026-04-13 10:58 CST
- 决策类型：架构审计整改 / 库存一致性决策
- 关联模块：TASK-002D 发料 Stock Entry Outbox
- 审计来源：TASK-002C1 审计通过后的风险项，要求 TASK-002D 补齐 outbox 最终字段、幂等键、事件键、Stock Entry worker 查询索引，不能把 TASK-002C scope 骨架当成最终库存同步表结构
- 结论：外发发料写接口只写本地事实和 `stock_action='issue'` outbox；ERPNext `Stock Entry / Material Issue` 必须由内部 worker 在本地事务提交后异步创建并提交；`event_key` 以幂等事件为单位生成，不包含 `issue_batch_no`

### 背景

外发发料会影响 ERPNext 库存台账。若在发料接口本地事务内直接调用 ERPNext，或重复提交时重复生成 Stock Entry，会造成跨系统不一致和库存重复落账。TASK-002C 已建立 outbox scope 骨架，但该结构不足以支撑生产级库存同步，必须在 TASK-002D 补齐幂等键、事件键、payload hash、重试字段、worker 查询索引和 ERPNext event_key 查重。

### 决策

1. `POST /api/subcontract/{id}/issue-material` 本地事务内只写发料事实、状态日志、操作审计和 `ly_subcontract_stock_outbox`。
2. 发料接口本地事务内禁止调用 ERPNext 写接口。
3. 一次发料请求生成一个 `issue_batch_no`、一条 `stock_action='issue'` outbox、多条 `ly_subcontract_material`。
4. 发料必须支持 `idempotency_key`。
5. 相同 `idempotency_key` + 相同 `payload_hash` 返回第一次结果，不新增事实、不新增 outbox。
6. 相同 `idempotency_key` + 不同 `payload_hash` 返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
7. outbox `event_key` 必须使用完整稳定 hash，禁止截断 digest；推荐 `sio:<64位sha256>`。
8. outbox `event_key` 必须以幂等事件为单位生成，推荐输入为 `stock_action + subcontract_id + idempotency_key + stable_payload_hash`。
9. outbox `event_key` 不得包含易变 `issue_batch_no/request_id/outbox_id/created_at/operator`。
10. `issue_batch_no` 是本地发料批次号，只用于业务展示和 material 分组，不参与库存同步幂等事件键。
11. outbox 必须增加 due 查询索引和资源 scope 查询索引。
12. Worker 本阶段只处理 `stock_action='issue'`，不得处理 `receipt`。
13. Worker 必须先按服务账号 `company/item_code/supplier/warehouse` scope 过滤，再 limit。
14. Worker 必须使用行级锁或租约字段，避免并发重复处理同一 outbox。
15. Worker 创建 ERPNext Stock Entry 时必须写入 `custom_ly_subcontract_no/custom_ly_subcontract_outbox_id/custom_ly_outbox_event_key/custom_ly_stock_action`。
16. ERPNext Stock Entry 必须创建并提交，提交成功后才能标记 outbox `succeeded`。
17. 重试前必须按 `custom_ly_outbox_event_key` 查询 ERPNext 是否已有 Stock Entry；已存在则补本地成功回写，不得重复创建。
18. ERPNext 创建成功但本地回写失败时，下一次 worker 必须通过 event_key 找回原单。
19. Worker 未开启、dry-run 未开启、权限源不可用、服务账号越权必须 fail closed。
20. `receive/inspect` 在 TASK-002D 完成后仍保持 fail closed。

### 验收

1. 发料接口能创建本地 material 和 pending issue outbox。
2. 发料接口不调用 ERPNext。
3. 同一幂等键同 payload 不重复生成 material/outbox。
4. 同一幂等键不同 payload 返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
5. outbox `event_key` 使用完整 hash，长输入不碰撞。
6. outbox `event_key` 不包含 `issue_batch_no`，同一幂等请求必须稳定命中同一 outbox。
7. Worker 主查询只处理 issue outbox，并按服务账号资源 scope 过滤。
8. Worker dry-run 不锁定 outbox、不增加 attempts、不调用 ERPNext，并写操作审计。
9. Worker 正式执行能创建并提交 ERPNext `Stock Entry / Material Issue`。
10. ERPNext 成功后回写真正 `stock_entry_name`，禁止伪造 `STE-ISS-*`。
11. ERPNext 已存在相同 event_key 的 Stock Entry 时，不重复创建，直接补本地成功回写。
12. Worker 并发执行不会重复处理同一 outbox。
13. `receive/inspect` 仍保持 fail closed。

## ADR-035：发料幂等必须前置且 ERPNext event_key 找回必须校验 docstatus

- 决策时间：2026-04-13 10:33 CST
- 决策类型：架构审计整改 / 库存一致性决策
- 关联模块：TASK-002D1 发料幂等与 Worker 库存一致性整改
- 审计来源：TASK-002D 审计意见书第 39 份，发料幂等重试被剩余数量校验误拒、Worker 找回 ERPNext Stock Entry 未校验 docstatus、Worker 持事务调用 ERPNext、关键测试缺失
- 结论：发料写接口必须先做幂等命中再做剩余数量校验；Worker 按 event_key 找回 ERPNext Stock Entry 时必须校验 docstatus；Worker 必须使用短事务，ERPNext 网络调用期间不得持有数据库事务或行锁

### 背景

外发发料是库存敏感动作，真实业务中前端重试、网关重试、PDA 重发都很常见。如果第一次全量发料已成功，第二次相同幂等键请求先执行剩余数量校验，就会把合法重试误判为超发，导致调用方以为发料失败。

同时，ERPNext Stock Entry 只有 `docstatus=1` 才代表库存已提交入账。若 Worker 只按 event_key 找到 draft 单据就本地标记成功，会造成本地显示库存同步成功，但 ERPNext Stock Ledger 实际未入账。

### 决策

1. `issue-material` 必须在剩余数量校验之前执行幂等查询。
2. 幂等查询维度为 `subcontract_id + stock_action='issue' + idempotency_key`。
3. 相同幂等键且 `stable_payload_hash` 一致，必须返回第一次 outbox 结果。
4. 相同幂等键但 `stable_payload_hash` 不一致，返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
5. 全量发料 `items=[]` 的 hash 基于请求语义，不得受发料后剩余数量变化影响。
6. `event_key` 正式以幂等事件为单位生成，不包含易变 `issue_batch_no`。
7. `ly_subcontract_stock_outbox` 是幂等权威表；`ly_subcontract_material` 通过 `stock_outbox_id` 关联 outbox 查询幂等上下文，不强制冗余 `idempotency_key/payload_hash`。
8. `ERPNextStockEntryService.find_by_event_key()` 必须返回结构化结果，至少包含 `name/docstatus`。
9. `docstatus=1` 才可直接补本地 `succeeded`。
10. `docstatus=0` 必须 submit 原 draft 单据，submit 成功后才能本地 `succeeded`。
11. `docstatus=2` 或未知状态不得本地成功。
12. 相同 event_key 查到多张 ERPNext Stock Entry 时必须 fail closed。
13. Worker 必须拆成 claim 短事务、ERPNext 网络调用、result 回写短事务。
14. Worker 调用 ERPNext 时不得持有数据库事务或行锁。
15. result 回写失败后，下一轮 worker 必须能通过 event_key 恢复本地状态。
16. TASK-002D1 不得恢复回料、验货成功路径。

### 验收

1. 第一次全量发料成功后，相同幂等键重试返回第一次 outbox 结果，不返回超发错误。
2. 同幂等键重试不新增 material、不新增 outbox、不增加 issued_qty。
3. `event_key` 不包含 `issue_batch_no`，同一幂等事件稳定不变。
4. 找到 ERPNext `docstatus=1` 单据时补本地 succeeded。
5. 找到 ERPNext `docstatus=0` draft 时 submit 原单，成功后补本地 succeeded。
6. 找到 ERPNext `docstatus=2` 或未知状态时不得本地成功。
7. 相同 event_key 多单据时 fail closed。
8. Worker 调用 ERPNext 时不持有数据库事务或行锁。
9. 并发 Worker 不重复处理同一 outbox。
10. 失败、重试、dead、event_key 恢复和日志脱敏测试齐全。

## ADR-036：外发回料必须通过 outbox 异步创建 ERPNext Material Receipt

- 决策时间：2026-04-13 11:01 CST
- 决策类型：架构审计整改 / 库存一致性决策
- 关联模块：TASK-002E 回料 Stock Entry Outbox
- 审计来源：TASK-002D1 审计通过后，审计官确认 TASK-002E 可进入但边界必须锁死在回料链路
- 结论：外发回料写接口只写本地回料事实和 `stock_action='receipt'` outbox；ERPNext `Stock Entry / Material Receipt` 必须由内部 worker 在本地事务提交后异步创建并提交；TASK-002E 不得引入验货、对账、结算逻辑

### 背景

回料是外发链路中第二个库存动作，会把加工结果回入本厂仓库。该动作必须进入 ERPNext Stock Ledger，但不能在 FastAPI 本地事务内直接调用 ERPNext，否则会产生跨系统不一致和重复入账风险。TASK-002D1 已验证发料 outbox 的幂等、event_key、docstatus 和短事务规则，TASK-002E 必须复用同一库存一致性标准到回料链路。

### 决策

1. `POST /api/subcontract/{id}/receive` 本地事务内只写回料事实、状态日志、操作审计和 `stock_action='receipt'` outbox。
2. 回料接口本地事务内禁止调用 ERPNext 写接口。
3. 本地回料事实与 receipt outbox 必须同事务提交；commit 失败不得调用 ERPNext。
4. 一次回料请求生成一个 `receipt_batch_no`、一条 receipt outbox、一组 `ly_subcontract_receipt`。
5. 回料必须支持 `idempotency_key`。
6. 回料幂等检查必须前置到剩余可回数量校验之前。
7. 回料 `event_key` 必须以幂等事件为单位生成，推荐输入为 `stock_action + subcontract_id + idempotency_key + stable_payload_hash`。
8. 回料 `event_key` 不得包含易变 `receipt_batch_no/request_id/outbox_id/created_at/operator`。
9. `receipt_batch_no` 只用于本地回料批次展示和 receipt 分组，不参与库存同步幂等事件键。
10. 回料 item 默认为外发单 `item_code`，禁止前端任意指定 item 成为回料事实。
11. 默认禁止超收，超收策略如需放开必须另出 ADR。
12. Worker 必须明确区分 `stock_action='issue'` 与 `stock_action='receipt'`，禁止 handler 混用。
13. Worker 处理 receipt 时必须创建并提交 ERPNext `Stock Entry / Material Receipt`。
14. ERPNext Material Receipt payload 必须使用 `t_warehouse`。
15. Worker 必须按 event_key 查询 ERPNext Stock Entry 并校验 `docstatus`；draft 必须 submit，cancelled/未知状态不得本地成功。
16. Worker 必须使用 claim 短事务、ERPNext 网络调用、result 回写短事务。
17. `inspect` 在 TASK-002E 完成后仍必须 fail closed，不得写验货事实、金额、对账或结算数据。
18. `waiting_inspection` 状态允许继续分批回料；该状态下只能继续写 receipt/outbox，不得在 TASK-002E 阶段执行验货成功路径。

### 验收

1. 回料接口能创建本地 receipt 和 pending receipt outbox。
2. 回料接口不调用 ERPNext。
3. 回料事实和 receipt outbox 同事务提交；commit 失败不调用 ERPNext。
4. 同一幂等键同 payload 不重复生成 receipt/outbox。
5. 同一幂等键不同 payload 返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
6. 回料 `event_key` 不包含 `receipt_batch_no`。
7. Worker 处理 receipt 时创建并提交 ERPNext `Stock Entry / Material Receipt`。
8. Worker 使用 `t_warehouse`，不得使用 issue 的 `s_warehouse`。
9. Worker 按 `docstatus` 区分已提交、草稿、已取消和异常状态。
10. Worker 并发执行不会重复处理同一 receipt outbox。
11. `inspect` 仍保持 fail closed。
12. `waiting_inspection` 状态可继续登记后续分批回料。

## ADR-037：外发库存同步重试必须按 outbox_id + idempotency_key + stock_action 精确定位

- 决策时间：2026-04-13 12:06 CST
- 决策类型：架构审计整改 / 库存一致性决策
- 关联模块：TASK-002E1 回料同步重试精确定位整改
- 审计来源：TASK-002E 审计意见书第 41 份，retry 接口按订单取最新 outbox，误重置已成功 receipt outbox
- 结论：外发库存同步 retry 必须精确定位目标 outbox；禁止按订单选择最新 outbox；禁止重置已成功、处理中或待处理 outbox

### 背景

外发单会同时存在发料 issue outbox 和回料 receipt outbox。TASK-002E 审计复现了一个高危场景：同一订单下 issue outbox 已失败，后续 receipt outbox 已成功，调用 retry 时系统按“最新 outbox”选择了 receipt，并把已成功 receipt 重置为 pending，真正失败的 issue 仍未重试。该行为会造成已成功库存同步重新入队，存在重复库存落账和本地状态污染风险。

### 决策

1. `/api/subcontract/{id}/stock-sync/retry` 必须接收 `outbox_id`。
2. retry 请求必须同时提供 `stock_action` 与原始 `idempotency_key`。
3. 服务端必须按 `outbox_id` 精确读取目标 outbox。
4. 目标 outbox 必须满足 `subcontract_id=路径 id`。
5. 目标 outbox 必须满足 `stock_action=请求 stock_action`。
6. 目标 outbox 必须满足 `idempotency_key=请求 idempotency_key`。
7. 不满足订单、动作或幂等键匹配时必须 fail closed，不得降级到最新 outbox。
8. retry 只允许 `failed/dead` 状态进入 pending。
9. `succeeded/processing/pending/blocked_scope` 禁止重置。
10. retry 资源权限以目标 outbox 的 `company/item_code/supplier/warehouse/stock_action` 为准。
11. 权限源不可用返回 `PERMISSION_SOURCE_UNAVAILABLE`，不得修改 outbox。
12. retry 接口不得调用 ERPNext。
13. retry 成功写操作审计；401/403/503 写安全审计。
14. `stable_payload_hash` 必须对发料和回料数量字段做 Decimal 语义归一化。
15. `waiting_inspection` 状态允许继续分批回料，但验货接口继续 fail closed。
16. Worker submit ERPNext draft 后必须确认最终 `docstatus=1`，否则不得本地 succeeded。

### 验收

1. issue failed + receipt succeeded 场景下，重试 issue 只重置 issue outbox。
2. 上述场景中 receipt outbox 保持 succeeded。
3. receipt failed 可按精确 outbox 重试。
4. succeeded/processing/pending outbox 调用 retry 不改变状态。
5. stock_action/idempotency_key/order 任一不匹配均返回明确错误码。
6. retry 接口不调用 ERPNext。
7. 无目标 outbox 资源权限或权限源不可用时不修改 outbox。
8. `10/10.0/10.000000` 生成相同 payload hash。
9. submit draft 后未确认 `docstatus=1` 不得标记 succeeded。

## ADR-038：外发验货金额必须以本地 inspection 事实固化 gross / deduction / net

- 决策时间：2026-04-13 12:31 CST
- 决策类型：架构审计整改 / 金额口径决策
- 关联模块：TASK-002F 验货扣款金额口径
- 审计来源：TASK-002E1 审计通过后允许进入 TASK-002F；历史 inspect 路径曾存在 `net_amount = inspected_qty - deduction_amount` 数量减金额错误
- 结论：外发验货从 TASK-002F 开始写本地 inspection 金额事实，金额公式固定为 `gross_amount - deduction_amount = net_amount`，并禁止验货接口调用 ERPNext 财务或库存写接口

### 背景

外发验货决定加工厂扣款和后续对账金额。如果金额公式混用数量和金额，会直接污染加工厂对账单和应付数据。TASK-002B~E 阶段一直保持 inspect fail closed，就是为了避免旧演示路径继续写入错误金额。TASK-002F 开始恢复验货成功路径，但必须先冻结金额口径、幂等规则、状态流转和与 ERPNext 的边界。

### 决策

1. `POST /api/subcontract/{id}/inspect` 必须写入 `ly_subcontract_inspection` 本地事实。
2. 验货必须绑定 `receipt_batch_no`。
3. 目标回料批次必须已完成 ERPNext Material Receipt 同步。
4. 验货接口不调用 ERPNext 写接口。
5. 验货接口不创建 `Stock Entry / Purchase Invoice / Payment Entry / GL Entry`。
6. `accepted_qty = inspected_qty - rejected_qty`。
7. `rejected_rate = rejected_qty / inspected_qty`。
8. `gross_amount = inspected_qty × subcontract_rate`。
9. `deduction_amount = rejected_qty × deduction_amount_per_piece`。
10. `net_amount = gross_amount - deduction_amount`。
11. `deduction_amount` 不得大于 `gross_amount`。
12. 金额字段使用 Decimal，保留 2 位小数，ROUND_HALF_UP。
13. 禁止 `net_amount = inspected_qty - deduction_amount`。
14. 验货写接口必须支持幂等。
15. 幂等检查必须前置到剩余可验数量校验之前。
16. 验货成功必须更新外发主单验货与金额汇总。
17. 状态流转按累计回料和累计验货决定：待验、待回料、完成。
18. 旧 `get_retry_target()` 最新 outbox 选择逻辑必须删除或显式废弃，避免 TASK-002E1 修复回潮。

### 验收

1. 验货成功写入 inspection 并返回金额字段。
2. 未同步成功的 receipt batch 不允许验货。
3. 相同幂等键相同 payload 不重复扣款。
4. 相同幂等键不同 payload 返回冲突。
5. `inspected_qty=100,rejected_qty=5,subcontract_rate=10,deduction_amount_per_piece=2` 时，`gross_amount=1000.00,deduction_amount=10.00,net_amount=990.00`。
6. 旧错误公式扫描无命中。
7. 验货接口不调用 ERPNext 写接口。
8. 并发验货不得超过批次可验数量。
9. 删除或废弃旧 `get_retry_target()` 后，retry 仍只按精确 outbox 定位。

## ADR-039：验货幂等必须以 receipt_batch_no 作为业务身份字段

- 决策时间：2026-04-13 13:05 CST
- 决策类型：架构审计整改 / 幂等与对账金额一致性决策
- 关联模块：TASK-002F1 验货幂等批次隔离与详情闭环整改
- 审计来源：TASK-002F 审计意见书第 43 份，同一 idempotency_key 换回料批次返回第一次验货结果，导致第二批漏验
- 结论：验货 inspection 的 payload hash 必须包含 `receipt_batch_no`；stock outbox 的易变字段排除规则不得复用于 inspection 幂等；外发详情必须返回 inspections；验货字段必须通过独立迁移补齐

### 背景

外发验货是按回料批次确认质量和扣款的业务事实。`receipt_batch_no` 对 stock outbox event_key 来说是本地展示批号，可以不参与库存同步事件键；但对 inspection 来说，`receipt_batch_no` 是验货对象身份。如果 inspection hash 排除该字段，同一幂等键切换到另一个回料批次会被误判为重复请求并返回第一次验货结果，造成第二批未验货但接口返回成功，后续对账金额少计。

### 决策

1. inspection payload hash 必须使用独立构造函数。
2. inspection payload hash 必须包含 `receipt_batch_no`。
3. inspection payload hash 不得复用 stock outbox 的易变字段排除清单。
4. 同一 `subcontract_id + idempotency_key + 同一 receipt_batch_no + 同一语义 payload` 返回第一次验货结果。
5. 同一 `subcontract_id + idempotency_key + 不同 receipt_batch_no` 返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
6. 同一 `subcontract_id + idempotency_key + 同一 receipt_batch_no + 不同数量/扣款/备注 payload` 返回 `SUBCONTRACT_IDEMPOTENCY_CONFLICT`。
7. Decimal 数量和金额字段继续做语义归一化。
8. `GET /api/subcontract/{id}` 必须返回 `inspections[]`。
9. 验货金额事实源为 `ly_subcontract_inspection` 和主单汇总字段。
10. `ly_subcontract_receipt` 上的历史金额字段不得作为 TASK-002F 后的金额事实源。
11. `gross_amount = inspected_qty × subcontract_rate` 是唯一有效加工费公式。
12. `gross_amount = accepted_qty × subcontract_rate` 废弃。
13. TASK-002F 验货字段必须通过独立 Alembic migration 补齐，不能只依赖已存在的 TASK-002C revision。

### 验收

1. 同 key 不同 receipt batch 返回幂等冲突。
2. 同 key 不同 receipt batch 不新增 inspection，不更新主单金额。
3. 同 key 同 batch 同语义 payload 返回第一次结果。
4. 外发详情返回 inspections 和金额快照。
5. Alembic 空库升级到 head 后包含验货字段。
6. 已执行 TASK-002C revision 的库可通过 TASK-002F 独立迁移补齐验货字段。
7. 代码和设计无有效 `gross_amount = accepted_qty × subcontract_rate` 口径。

## ADR-040：外发验货必须通过并发回归测试证明不超量

- 决策时间：2026-04-13 13:26 CST
- 决策类型：架构审计整改 / 并发一致性与接口安全顺序决策
- 关联模块：TASK-002F2 并发验货不超量回归测试
- 审计来源：TASK-002F1 审计意见书第 44 份，缺少两个请求同时验同一回料批次时不超量的回归测试
- 结论：外发验货必须用数据库事务锁和并发回归测试证明不会超过批次已同步回料数量；外发详情必须先资源鉴权再读取子表明细；TASK-002F 验货字段必须通过独立 migration 补齐

### 背景

验货直接决定加工费和扣款金额。如果两个请求同时验同一回料批次，任何读旧剩余数量后分别写入的竞态都会导致超验、超扣或对账金额污染。当前实现依赖生产数据库 `FOR UPDATE`，但主测试环境是 SQLite，无法真实验证行锁语义，因此必须补充明确的并发回归测试或 PostgreSQL 集成测试。

### 决策

1. 两个请求同时验同一 `receipt_batch_no` 时，最终累计验货数量不得超过该批次已同步回料数量。
2. 验货剩余可验数量必须在数据库锁内重新计算。
3. 插入 inspection、更新主单汇总、写状态日志必须与剩余可验校验在同一事务内完成。
4. 禁止用进程内全局锁替代数据库事务锁。
5. 禁止用 sleep 作为唯一并发保护。
6. 必须补充两个独立 session/client 的并发回归测试。
7. SQLite 无法证明行锁时，必须补 PostgreSQL 集成测试或明确 skip，并保留服务层防回归测试。
8. `GET /api/subcontract/{id}` 必须先做订单级资源权限校验，再读取 `inspections[]/receipts[]/status_logs[]/latest_outbox`。
9. 无权限或权限源不可用时不得读取子表明细。
10. TASK-002F 验货字段必须通过独立 Alembic migration 补齐，不能只依赖已执行过的 TASK-002C revision。

### 验收

1. 并发验货测试使用两个独立 session/client。
2. 两个请求数量之和大于批次已回料数量时，最终累计验货数量不超过已回料数量。
3. 第二个请求返回 `SUBCONTRACT_INSPECTION_QTY_EXCEEDED` 或明确并发/锁冲突错误信封。
4. 并发失败不写超量 inspection，不超量更新主单汇总。
5. 无权限详情请求不读取 inspections 明细。
6. 权限源不可用详情请求不读取子表明细。
7. 独立 TASK-002F migration 可补齐已执行 TASK-002C revision 的数据库。

## ADR-041：PostgreSQL 并发验货测试必须真实执行并进入交付门禁

- 决策时间：2026-04-13 13:41 CST
- 决策类型：架构审计整改 / 测试门禁决策
- 关联模块：TASK-002F3 PostgreSQL 并发验货集成测试门禁整改
- 审计来源：TASK-002F2 审计意见书第 45 份，`test_inspect_concurrent_requests_do_not_overinspect` 无条件 skip 且 `postgresql` marker 未注册
- 结论：PostgreSQL 并发验货测试必须在提供 `POSTGRES_TEST_DSN` 时真实执行；`postgresql` marker 必须注册；TASK-002F3 复审必须提供 PostgreSQL 并发测试运行结果；inspect 审计快照必须资源鉴权后读取

### 背景

验货并发安全依赖 PostgreSQL 行锁语义，SQLite 默认测试无法证明生产环境双事务竞争不会超验。无条件 skip 的占位测试会让并发风险长期无门禁保护，unknown marker warning 也会淹没真实测试配置问题。

### 决策

1. PostgreSQL 并发验货测试不得无条件 skip。
2. 无 `POSTGRES_TEST_DSN` 时可以明确 skip。
3. 有 `POSTGRES_TEST_DSN` 时必须真实执行双连接或双 Session 并发测试。
4. 并发测试必须证明最终累计验货数量不超过批次已同步回料数量。
5. 必须注册 `postgresql` pytest marker。
6. TASK-002F3 交付复审必须提供带 `POSTGRES_TEST_DSN` 的测试运行结果。
7. 后续合并门禁应包含 PostgreSQL job；短期没有 CI 时，以本地 PostgreSQL 测试输出作为最低门禁。
8. `POST /api/subcontract/{id}/inspect` 的 `before_data/get_order_snapshot()` 必须在资源权限校验之后读取。
9. 无权限或权限源不可用时，inspect 不得读取 latest outbox 或 inspections 子表快照。

### 验收

1. `pytest.ini` 注册 `postgresql` marker。
2. 全量 pytest 无 unknown marker warning。
3. 无 DSN 时 PostgreSQL 测试明确 skip。
4. 有 DSN 时 PostgreSQL 测试真实执行并通过。
5. 两个并发请求不会造成累计验货数量超过批次已回料数量。
6. inspect 无权限和权限源不可用路径不读取子表快照。

## ADR-042：外发前端按钮必须以后端权限动作和单据状态共同驱动

- 决策时间：2026-04-13 13:58 CST
- 决策类型：前端权限与状态联动决策
- 关联模块：TASK-002G 前端状态与权限联动
- 审计来源：TASK-002F3 通过后进入外发前端闭环；现有前端仍有 receive/retry/inspect 旧契约残留
- 结论：外发前端 UI 只能展示后端授权且当前状态允许的操作；前端 API 类型必须与后端现行契约一致；内部 worker API 禁止进入普通前端

### 背景

外发后端已完成权限、资源权限、发料/回料 outbox、验货金额和库存同步重试。前端如果继续使用旧入参或只按本地状态展示按钮，会出现“页面可点但后端拒绝”或误调用旧接口的情况。特别是 retry 已经修复为精确 outbox 定位，前端无 body 调用会直接回到高危历史问题。

### 决策

1. 外发前端按钮权限以 `/api/auth/actions` 的 `button_permissions` 为权威来源。
2. 前端按钮必须同时检查后端权限、单据状态、资源范围状态和结算状态。
3. `subcontract:stock_sync_worker` 是内部权限，不得展示普通 UI 按钮。
4. 前端不得调用 `/api/subcontract/internal/stock-sync/run-once`。
5. 回料前端 payload 不得包含验货字段。
6. 验货前端 payload 必须使用 `deduction_amount_per_piece`。
7. retry 前端 payload 必须包含 `outbox_id + stock_action + idempotency_key`。
8. 写操作成功后必须重新加载详情。
9. 详情页必须展示回料批次和验货明细。
10. 如后端详情缺少前端必要只读字段，可在 TASK-002G 补只读字段，但不得新增业务写逻辑。

### 验收

1. 无 read 权限不展示业务列表。
2. 无 create 权限不展示新建按钮。
3. 发料、回料、验货、retry、取消按钮均按权限和状态显示。
4. 前端 receive/retry/inspect 入参与后端契约一致。
5. 前端不调用内部 worker API。
6. 详情页展示真实回料批次、验货明细和金额快照。
7. 401/403/503 错误提示友好且不泄露敏感信息。

## ADR-043：外发前端 API 必须统一鉴权，列表同步摘要不得逐行拉详情

- 决策时间：2026-04-13 14:20 CST
- 决策类型：架构审计整改 / 前端请求安全与列表性能决策
- 关联模块：TASK-002G1 前端外发 API 统一鉴权与列表摘要整改
- 审计来源：TASK-002G 审计意见书第 47 份，`src/api/subcontract.ts` 使用裸 `fetch()`，列表页逐行调用详情接口获取同步状态
- 结论：外发前端所有 `/api/subcontract/` 请求必须走统一鉴权请求封装；外发列表同步摘要纳入 list API 轻量字段，不允许前端逐行拉详情

### 背景

外发模块后端已经以当前用户、动作权限、资源权限和审计为安全边界。前端 API 如果直接 `fetch()`，生产登录态下会缺少 `Authorization`、Cookie 凭证和统一错误信封处理，导致真实环境下 401/403 或错误提示失真。另一方面，列表页逐行拉详情会把一次列表查询放大为 N+1 次 HTTP 请求，并重复触发资源权限、子表读取和错误处理，后续数据量扩大后会成为性能和审计噪声风险。

### 决策

1. `src/api/subcontract.ts` 禁止直接调用 `fetch()`。
2. 外发前端 API 必须使用统一 `request<T>()` 或等价封装。
3. 统一请求封装必须包含 `credentials: 'include'`。
4. 统一请求封装必须按项目现有 token 约定写入标准 `Authorization` 头。
5. 统一请求封装必须解析 `{code,message,data}` 成功和失败信封。
6. 401、403、503 必须转成前端友好错误提示。
7. 前端错误不得展示 Authorization、Cookie、token、password、secret、SQL 原文或堆栈。
8. 外发列表页禁止逐行调用 `GET /api/subcontract/{id}` 获取同步状态。
9. 外发列表同步状态正式纳入 `GET /api/subcontract/` list item 轻量摘要字段。
10. list 摘要字段包含最新 issue/receipt outbox id、sync status、stock entry name、idempotency key 和 error code。
11. list 摘要必须在资源权限过滤后计算。
12. 后端必须用批量查询或窗口函数计算 list 摘要，不得循环查详情。
13. 如果 list 摘要暂不可用，列表页只能提示“进入详情查看”，不得用 N+1 兜底。
14. `allowed_actions_for_current_order` 允许作为详情接口只读增强字段，但不替代后端写接口鉴权。
15. TASK-002H 不得启动，直到 TASK-002G1 通过审计。

### 验收

1. `src/api/subcontract.ts` 无直接 `fetch(`。
2. 所有 `/api/subcontract/` 请求带登录凭证和统一错误处理。
3. 401/403/503 提示友好且不泄露敏感信息。
4. 列表页无 `fetchSubcontractOrderDetail(row.id)` 或等价逐行详情请求。
5. list item 可展示 latest issue/receipt 同步摘要。
6. 后端 list 摘要不返回 payload_json、原始异常和敏感凭证。
7. retry 仍按 `outbox_id + stock_action + idempotency_key` 精确提交。
8. 前端不调用内部 worker run-once API。

## ADR-044：外发列表库存同步状态采用后端轻量摘要字段封版

- 决策时间：2026-04-13 14:31 CST
- 决策类型：架构审计整改 / 前后端列表契约封版决策
- 关联模块：TASK-002G2 外发列表同步摘要字段封版整改
- 审计来源：TASK-002G1 审计意见书第 48 份，前端列表读取 latest issue/receipt sync status，但后端 list 接口未返回字段
- 结论：保留外发列表“库存同步状态”列，后端 `GET /api/subcontract/` 补齐 latest issue/receipt 轻量同步摘要字段；禁止恢复 N+1 详情查询

### 背景

TASK-002G1 已经修复裸 `fetch()` 和列表 N+1 高危问题，但前端列表库存同步状态仍没有稳定数据来源。如果简单删除该列，会降低外发单列表的运营可见性；如果恢复逐行拉详情，又会回到审计已经关闭的 N+1 和权限检查放大问题。因此本次封版选择后端 list item 返回轻量只读摘要字段。

4. 摘要字段只读，不触发 retry、worker、ERPNext 调用或 outbox 状态变化。
5. 最新 outbox 以 `subcontract_id + stock_action` 分组，按 `created_at desc, id desc` 取 1 条。
6. issue 和 receipt 摘要必须分开计算，不得串扰。
7. 摘要计算必须发生在订单资源权限过滤之后。
8. 摘要查询必须批量完成，不得前端或后端逐订单调用详情。
9. list item 不得返回 `payload_json`、`last_error_message`、ERPNext 原始异常或敏感凭证。
10. 前端列表读取 list item 字段展示同步状态；字段为空显示“暂无同步任务”。
11. `latest_*_idempotency_key` 可用于精确 retry payload，但不得作为普通文本展示。
12. TASK-002H 不得启动，直到 TASK-002G2 通过审计。

### 验收

1. `GET /api/subcontract/` 返回 `latest_issue_sync_status/latest_receipt_sync_status`。
2. 同一订单 issue/receipt 摘要互不串扰。
3. 多条 outbox 返回最新一条。
4. 无 outbox 时返回 null 字段。
5. 无权限订单不泄露摘要。
6. 前端无逐行详情请求。
7. 前端库存同步状态列不再长期显示“进入详情查看”。
8. 后端和前端均不恢复裸 fetch 或内部 worker run-once 调用。

## ADR-046：外发对账数据出口以验货明细为结算事实源

- 决策时间：2026-04-13 14:51 CST
- 决策类型：模块边界 / 对账数据出口决策
- 关联模块：TASK-002H 对账数据出口；TASK-006 加工厂对账单
- 审计来源：TASK-002G3 审计意见书第 50 份通过，允许进入 TASK-002H；第 50 份风险项 `_latest_outbox_by_order_ids()` 后续优化不阻塞本任务
- 结论：外发对账数据出口以 `ly_subcontract_inspection` 为最小结算明细事实源；TASK-002H 只提供候选、预览和锁定服务，TASK-006 才创建对账单和应付

### 背景

外发验货已经形成 `gross_amount/deduction_amount/net_amount` 金额事实。加工厂对账需要按照供应商和期间汇总这些金额，但一个外发单可能多批回料、多次验货，订单级汇总无法表达部分批次先结算、后续批次再结算的场景。因此，结算出口必须以 inspection 行为明细粒度。

### 决策

1. `ly_subcontract_inspection` 是外发结算明细事实源。
2. `ly_subcontract_order` 金额字段仅作为订单汇总展示和校验辅助。
3. 一条 inspection 只能进入一个未释放/未取消的 statement。
4. 对账候选必须只包含已验货、回料已同步成功、未结算、资源权限允许的 inspection。
5. 对账候选金额读取 inspection 金额事实，不重新按工价公式计算。
6. TASK-002H 新增 `settlement_status/statement_id/statement_no/settlement_line_key/settlement_locked_by/settlement_locked_at/settled_by/settled_at/settlement_request_id` 等字段到 inspection。
7. `statement_id` 当前阶段 nullable 且不强制 FK，正式 FK 由 TASK-006 创建对账单表后接入。
8. 锁定服务必须使用事务和行级锁，防止重复结算。
9. 同一 statement 重复锁定同一批 inspection 返回幂等成功。
10. 不同 statement 锁定同一 inspection 必须返回冲突。
11. release 只允许释放 `statement_locked` 行，不允许释放 `settled` 行。
12. TASK-002H 不创建 `factory_statement` 表，不创建 ERPNext Purchase Invoice，不创建 Payment Entry，不写 GL。
13. TASK-006 负责对账单主表、对账编号、确认、取消、应付草稿和打印。
14. 列表同步状态只展示最新发料/最新回料 outbox 摘要；多批次同步流水不进入外发列表。

### 验收

1. 可结算候选接口按 inspection 行返回。
2. 未同步回料、未验货、已锁定、已结算、取消和 blocked_scope 明细不返回。
3. 候选接口和锁定接口均做动作权限与资源权限校验。
4. 预览汇总金额来自 inspection 金额事实。
5. 锁定同一批 inspection 具备幂等性。
6. 不同 statement 重复锁定同一 inspection 返回冲突。
7. 失败 rollback，不产生半锁定。
8. 不调用 ERPNext 写接口。
9. 不创建对账单主表或财务单据。

## ADR-047：结算锁定与释放幂等必须使用 append-only 操作表

- 决策时间：2026-04-13 15:26 CST
- 决策类型：架构审计整改 / 幂等与结算状态安全决策
- 关联模块：TASK-002H1 结算幂等重放漏洞整改
- 审计来源：TASK-002H 审计意见书第 51 份，锁定/释放共用 `settlement_request_id` 导致旧锁定请求可在释放后重放
- 结论：结算锁定/释放幂等历史必须使用 append-only 操作表；inspection 可变字段不得作为幂等权威来源；候选/预览 summary 定义为筛选条件全量汇总

### 背景

结算锁定和释放是对账前置财务状态。如果幂等历史记录在 inspection 行的单个可变字段上，释放操作会覆盖锁定操作的幂等 key，导致旧锁定请求在网络重试时被误判为新请求，从而把已经释放的明细重新锁定。这种状态重放会直接污染对账单候选和结算锁定边界。

### 决策

1. 新增 `ly_subcontract_settlement_operation` append-only 操作表。
2. `operation_type + idempotency_key` 必须唯一。
3. `request_hash` 必须包含 operation_type、statement_id、statement_no 和排序后的 inspection_ids。
4. 同 key 同 hash 返回首次成功响应，不再次修改 inspection。
5. 同 key 不同 hash 返回 `SUBCONTRACT_SETTLEMENT_IDEMPOTENCY_CONFLICT`。
6. `lock K -> release R -> retry lock K` 不得重新锁定已释放行。
7. `release R -> retry release R` 不得再次修改行。
8. operation 记录写入与 inspection 状态变更必须在同一事务内完成。
9. operation 写入失败必须 rollback inspection 状态变更。
10. `inspection.settlement_request_id` 不再作为幂等历史权威来源。
11. `idempotency_key` 在 API Schema、模型和迁移中统一长度 128。
12. `settlement-candidates.summary` 和 `settlement-preview.summary` 默认表示筛选条件全量汇总；分页汇总必须另命名 `page_summary`。
13. TASK-002H1 不得新增对账单主表、应付、付款或 GL。

### 验收

1. append-only 操作表存在并有唯一约束。
2. 旧锁定请求在 release 后重试不会重新锁定。
3. 同 key 不同 payload 返回冲突。
4. operation 写失败会 rollback 业务状态。
5. 128 字符幂等键可用，超过 128 字符被拒绝。
6. summary 语义明确且有测试覆盖。
7. 不调用 ERPNext 写接口。

## ADR-048：结算 operation 唯一冲突必须转为幂等 Replay

- 决策时间：2026-04-13 15:43 CST
- 决策类型：架构审计整改 / 并发幂等决策
- 关联模块：TASK-002H2 结算并发幂等 Replay 整改
- 审计来源：TASK-002H1 审计意见书第 52 份，同 key 并发重复请求可能在 operation 唯一约束处返回 `DATABASE_WRITE_FAILED`
- 结论：同 `operation_type + idempotency_key` 并发唯一冲突必须转为 replay 首次响应或幂等冲突；lock/release 响应必须显式返回 `idempotent_replay`

### 背景

append-only operation 表解决了释放后旧锁定请求重放的高危问题，但并发场景仍有窗口：两个相同幂等请求同时进入，第二个可能在首次读取 operation 时未命中，随后在写 operation 时撞唯一约束。该情况是正常幂等并发，不应暴露为数据库写失败。

### 决策

1. lock/release 必须在获得 inspection 行锁后再次读取 operation。
2. operation 二次读取命中且 hash 相同，直接 replay 首次响应。
3. operation 二次读取命中但 hash 不同，返回幂等冲突。
4. operation insert/update 遇到 `operation_type + idempotency_key` 唯一冲突时，必须 reload operation。
5. reload 后 hash 相同，返回首次响应，且 `idempotent_replay=true`。
6. reload 后 hash 不同，返回 `SUBCONTRACT_SETTLEMENT_IDEMPOTENCY_CONFLICT`。
7. replay 不得新增第二条 operation，不得再次修改 inspection。
8. 首次 lock/release 成功返回 `idempotent_replay=false`。
9. lock/release 成功响应包含 `operation_id/idempotency_key/idempotent_replay`。
10. `settlement_request_id` 保留为兼容/最近请求标识，不是幂等历史事实源。
11. PostgreSQL 有 `POSTGRES_TEST_DSN` 时必须执行同 key 并发 lock/release 测试。
12. TASK-002H2 不得创建对账单主表、应付、付款或 GL。

### 验收

1. 同 key 同 payload 并发 lock 不返回 `DATABASE_WRITE_FAILED`。
2. 同 key 同 payload 并发 release 不返回 `DATABASE_WRITE_FAILED`。
3. replay 返回 `idempotent_replay=true`。
4. 首次执行返回 `idempotent_replay=false`。
5. 同 key 不同 payload 返回幂等冲突。
6. operation 表不追加重复记录。
7. inspection 不被 replay 再次修改。
8. 不调用 ERPNext 写接口。

## ADR-049：TASK-006 前必须补跑 PostgreSQL 结算并发集成验证

- 决策时间：2026-04-13 15:57 CST
- 决策类型：测试门禁 / 数据库并发语义验证决策
- 关联模块：TASK-002H3 PostgreSQL 结算并发集成验证；TASK-006 加工厂对账单
- 审计来源：TASK-002H2 审计意见书第 53 份通过，剩余风险为 `POSTGRES_TEST_DSN` 未提供导致真实 PostgreSQL 并发语义未非 skip 验证
- 结论：进入 TASK-006 前必须在专用 PostgreSQL 测试库上验证 settlement lock/release 同 key 并发 replay；无 DSN 不算完成

### 背景

结算锁定/释放依赖 PostgreSQL 的行锁和唯一约束等待语义。SQLite 和 mock 测试能覆盖服务逻辑，但不能证明真实数据库并发等待、唯一冲突识别和事务提交后的 replay 行为。加工厂对账单会建立在这些锁定结果之上，因此必须在 TASK-006 前补一次非 skip 集成验证。

### 决策

1. 必须提供专用 `POSTGRES_TEST_DSN`。
2. 禁止连接生产或准生产数据库。
3. 必须注册并使用 `postgresql` pytest marker。
4. 同 key 并发 lock 必须两个请求成功，且只有一条 lock operation。
5. 同 key 并发 release 必须两个请求成功，且只有一条 release operation。
6. replay 响应必须 `idempotent_replay=true`。
7. 首次响应必须 `idempotent_replay=false`。
8. 同 key 不同 payload 必须返回幂等冲突。
9. PostgreSQL migration 必须确认 operation 唯一约束存在并生效。
10. 无 `POSTGRES_TEST_DSN` 时允许明确 skip，但 TASK-002H3 不算完成。
11. TASK-002H3 通过审计后，才允许进入 TASK-006。

### 验收

1. `-m postgresql` 非 skip 运行通过。
2. 同 key lock/release 并发测试通过。
3. operation 表唯一约束验证通过。
4. 常规全量 pytest/unittest/py_compile 继续通过。
5. 审计记录补记 PostgreSQL 非 skip 结果。

## ADR-050：PostgreSQL 破坏性集成测试必须双重安全门禁

- 决策类型：测试安全 / 数据保护决策
- 关联模块：TASK-002H4 PostgreSQL 测试库破坏性操作门禁整改
- 审计来源：TASK-002H3 审计意见书第 54 份，PostgreSQL 集成测试仅凭 `POSTGRES_TEST_DSN` 即执行 `DROP SCHEMA ly_schema CASCADE`

### 决策

1. `POSTGRES_TEST_DSN` 只表示“可连接 PostgreSQL”，不表示允许破坏性操作。
2. destructive SQL 前必须检查 `POSTGRES_TEST_ALLOW_DESTRUCTIVE=true`。
3. destructive SQL 前必须连接数据库并读取 `current_database()`。
4. 数据库名必须匹配 `*_test`、`test_*`、`lingyi_test_*` 或 `tmp_lingyi_*`。
5. 两项门禁必须同时满足。
6. 不满足门禁时必须 skip 或 fail，且不得执行 drop schema。
7. skip/fail 文案不得泄露完整 DSN、账号、密码。
8. PostgreSQL 测试输出允许显示脱敏数据库名和 schema 名。
9. TASK-002H4 不修改结算业务逻辑。
10. TASK-002H4 通过后，才能重新实跑 TASK-002H3。

### 验收

1. 无 allow env 时不执行 drop schema。
2. 非测试库名时不执行 drop schema。
3. allow=true 且测试库名匹配时才允许 destructive SQL。
4. 门禁测试覆盖通过。
5. 常规 pytest/unittest/py_compile 通过。

## ADR-052：PostgreSQL 非 Skip 集成测试必须成为 CI 硬门禁

- 决策时间：2026-04-13 16:35 CST
- 决策类型：CI 门禁 / 测试可靠性决策

- 审计来源：TASK-002H5 审计意见书第 56 份通过，剩余风险为缺 DSN、缺 allow env 或缺驱动时 PostgreSQL marker 会安全 skip，CI 若无执行数量断言可能误绿
- 结论：PostgreSQL marker 集成测试必须在 CI 中硬断言 `tests=4` 且 `skipped=0`；全部 skip 或执行数量不符必须失败

TASK-002H5 已证明结算 lock/release 并发语义可在一次性 PostgreSQL 测试库上非 skip 通过。但本地开发环境无 DSN 时仍会安全 skip，这是保护开发环境和避免误删数据的正确策略。CI 场景不同：CI 是发布前质量门禁，不能把 skip 当成功。因此必须增加专用 CI job 和报告断言，确保 PostgreSQL 并发语义持续被真实执行。

5. CI 必须安装 PostgreSQL 驱动依赖。
6. CI 必须运行 `pytest -q -m postgresql tests/test_subcontract_settlement_postgresql.py`。
7. CI 必须生成 JUnit XML 或等效测试报告。
8. CI 必须断言 PostgreSQL marker `tests=4`。
9. CI 必须断言 `skipped=0`。
10. CI 必须在 failures/errors 大于 0 时失败。
11. CI 缺 DSN、缺 allow env、缺驱动、报告缺失、测试数量不等于 4 或 skipped 大于 0 时必须失败。
12. 本地开发可继续安全 skip，但 CI job 不允许 skip 后通过。
13. CI 日志不得泄露完整 DSN、账号或密码。
14. TASK-002H6 通过审计后，才允许进入 TASK-006。

### 验收

1. CI hard gate 脚本存在并可解析 pytest JUnit XML。
2. skipped=4 时断言失败。
3. tests=0、tests=3、tests=5 时断言失败。
4. tests=4 且 skipped=0 时断言通过。
5. CI workflow 或等效 CI job 配置一次性 PostgreSQL 测试库。
6. CI workflow 执行 PostgreSQL marker 并调用 hard gate 断言。

## ADR-058：生产计划创建 Work Order 必须采用 Outbox / Worker

- 决策时间：2026-04-13 17:28 CST
- 决策类型：跨系统一致性 / 生产计划集成决策
- 关联模块：TASK-004A 生产计划后端基线与 Work Order Outbox
- 架构来源：TASK-002 Stock Entry outbox 和 TASK-003 Job Card outbox 的一致性经验；生产计划同样涉及 FastAPI 本地事务与 ERPNext `Work Order` 跨系统写入
- 结论：生产计划写接口不得在本地事务提交前调用 ERPNext 创建 `Work Order`；必须先落本地计划、审计、outbox，再由 worker 短事务 claim 后调用 ERPNext

### 背景

生产计划集成需要把 ERPNext `Sales Order` 转换为 `Work Order` 和 `Job Card`。如果请求事务内直接调用 ERPNext，一旦 ERPNext 已创建工单但 FastAPI 本地事务回滚，会出现本地计划、审计、工单映射不一致。TASK-002 和 TASK-003 已证明跨系统写入必须采用 outbox/worker 才能保证失败可重试、状态可追溯、审计可闭环。

### 决策

1. ERPNext `Sales Order` 是销售订单事实源。
2. FastAPI `ly_production_plan` 是本地生产计划事实源。
3. 创建 `Work Order` 必须先写 `ly_production_work_order_outbox`，不得在请求事务内直接调用 ERPNext。
4. Worker 必须采用短事务：先 claim 并提交，再调用 ERPNext，再短事务回写结果。
5. `event_key` 必须稳定，不得包含 `outbox_id/request_id/created_at/operator` 等易变字段。
6. ERPNext `Work Order` 必须使用 `custom_ly_plan_id/custom_ly_plan_no` 或等效自定义字段辅助幂等恢复。
7. Worker 找回 ERPNext `Work Order` 时必须校验 `docstatus` 和取消状态，不得把草稿、取消或异常状态误判为成功。
8. `Job Card` 同步只读取 ERPNext 并写入本地 `ly_production_job_card_link` 映射，完成数量仍由 TASK-003 工票链路负责。
9. BOM 中外发工序只在 TASK-004A 标记和映射，不自动创建外发单；外发单仍由 TASK-002 链路控制。
10. 权限资源来源必须来自 ERPNext `Sales Order`、BOM、Item、Company 和 ERPNext User Permission 聚合结果，不能信任前端传入 company。
11. 权限源、ERPNext 主数据源或数据库源不可用时必须 fail closed。
12. TASK-006 仍受 GitHub 平台门禁 pending 阻塞；TASK-004A 可以独立启动，但不得进入 TASK-006 对账开发。

### 验收

1. 生产计划创建接口只落本地计划和审计，不调用 ERPNext 创建 `Work Order`。
2. 创建 `Work Order` 接口只写 outbox 并返回 outbox 状态。
3. Worker claim 与 ERPNext API 调用不在同一个数据库事务内。
4. 相同计划和相同幂等请求稳定命中同一 outbox。
5. ERPNext 草稿、取消、异常 `Work Order` 不会被本地标记为成功。
6. Job Card 同步只写本地映射，不改工票计薪逻辑。
7. 权限、审计、错误信封和日志脱敏延续 TASK-001/TASK-003/TASK-002 标准。

## ADR-059：生产计划接口契约和 Work Order Worker Claim 必须作为复审门禁

- 决策时间：2026-04-13 18:13 CST
- 决策类型：审计整改 / 接口契约 / 并发一致性决策
- 关联模块：TASK-004A1 生产计划审计阻断整改
- 审计来源：审计意见书第 62 份，TASK-004A 不通过，高危 2 / 中危 3
- 结论：TASK-004A1 必须同时补齐请求体契约、仓库字段、Work Order outbox payload、原子 claim、lease 恢复、映射表约束和测试门禁；未闭环前不得进入 TASK-004B、TASK-005 或 TASK-006

10. `ly_production_job_card_link` 必须持久化 `company/item_code/operation_sequence/synced_at`，避免后续生产页面和工票模块只能绕回计划表推断资源范围。
11. TASK-004A1 复审前必须补齐仓库契约、DB/AUDIT 失败、Closed/Cancelled SO、BOM mismatch、worker 并发和 lease 恢复测试。
12. 本次整改不得进入前端、生产入库、外发自动创建、TASK-005 或 TASK-006。

### 验收

1. `material-check` 缺少仓库返回 `PRODUCTION_WAREHOUSE_REQUIRED`。
2. `create-work-order` 缺少成品仓或 WIP 仓返回 `PRODUCTION_WAREHOUSE_REQUIRED`。
3. Work Order outbox payload 含 `fg_warehouse/wip_warehouse/start_date`。
4. 相同 idempotency_key 相同 payload 能 replay，相同 idempotency_key 不同 payload 返回冲突。
5. 并发 worker 不重复调用 ERPNext。
6. lease 过期的 `processing` outbox 可以恢复。
7. `ly_production_work_order_link.plan_id` 唯一约束生效。
8. Job Card 映射写入 `company/item_code/operation_sequence/synced_at`。
9. 全量 pytest、unittest、py_compile 通过。
10. 未修改外发结算、工票计薪、TASK-005、TASK-006。

## ADR-061：`planned_start_date` 是生产计划事实字段，Work Order `start_date` 是工单同步参数

- 决策时间：2026-04-13 19:14 CST
- 决策类型：前后端契约 / 数据事实边界决策
- 关联模块：TASK-004B1 生产计划前端契约整改
- 审计来源：审计意见书第 64 份，TASK-004B 有条件通过，中危 3 / 低危 1
- 结论：`planned_start_date` 必须作为本地生产计划事实字段保存和返回；Work Order `start_date` 仅作为创建 ERPNext Work Order outbox 的工单开工参数，两者允许默认联动但不得混用或互相替代

### 背景

TASK-004B 前端已经实现生产计划页面，但审计发现用户在新建计划弹窗中选择 `planned_start_date` 后，payload 没有提交，后端也没有返回该字段，列表和详情会长期显示空值。这会让生产计划员误判计划开工信息。与此同时，`create-work-order` 已有 `start_date` 字段，它服务于 ERPNext Work Order 创建，不应替代本地生产计划事实字段。

### 决策

1. `planned_start_date` 是 `ly_production_plan` 的业务事实字段。
2. `planned_start_date` 必须进入创建生产计划请求 DTO。
3. `planned_start_date` 必须由后端保存。
4. `planned_start_date` 必须在生产计划列表 DTO 中返回。
5. `planned_start_date` 必须在生产计划详情 DTO 中返回。
6. 创建计划幂等 `request_hash` 必须包含 `planned_start_date`。
7. 相同 idempotency_key 但不同 `planned_start_date` 必须返回幂等冲突。
8. Work Order `start_date` 是创建 ERPNext Work Order outbox 的请求字段。
9. 前端创建 Work Order 时可以默认把 `planned_start_date` 带入 `start_date`，但提交字段必须仍叫 `start_date`。
10. 前端不得删除 `planned_start_date` 输入来规避契约问题。
11. Work Order 映射详情必须展示后端返回的 `erpnext_docstatus/erpnext_status/sync_status/last_synced_at`。
12. 物料快照详情必须展示后端返回的 `checked_at`。
13. 前端必须在 actions 和 button_permissions 两层过滤内部 worker 权限。
14. 本次整改不得进入 TASK-005/TASK-006。

### 验收

1. 创建计划提交 `planned_start_date`。
2. 后端保存并在列表/详情返回 `planned_start_date`。
3. 不同 `planned_start_date` 触发幂等冲突。
4. Work Order `start_date` 仍作为独立字段提交。
5. `work_order_worker` 按钮权限被前端强制清零。
6. Work Order 映射和物料快照展示真实 DTO 字段。
7. 状态中文标签覆盖任务单要求。
8. 前端无内部 worker 调用、无 ERPNext 直连、无裸 `fetch()`。
9. 后端 production 定向和全量回归通过。
10. 未进入 TASK-005/TASK-006。

## ADR-074：C11 平台证据 SHA 口径必须区分准备提交、证据提交和待推送 HEAD

- 决策时间：2026-04-13 21:59 CST
- 决策类型：平台证据 / SHA 口径治理
- 关联模块：TASK-004C12 C11 证据 SHA 口径修正
- 审计来源：审计意见书第 77 份，TASK-004C11 有条件通过；最高风险为 C11 证据 SHA 口径不清，需明确 `fc0dc2c`、`62e70bd` 与“待推送 HEAD”的关系
- 结论：C11 平台证据必须明确区分 C11 文档准备提交、C11 平台证据提交、C12 修正提交和最终远端 `main` HEAD；缺少该口径不得进入 GitHub 平台实跑和 TASK-005/TASK-006

### 背景

TASK-004C11 尝试推进 GitHub 平台最终闭环，但当前本地仍没有 remote，无法完成 push、hosted runner 和 required check。审计意见书第 77 份指出，C11 证据中对 `fc0dc2c`、`62e70bd` 和“待推送 HEAD”的关系描述不够清楚，管理员后续可能把准备提交、证据提交或待推送 HEAD 混用，导致 hosted runner 证据无法与真实远端 commit 对齐。

### 决策

1. `b32585c` 定义为 TASK-004C10 前基线提交，用于记录 C9 blocker docs。
2. `fc0dc2c` 定义为 TASK-004C11 文档准备提交，包含 C11 任务单、第 76 份审计后续架构记录等。
3. `62e70bd` 定义为 TASK-004C11 平台证据 docs-only 提交，是第 77 份审计时的本地待推送 HEAD。
4. TASK-004C12 若形成新的 docs-only 修正提交，则 C12 新提交成为新的本地待推送 HEAD；`62e70bd` 只能作为其父提交记录。
5. GitHub Hosted Runner 的最终证据必须以实际 push 到 GitHub 的 `main` HEAD 为准，Run URL 里的 Commit SHA 必须等于远端 `main` HEAD。
6. TASK-004C12 只能修正文档证据和架构记录，禁止配置 remote、禁止 push、禁止修改前后端业务代码。
7. TASK-004C12 docs-only commit message 固定为 `docs: clarify frontend ci platform sha lineage`。
8. TASK-005/TASK-006 在 GitHub platform closure 和复审通过前继续阻塞。

### 验收

1. `TASK-004C11_GitHub平台最终闭环证据.md` 新增 SHA 口径修正章节。
2. `TASK-004C12_C11证据SHA口径修正证据.md` 记录提交链、当前 HEAD、是否配置 remote 和最终远端 HEAD 判定规则。
3. `git log --oneline -6` 与证据文档中的 SHA 关系一致。
4. staged 文件清单仅包含白名单文档。
5. 敏感信息扫描通过。
6. 未配置 remote、未 push、未进入 TASK-005/TASK-006。
