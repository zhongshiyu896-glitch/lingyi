## 一、业务目标

加工厂对账单用于月底财务与外发加工厂核对加工费、扣款和实付金额。系统按加工厂和日期范围汇总外发加工的回料、验货、扣款事实，生成可打印、可确认的对账单；对账单确认后才允许进入 ERPNext 应付草稿生成流程。

本模块解决以下问题：

1. 将 TASK-002 外发加工中的收货、验货、扣款事实汇总成加工厂维度对账单。
2. 防止同一加工厂、同一期间、同一外发明细被重复对账。
3. 锁定已确认对账单金额，避免确认后被外发明细变更污染。
4. 为 ERPNext `Purchase Invoice` 应付草稿提供可信来源和审计链路。
5. 为后续打印、财务确认、调整、取消重建提供统一业务边界。

## 二、系统边界

### FastAPI 自建负责

| 范围 | 说明 |
| --- | --- |
| 对账单生成 | 按 `company + supplier + from_date + to_date` 从外发验货/结算事实生成本地对账单草稿。 |
| 对账明细快照 | 将每条外发可对账明细的数量、扣款、净额、来源单据状态快照写入 `ly_factory_statement_item`。 |
| 幂等与重复防护 | 按 `idempotency_key`、`request_hash` 和明细唯一约束防止重复生成或重复确认。 |
| 对账确认 | 本地状态从 `draft` 变为 `confirmed`，确认后锁定明细金额。 |
| 应付草稿触发 | 确认后才能触发 ERPNext `Purchase Invoice` 草稿生成；具体实现须在后续 TASK-006D 走独立任务单。 |
| 权限与审计 | 执行登录鉴权、动作权限、供应商/公司资源权限、安全审计和操作审计。 |

### ERPNext 负责

| ERPNext 能力 | 用途 | 边界 |
| --- | --- | --- |
| `Supplier` | 校验加工厂供应商存在、状态可用、归属公司可访问。 | 不在 FastAPI 复制供应商主数据。 |
| `Account` | 校验应付科目合法。 | FastAPI 只保存本次请求使用的 account 引用。 |
| `Cost Center` | 校验成本中心合法。 | FastAPI 只保存引用，不维护成本中心主数据。 |
| `Purchase Invoice` | 生成应付草稿。 | 不允许在生成对账单草稿时直接写 ERPNext 应付。 |
| `Role / User Permission` | 权限权威来源。 | FastAPI 可缓存/聚合动作权限，但生产口径以 ERPNext 权限为准。 |

| 接口名称 | 方法 | 路径 | 入参 | 出参 | 权限动作 |

| 取消对账单 | POST | `/api/factory-statements/{id}/cancel` | `idempotency_key, reason` | `name, status` | `factory_statement:cancel` |

### 接口边界

1. `confirmed_by` 不允许从前端入参写入，必须使用当前登录用户。
2. `POST /api/factory-statements/` 不允许直接创建 ERPNext `Purchase Invoice`。
3. `POST /api/factory-statements/{id}/payable-draft` 只能在 `status='confirmed'` 后调用。
4. `payable-draft` 是否采用 outbox、同步 REST 或异步 worker，必须在 TASK-006D 单独设计和审计后实现。
5. 详情接口必须先做动作权限，再查询具体 ID，避免 403/404 枚举存在性。

## 五、业务流程

### 1. 生成对账单草稿

1. 用户选择公司、加工厂、起止日期。
2. 后端校验登录态、`factory_statement:create` 动作权限和加工厂资源权限。
3. 后端通过 ERPNext 校验 Supplier 存在且可用；ERPNext 不可用时 fail closed。
4. 后端读取 TASK-002 外发验货/结算事实，筛选可对账明细。
5. 后端检查这些明细是否已进入未取消对账单。
6. 后端计算明细金额并保存快照。
7. 后端汇总生成对账单草稿和操作日志。

### 2. 确认对账单

1. 用户打开草稿对账单详情。
2. 后端先做 `factory_statement:confirm` 动作权限和资源权限。
3. 后端行锁锁定对账单，确认状态仍为 `draft`。
4. 后端复核明细金额与头表汇总一致。
5. 后端写入 `confirmed_by=current_user`、`confirmed_at`，状态改为 `confirmed`。
6. 确认后明细金额不得修改。

### 3. 生成 ERPNext 应付草稿

1. 仅 `confirmed` 对账单允许生成应付草稿。
2. 后端校验 `factory_statement:payable_draft` 权限。
3. 后端校验 Supplier、Account、Cost Center。
4. ERPNext 不可用、权限源不可用或主数据不可校验时 fail closed。
5. 后续 TASK-006D 必须明确是否走 outbox；在未通过 TASK-006D 审计前不得直接实现生产写入。

## 六、金额与状态规则

### 金额公式

| 指标 | 公式 |
| --- | --- |
| 加工费 | `gross_amount = sum(item.gross_amount)` |
| 扣款 | `deduction_amount = sum(item.deduction_amount)` |
| 实付金额 | `net_amount = gross_amount - deduction_amount` |
| 不合格率 | `rejected_rate = total_rejected_qty / inspected_qty`，`inspected_qty=0` 时为 `0` 或 fail closed，TASK-006A 需冻结。 |

验收例：加工费 `5000`，扣款 `300`，`net_amount` 必须返回 `4700`。

### 状态机

```text
draft -> confirmed -> payable_draft_created
  |          |
  v          v
cancelled   cancelled(仅在未生成应付草稿或反冲策略明确后允许)
```

V0.1 冻结口径：

1. `draft` 可取消，可重新生成。
2. `confirmed` 锁定明细金额，不允许直接修改。
3. `confirmed` 后如需调整，必须走调整记录或取消后重建，不允许原地改金额。
4. 已生成 ERPNext 应付草稿后的取消/反冲规则必须单独进入 TASK-006E 设计，不允许本阶段隐式实现。

## 七、权限、审计与 Fail Closed

### 权限动作

| 动作 | 说明 |
| --- | --- |
| `factory_statement:read` | 查询列表、详情、打印。 |
| `factory_statement:create` | 生成草稿。 |
| `factory_statement:confirm` | 确认对账单。 |
| `factory_statement:payable_draft` | 生成 ERPNext 应付草稿。 |
| `factory_statement:cancel` | 取消对账单。 |

### 资源权限

1. 列表必须按用户可访问的 `company + supplier` 过滤。
2. 详情、确认、应付草稿、取消必须做资源级校验。
3. Company-only 权限不能替代 Supplier 权限，无法解析 Supplier 权限时 fail closed。
4. ERPNext `Role / User Permission` 或 FastAPI 权限聚合不可用时，不得放行敏感写操作。

### 审计要求

1. 401、403、503 权限源不可用必须写安全审计。
2. create、confirm、payable_draft、cancel 成功和失败必须写操作审计。
3. 审计日志不得写入 Authorization、Cookie、密码、Secret、Token 明文。
4. `request_id` 必须使用统一归一化逻辑，禁止原样记录敏感 request id。

## 十、验收标准

1. TASK-006A 输出基线盘点文档，明确外发来源字段、金额公式、幂等规则、权限规则和 ERPNext 边界。
2. TASK-006B 之后 `POST /api/factory-statements/` 输入 `supplier/from_date/to_date/idempotency_key` 可生成草稿并返回 `data.name`。
3. 同一供应商、同一日期范围、同一外发来源明细不得重复进入未取消对账单。
4. 加工费 `5000`、扣款 `300` 时，`net_amount` 返回 `4700`。
5. `GET /api/factory-statements/{id}` 返回 `statement/items/logs`。
6. 对账单确认后不允许修改明细金额。
7. 无供应商资源权限的用户无法查看或操作对账单。
8. ERPNext 不可用时，对账确认和应付草稿生成必须 fail closed。
9. 应付草稿生成不得绕过审计日志和幂等键。

## 十一、V0.2 补充：TASK-006B 后端基线冻结

- 更新时间：2026-04-15 17:25 CST
- 触发来源：TASK-006A 审计意见书第 161 份通过
- 结论：允许进入 TASK-006B，但仅限后端模型、迁移、草稿生成、列表和详情基线。

### 1. 006A 已冻结事实

1. 对账来源粒度固定为 `ly_subcontract_inspection` 验货记录粒度。
2. 金额公式固定为 `gross_amount=sum(item.gross_amount)`。
3. 金额公式固定为 `deduction_amount=sum(item.deduction_amount)`。
4. 金额公式固定为 `net_amount=gross_amount-deduction_amount`。
5. `inspected_qty=0` 时 `rejected_rate=0`。
6. 幂等固定为 `idempotency_key + request_hash`。
7. 异 hash 冲突返回 `FACTORY_STATEMENT_IDEMPOTENCY_CONFLICT`。
8. TASK-006B 禁止创建 ERPNext `Purchase Invoice`。

## 十三、V0.4 补充：TASK-006C 确认、取消释放与冲突语义

### 2. payable-draft 请求接口边界

`POST /api/factory-statements/{id}/payable-draft` 只负责：

1. 校验动作权限和资源权限。
2. 校验 statement 必须为 `confirmed`。
3. 校验 `payable_account/cost_center` 在 ERPNext 中存在且属于 company。
4. 创建本地 payable outbox。
5. 写操作审计。

该接口不得：

1. 直接创建 ERPNext Purchase Invoice。
2. 伪造 `purchase_invoice_name`。
3. 提交 ERPNext Purchase Invoice。
4. 创建付款或 GL。

### 3. worker 边界

内部 worker 负责：

1. claim due outbox 并先提交 claim。
2. 调用 ERPNext 创建 `Purchase Invoice` 草稿。
3. 校验 ERPNext 返回 `docstatus=0`。
4. 回写 outbox 和 statement。
5. 失败时进入 failed/dead 重试状态。

worker 必须服务账号最小权限控制，普通财务角色不得调用 internal worker。

### 4. 幂等和事件键

payable outbox 必须具备：

1. `company + statement_id + idempotency_key` 请求幂等。
2. `request_hash` 覆盖 statement、supplier、net_amount、payable_account、cost_center、posting_date、remark。
3. `event_key` 不包含 outbox_id、request_id、created_at、operator 等易变字段。
4. ERPNext 已存在同 event_key 草稿时可 replay；已提交/取消或未知状态必须 fail closed。

### 5. 审计和错误信封

1. 权限拒绝、资源拒绝、权限源不可用必须写安全审计。
2. outbox 创建、worker 成功/失败、dry-run、retry 必须写操作审计。
3. ERPNext 不可用返回 `FACTORY_STATEMENT_ERPNEXT_UNAVAILABLE`。
4. ERPNext 返回非草稿 PI 返回 `FACTORY_STATEMENT_ERPNEXT_PURCHASE_INVOICE_INVALID_STATUS`。
5. 所有响应保持统一 `{code, message, data/detail}` 错误信封。

## 十五、V0.6 补充：TASK-006D1 Payable Outbox 与 Worker 状态机安全收口

- 更新时间：2026-04-15 19:47 CST
- 触发来源：TASK-006D 审计不通过，审计意见书第 165 份
- 结论：进入 TASK-006E 前，必须先修复 payable outbox 与 worker 状态机安全问题。

### 1. cancel 与 payable outbox 互斥

对账单存在以下 active payable outbox 时，取消必须 fail closed：

```text
pending
processing
succeeded
```

取消失败不得释放 `ly_subcontract_inspection`，不得修改 statement 状态，必须返回明确错误码并写操作失败审计。

### 2. worker 外部调用前本地状态复核

payable worker 在任何 ERPNext `find/create` 调用前，必须重新读取 statement 并校验：

1. statement 存在。
2. outbox 属于该 statement。
3. statement 状态仍允许处理。
4. statement 不是 `cancelled/draft/payable_draft_created`。
5. payload_hash/request_hash 与冻结金额口径一致。

### 3. claim due/lease 原子条件

`claim_due()` 第二阶段 `UPDATE` 必须重复校验 due/lease 条件：

```text
(status IN ('pending','failed') AND (next_retry_at IS NULL OR next_retry_at <= now))
OR
(status = 'processing' AND locked_until < now)
```

不得抢占 `processing` 且 `locked_until > now` 的未过期 lease。
