## 1. 业务目标

工票/车间管理模块负责记录车间工人对生产工序的实际完成数量，并根据工价档案计算员工日薪。该模块是生产执行层和工资结算层的核心，向上关联 ERPNext `Job Card / Work Order`，向下兼容人工录入、PDA 扫码、MES 推送等数据来源。

目标包括：

1. 建立工票登记、撤销、批量导入能力。
2. 建立本厂工序计件工价档案。
3. 工票创建时快照计件单价，保证历史工资可追溯。
4. 按员工、日期、工序、款式汇总日薪。
5. 与 ERPNext `Job Card` 建立关联并同步本地净完成数量。
6. 沿用 BOM 模块已落地的鉴权、安全审计、日志脱敏和 request_id 规范。

用途：工票事实表，登记与撤销都保留为独立记录。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `ticket_no` | 系统工票编号 |
| `ticket_key` | 外部幂等键 |
| `job_card` | ERPNext `Job Card.name` |
| `work_order` | ERPNext `Work Order.name` |
| `bom_id` | `ly_apparel_bom.id` |

| `employee` | ERPNext `Employee.name` |
| `process_name` | 工序名称 |
| `color` | 颜色 |
| `size` | 尺码 |
| `operation_type` | `register/reversal` |
| `qty` | 数量，必须大于 0 |

| `source` | `manual/pda/mes/import` |

1. 幂等唯一索引：`ticket_key, process_name, color, size, operation_type, work_date`。

用途：本厂工序计件工价。

关键字段：

### 3.4 `ly_schema.ys_workshop_job_card_sync_outbox`

用途：保存待同步 ERPNext `Job Card` 的异步任务。

关键字段：`event_key, job_card, work_order, item_code, company, local_completed_qty, source_type, source_ids, status, attempts, max_attempts, next_retry_at, locked_by, locked_at, last_error_code, last_error_message, request_id, created_by, created_at, updated_at`。

状态：`pending/processing/succeeded/failed/dead`。

核心约束：

1. `event_key` 全局唯一。
2. `status + next_retry_at` 建索引，支持 worker 扫描待处理任务。
3. 同步任务必须在本地工票事务中写入，但 ERPNext 调用必须发生在本地事务提交之后。

### 3.5 `ly_schema.ys_workshop_job_card_sync_log`

用途：记录本地工票净数量同步 ERPNext `Job Card` 的结果。

关键字段：`outbox_id, job_card, sync_type, local_completed_qty, attempt_no, erpnext_status, error_code, error_message, request_id, created_at`。

## 5. 业务规则

1. 工票登记和撤销都新增记录，不物理删除。
2. 幂等键相同且 payload 一致时返回原记录。
3. 幂等键相同但 payload 不一致时返回 `WORKSHOP_IDEMPOTENCY_CONFLICT`。
4. 撤销数量不得超过同维度已登记未撤销数量。
5. 日薪按工票单价快照计算，不随工价档案后续变化重算。
6. 工价匹配优先级：款式专属工价优先，通用工价兜底。
7. Job Card 同步失败不回滚本地工票，但必须标记 `sync_status=failed` 并支持重试。
8. 款式专属工价必须绑定明确 `company`，历史 `item_code IS NOT NULL AND company IS NULL` 工价不得参与计薪。

### 5.1 批量导入异常边界

`POST /api/workshop/tickets/batch` 必须明确区分行级业务失败和系统级失败。

允许进入 `failed_items` 的行级业务失败：

1. `WORKSHOP_INVALID_QTY`
2. `WORKSHOP_JOB_CARD_NOT_FOUND`
3. `WORKSHOP_EMPLOYEE_NOT_FOUND`
4. `WORKSHOP_JOB_CARD_STATUS_INVALID`
5. `WORKSHOP_PROCESS_MISMATCH`
6. `WORKSHOP_ITEM_MISMATCH`
7. `WORKSHOP_WAGE_RATE_NOT_FOUND`
8. `WORKSHOP_IDEMPOTENCY_CONFLICT`
9. `WORKSHOP_REVERSAL_EXCEEDS_REGISTERED`
10. 单行资源级 `AUTH_FORBIDDEN`

必须中断批次并向上返回标准错误响应的系统级失败：

1. `AUTH_UNAUTHORIZED`
2. 缺少 `workshop:ticket_batch` 的动作级 `AUTH_FORBIDDEN`
3. `PERMISSION_SOURCE_UNAVAILABLE`
4. `ERPNEXT_SERVICE_UNAVAILABLE`
5. `AUDIT_WRITE_FAILED`
6. `DATABASE_WRITE_FAILED`
7. `DATABASE_READ_FAILED`
8. `WORKSHOP_INTERNAL_ERROR`

系统级失败禁止包装成 `200/code=0` 的普通失败行。

## 6. 公式口径

1. `available_qty = sum(register.qty) - sum(reversal.qty)`。
2. `reversal.qty <= available_qty`。
3. `ticket.wage_amount = ticket.qty * ticket.unit_wage`。
4. `register_amount = sum(register.qty * register.unit_wage)`。
5. `reversal_amount = sum(reversal.qty * reversal.unit_wage)`。
6. `daily_wage_amount = register_amount - reversal_amount`。
7. `net_qty = register_qty - reversal_qty`。
8. `ticket.unit_wage` 必须来自 `item_code/company/process_name/work_date` 同口径匹配的 active 工价。
9. `item_code IS NOT NULL AND company IS NULL` 的工价不得参与 `ticket.unit_wage` 计算。

## 7. ERPNext 集成方案

1. 登记前通过 ERPNext REST API 校验 `Job Card` 存在。
2. 登记前通过 ERPNext REST API 校验 `Employee` 有效。
3. Job Card 状态为 `Cancelled/Closed` 时禁止新增工票。
4. Job Card `operation` 必须与 `process_name` 一致。
5. 工票写入成功后，在本地事务中写入 `ys_workshop_job_card_sync_outbox`。
6. FastAPI 只通过 ERPNext API 修改 Job Card，不直接写 ERPNext 数据库表。
7. ERPNext Job Card 更新必须发生在本地事务提交之后，由 after-commit 触发、后台 worker 或手动重试处理。
8. Worker 必须使用最终态覆盖方式同步 `completed_qty`，不得使用增量累加方式。
9. ERPNext 同步失败不回滚本地工票事实，必须更新 outbox 状态并写同步日志。

### 7.1 Outbox 同步规则

1. 本地事务中允许写工票、日薪、操作审计和 outbox。
2. 本地事务中禁止调用 ERPNext 写接口。
3. 本地 commit 失败时，不得调用 ERPNext。
4. 操作审计失败导致本地事务回滚时，不得调用 ERPNext。
5. outbox 写入失败时，本地事务整体回滚并返回 `DATABASE_WRITE_FAILED`。
6. outbox pending/failed 任务由 worker 按 `next_retry_at` 拉取处理。
7. 同一 outbox 重试必须幂等，不得导致 ERPNext `completed_qty` 重复累加。
8. 超过最大重试次数后，outbox 状态置为 `dead`。
9. 服务账号必须最小权限，只允许读 `Job Card/Work Order` 和更新 Job Card 完成数量。
10. 内部 Worker HTTP 触发接口必须只允许服务账号或系统级集成账号调用，普通业务角色不得触发批量同步。

### 7.2 Outbox Event Key 规则

`event_key` 用于 outbox 幂等，必须避免长 `job_card` 截断导致 digest 丢失。

生成规则：

1. 禁止先拼接 `job_card:digest` 再截断。
2. 禁止截断 digest。
3. 推荐格式：`wjc:{sha256_hex}`。
4. `sha256_hex` 长度为 64，最终 `event_key` 长度为 68，小于 `varchar(140)`。
5. hash 输入必须包含 `job_card`、`local_completed_qty`、`source_type`、排序后的 `source_ids`。
6. `source_ids` 必须排序后参与摘要。
7. `local_completed_qty` 必须数值规范化，确保 `1`、`1.0`、`1.000000` 语义相同时生成相同 key。
8. `event_key` 不保存完整 `job_card` 明文；排查使用 outbox 独立 `job_card` 字段。
9. 138/139/140/200 字符 `job_card` 下，不同同步事件仍必须生成不同 `event_key`。

### 8.1 写接口资源级权限

工票写接口不能只校验 `workshop:*` 动作权限，必须同时校验目标资源权限。

资源识别规则：

1. 登记、撤销、批量导入、Job Card 同步必须先读取 ERPNext `Job Card`。
2. 由 `Job Card.work_order` 继续读取 ERPNext `Work Order`。
3. 从 `Work Order.production_item` 派生真实 `item_code`。
4. 从 `Work Order.company` 派生真实 `company`。
5. 前端传入的 `item_code` 不可信，只能用于一致性校验，不能作为权限依据。
6. 请求体 `item_code` 与 ERPNext 派生 `item_code` 不一致时返回 `WORKSHOP_ITEM_MISMATCH`。
7. 无法派生 `item_code` 时返回 `WORKSHOP_JOB_CARD_ITEM_NOT_FOUND`。
8. 无法派生 `company` 时返回 `WORKSHOP_JOB_CARD_COMPANY_NOT_FOUND`。
9. 权限来源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`，禁止 fail open。

必须校验资源权限的接口：

| 接口 | 资源权限 |
| --- | --- |
| `POST /api/workshop/tickets/register` | Job Card 派生 `item_code/company` |
| `POST /api/workshop/tickets/reversal` | 原工票 + Job Card 派生 `item_code/company` |
| `POST /api/workshop/tickets/batch` | 每行 Job Card 派生 `item_code/company` |
| `POST /api/workshop/job-cards/{job_card}/sync` | Job Card 派生 `item_code/company` |
| `POST /api/workshop/wage-rates` | `item_code` 资源权限；通用工价需全局权限 |
| `POST /api/workshop/wage-rates/{id}/deactivate` | 目标工价 `item_code` 资源权限；通用工价需全局权限 |

### 8.2 工价资源权限

工价属于工资与成本敏感数据，读取和维护都必须执行资源级权限。

资源规则：

1. `GET /api/workshop/wage-rates` 必须按数据库查询层过滤 `item_code/company`，禁止先查全量再由前端过滤。
2. 用户有明确 Item 限制时，只返回 `item_code in allowed_items` 的款式专属工价。
3. 用户同时有 Company 限制时，还必须满足 `company in allowed_companies`。
4. 用户只有 Company 权限、没有 Item 权限时，默认不得读取款式专属工价；Company-only 不等于 Item 权限。
5. 仅当 ERPNext 明确返回 Item unrestricted 时，才能按 Company 限制读取款式专属工价。
6. `POST /api/workshop/wage-rates` 必须完成 `item_code -> company` 解析。
7. 无法解析 company 时返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`，禁止写入。
8. 创建或停用指定 `item_code` 工价时，必须同时满足 `workshop:wage_rate_manage`、目标 Item 权限和目标 Company 权限。
9. 通用工价 `item_code` 为空，必须具备 `workshop:wage_rate_read_all` 才可读取，具备 `workshop:wage_rate_manage_all` 才可创建或停用。
10. 权限来源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`，禁止 fail open。

### 8.2.1 历史工价 Company 治理

`ly_operation_wage_rate.item_code IS NOT NULL AND company IS NULL` 定义为历史脏数据。

规则：

1. 款式专属工价必须绑定明确 `company`。
2. `company=NULL` 不得用于款式专属 active 工价。
3. `company=NULL` 只允许用于明确的全局通用工价，并且必须满足 `item_code IS NULL` 和全局工价权限。
4. 工价列表、工价维护、工票登记计薪必须使用同一套 `item_code/company` scope。
5. 工票登记计薪不得命中 `item_code IS NOT NULL AND company IS NULL` 的历史工价。
6. 创建或更新款式专属工价时，`company` 为空必须返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`。
7. 历史工价补数必须记录补数日志；无法唯一解析 company 的 active 工价必须停用或标记 `blocked_scope`。
8. 计薪只允许使用 ERPNext `Job Card -> Work Order` 派生的真实 company，不得使用前端传入 company。

### 8.2.2 Company 缺失规范化

`NULL`、空字符串 `''`、空白字符串 `'   '` 在工价 company 语义上全部等价为“缺失 company”。

规则：

1. `is_missing_company(company) = company IS NULL OR trim(company) = ''`。
2. `normalize_company(company) = NULL if is_missing_company(company) else trim(company)`。
3. 迁移补数扫描必须覆盖 `company IS NULL OR btrim(company) = ''`。
4. 工价列表不得返回 active `item_code IS NOT NULL AND is_missing_company(company)` 的记录。
5. 工票登记计薪不得命中 active `item_code IS NOT NULL AND is_missing_company(company)` 的记录。
6. 创建或更新款式专属工价时，normalize 后 company 为空必须返回 `WORKSHOP_WAGE_RATE_COMPANY_REQUIRED`。
7. 如果计薪发现唯一候选是缺失 company 的历史 item 工价，必须返回 `WORKSHOP_WAGE_RATE_SCOPE_REQUIRED`，不是 `WORKSHOP_WAGE_RATE_NOT_FOUND`。
8. `WORKSHOP_WAGE_RATE_NOT_FOUND` 只用于确实不存在任何有效工价候选的场景。

### 8.2.3 工价补数 Dry-Run 只读语义

`backfill_wage_rate_company_scope(dry_run=True)` 必须是真正只读预演。

规则：

1. dry-run 路径不得调用 `session.add()`、`session.add_all()`、`session.merge()`、`session.delete()`、`session.flush()`、`session.commit()`。
2. dry-run 路径不得修改已加载 ORM 实例字段。
3. dry-run 路径不得创建并挂载 `ly_operation_wage_rate_company_backfill_log` ORM 对象。
4. dry-run 返回值必须使用普通 dict、dataclass、Pydantic schema 或不可持久化 DTO。
5. dry-run 执行结束后，`session.new/session.dirty/session.deleted` 必须为空。
6. dry-run 后调用方即使执行 `session.commit()`，也不得新增补数日志、不得修改工价、不得新增审计。
7. 推荐将补数流程拆为 `build_plan` 和 `apply_plan` 两阶段；dry-run 只执行 plan，正式执行才 apply。
8. 正式执行 `dry_run=False` 仍必须写补数日志并更新工价 company/status。

### 8.2.4 工价补数读取失败 Fail Closed

补数链路中数据库读取失败属于系统级失败，不得降级为业务解析失败。

规则：

1. `_resolve_backfill_companies()` 捕获 `SQLAlchemyError` 时必须转换为 `DATABASE_READ_FAILED`。
2. 数据库读取失败不得返回空候选、`ambiguous_company` 或 `company_unresolved`。
3. `build_wage_rate_company_backfill_plan()` 遇到 `DATABASE_READ_FAILED` 必须立即中断，不得返回成功计划。
4. `backfill_wage_rate_company_scope(dry_run=True)` 读取失败时不得返回成功 dry-run 报告。
5. `backfill_wage_rate_company_scope(dry_run=False)` 读取失败时必须 rollback，不得更新工价、不得写补数日志。
6. 业务上的 `ambiguous_company/company_unresolved` 只允许在数据库读取成功、事实完整但业务无法唯一判断时产生。
7. 读取失败日志必须脱敏，不得记录 SQL 原文、参数、连接串、密码、token、Authorization、Cookie。

### 8.2.5 ERPNext Item 候选查询不可用 Fail Closed

补数链路依赖 ERPNext Item 主数据判断历史工价所属 company。ERPNext Item 候选查询不可用属于外部主数据源不可用，不得被当成“查询成功但无候选”。

规则：

1. ERPNext Item 候选查询超时、连接失败、HTTP 非成功、响应解析失败或权限源不可用时，必须 fail closed。
2. 如果候选查询通过 ERPNext REST API 执行，不可用时返回 `ERPNEXT_SERVICE_UNAVAILABLE` 或等价主数据源不可用错误。
3. 如果候选查询通过共享 PostgreSQL `public.tabItem` 执行，SQLAlchemy 读取失败时返回 `DATABASE_READ_FAILED`。
4. 不可用场景不得返回空候选集合。
5. 不可用场景不得返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
6. 不可用场景不得把历史工价停用、标记 `blocked_scope` 或写入补数日志。
7. dry-run 遇到 ERPNext Item 候选查询不可用时不得返回成功预演报告。
8. execute 遇到 ERPNext Item 候选查询不可用时必须 rollback，不得更新工价 company/status。
9. 只有在 ERPNext Item 查询成功且确认无可用候选时，才允许按业务规则返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
10. 错误日志和审计上下文必须脱敏，不得记录 ERPNext token、Cookie、Authorization、响应原文中的敏感字段。

### 8.3 内部 Worker 接口权限

`POST /api/workshop/internal/job-card-sync/run-once` 属于内部 Worker 触发接口，不是普通前端业务能力。

权限规则：

1. 调用主体必须已认证。
2. 调用主体必须具备 `workshop:job_card_sync_worker`。
3. 调用主体必须是服务账号、系统级集成账号或 `System Manager`。
4. 普通 `Workshop Manager / Workshop User / Production Manager` 即使具备 `workshop:job_card_sync`，也不得调用内部 Worker。
5. `workshop:job_card_sync` 仅用于 `POST /api/workshop/job-cards/{job_card}/sync` 单个 Job Card 手动重试。
6. `workshop:job_card_sync_worker` 不得作为普通前端按钮权限返回给普通用户。
7. 生产环境如保留 HTTP 内部 Worker 接口，必须支持 `ENABLE_INTERNAL_WORKER_API` 开关；未开启时返回 `INTERNAL_API_DISABLED`。
8. 未登录、权限不足、权限来源不可用、内部接口关闭时，不得锁定、更新或重试任何 outbox。
9. 未授权调用不得调用 ERPNext Job Card 写接口。
10. 所有拒绝必须写安全审计，成功执行必须写操作审计。
11. 安全审计和操作审计不得记录 Authorization、Cookie、service token、secret、password 明文。

### 8.4 服务账号最小权限策略

服务账号不能默认拥有全模块全资源。`workshop:job_card_sync_worker` 只代表可以运行内部 Worker，不代表可以处理所有 `company/item_code` 的 outbox。

策略来源：

1. 服务账号身份来源为 ERPNext `User + Role`。
2. 服务账号必须具备 `LY Integration Service` 或 `System Manager`。
3. 动作权限仍需具备 `workshop:job_card_sync_worker`。
4. 资源权限来源为 ERPNext `User Permission`。
5. 服务账号资源权限必须明确到 `Company` 和 `Item`。
6. 没有明确 Company 或 Item 授权时，默认 fail closed。
7. 禁止使用 `*`、`all`、空数组表示服务账号拥有全部资源。
8. ERPNext 权限来源不可用时返回 `PERMISSION_SOURCE_UNAVAILABLE`，不得处理 outbox。

Worker 处理规则：

1. 每处理一条 outbox 前，必须校验 outbox 的 `company` 和 `item_code`。
2. outbox 缺少 `company` 或 `item_code` 时，不得调用 ERPNext。
3. 服务账号对 outbox 的 `company/item_code` 无权限时，不得调用 ERPNext。
4. 推荐在查询 pending outbox 时按服务账号允许的 `company/item_code` 过滤，避免锁定未授权行。
5. 未授权 outbox 不得标记为 `succeeded`，不得写成功 sync_log。
6. 服务账号资源越权必须写安全审计。
7. 服务账号成功处理 outbox 必须写操作审计，并记录 `principal/outbox_id/job_card/item_code/company/request_id`。

### 8.5 Outbox 队头阻塞防护

服务账号 scope 过滤必须发生在主处理查询阶段，不能先按 `id asc limit` 取出 pending due 窗口，再在 Python 中逐行跳过越权行。

主处理查询顺序：

1. `status in ('pending', 'failed')`。
2. `next_retry_at <= now()`。
3. `company in allowed_companies`。
4. `item_code in allowed_items`。
5. `order by id asc`。
6. `limit batch_size`。

规则：

1. `limit` 只能作用在当前服务账号有权处理的候选集上。
2. 越权 outbox 不得进入当前服务账号主处理窗口。
3. 越权 outbox 不得被当前服务账号锁定、增加 attempts、更新 `locked_by/locked_at` 或标记成功。
4. 缺少 `company` 或 `item_code` 的 outbox 不属于任何服务账号可处理范围，必须转为 `dead` 或 `blocked_scope`，并写 `SERVICE_ACCOUNT_RESOURCE_SCOPE_REQUIRED` 安全审计。
5. 如果保留 `skipped_forbidden_count`，只能来自独立诊断扫描，不得消费主处理 `limit`。
6. 服务账号没有任何 Company/Item scope 时，不得查询全量 outbox。
7. 权限来源不可用时，不得执行 outbox 查询锁定或 ERPNext 调用。

### 8.6 越权诊断节流与审计去重

越权 outbox 对当前服务账号无权，但可能被其他服务账号处理，因此不得简单全部置为 dead；同时也不得在每轮 `run-once` 中重复枚举并写安全审计。

规则：

1. 默认 `run-once` 只处理当前服务账号有权的 outbox，不扫描越权 outbox。
2. 越权诊断必须显式启用，例如 `include_forbidden_diagnostics=true` 或独立运维任务。
3. 越权诊断必须有单次扫描上限，默认不超过 50 行。
4. 同一 `outbox_id + principal + reason_code + scope_hash` 在冷却期内只能写 1 条安全审计。
5. 推荐冷却期为 6 小时，可通过配置调整。
6. 冷却期内重复发现只更新 `seen_count/last_seen_at`，不得重复写安全审计。
7. 冷却期到期、scope 变化或 reason 变化时，才允许再次写安全审计。
8. 去重 key 不得包含 Authorization、Cookie、token、secret、password 明文。
9. 越权诊断不得锁定 outbox，不得增加 attempts，不得更新 `next_retry_at`，不得调用 ERPNext。
10. 缺 scope outbox 已转为 `dead/blocked_scope` 后，不得继续进入越权诊断扫描。

### 8.7 内部 Worker Dry-Run 审计

`dry_run=true` 不修改业务数据，但会读取内部 outbox、服务账号 scope、候选数量和诊断状态，属于敏感内部诊断行为。

规则：

1. 生产环境默认禁用 dry-run，除非 `WORKSHOP_ENABLE_WORKER_DRY_RUN=true`。
2. dry-run 被禁用时，返回 `WORKSHOP_DRY_RUN_DISABLED`，写安全审计，不查询 outbox。
3. dry-run 授权成功时，必须写操作审计。
4. dry-run 操作审计失败时返回 `AUDIT_WRITE_FAILED`，不得静默成功。
5. dry-run 不得锁定 outbox，不得增加 attempts，不得更新 `next_retry_at`，不得调用 ERPNext，不得写成功 sync_log。
6. dry-run 操作审计必须记录 `principal/dry_run/batch_size/include_forbidden_diagnostics/would_process_count/request_id`。
7. dry-run 操作审计不得记录 Authorization、Cookie、token、secret、password 明文。
8. dry-run 与 forbidden diagnostics 联用时，仍必须遵守诊断节流和安全审计去重规则。
9. 每一次授权 dry-run 成功都必须有 1 条操作审计，即使 `would_process_count=0`。

### 8.8 Dry-Run 禁用判断顺序

生产环境 dry-run 禁用是本地配置门禁，不得依赖 ERPNext 权限源可用性。

`dry_run=true` 必须按以下顺序处理：

1. 规范化 `request_id`。
2. 解析当前用户；未登录返回 `AUTH_UNAUTHORIZED`。
3. 校验内部 Worker 动作权限 `workshop:job_card_sync_worker`；无权限返回 `AUTH_FORBIDDEN`。
4. 校验调用主体是服务账号、系统级集成账号或 `System Manager`；不符合返回 `AUTH_FORBIDDEN`。
5. 若 `APP_ENV=production` 且 `WORKSHOP_ENABLE_WORKER_DRY_RUN=false`，立即返回 `WORKSHOP_DRY_RUN_DISABLED`，并写安全审计。
6. 只有 dry-run 未被禁用时，才允许读取 ERPNext User Permission、解析服务账号资源策略、查询 outbox 候选。

要求：

1. 生产 dry-run 禁用时，不得读取 ERPNext User Permission。
2. 生产 dry-run 禁用时，不得解析服务账号 `allowed_companies/allowed_items`。
3. 生产 dry-run 禁用时，不得查询 outbox 或 forbidden diagnostics。
4. ERPNext 权限源超时不得覆盖本地 `WORKSHOP_DRY_RUN_DISABLED`。
5. 未登录、无动作权限、非服务账号主体仍优先返回对应鉴权错误，不得被 dry-run disabled 掩盖。

### 8.9 内部 Worker 响应字段语义

`POST /api/workshop/internal/job-card-sync/run-once` 的响应必须区分主处理指标和诊断指标。

规则：

1. 主处理指标只能使用 `would_process_count/processed_count/succeeded_count/failed_count/blocked_scope_count/dead_count`。
2. 越权诊断指标只能使用 `forbidden_diagnostic_count`。
3. `skipped_forbidden_count` 和 `skipped_forbidden` 是历史兼容字段，不得作为新契约推荐给前端或运维。
4. 如本版本暂不删除兼容字段，必须在 Pydantic Field、OpenAPI description 和接口文档中标注 `deprecated`、`diagnostics-only`、`alias of forbidden_diagnostic_count`。
5. `skipped_forbidden_count/skipped_forbidden` 不得描述为“实际跳过的越权 outbox 数”。
6. 默认主处理模式下，`include_forbidden_diagnostics=false` 且全局诊断未启用时，`forbidden_diagnostic_count/skipped_forbidden_count/skipped_forbidden` 必须均为 0。
7. 显式启用诊断时，`skipped_forbidden_count/skipped_forbidden` 如保留，值只能等于 `forbidden_diagnostic_count`，并必须附带诊断语义说明。
8. 推荐新增 `forbidden_diagnostics_enabled` 或等价字段，明确本次响应是否执行了越权诊断扫描。
9. 操作审计 `after_data` 中如保留兼容字段，也必须保留规范字段，审计判断以 `forbidden_diagnostic_count` 为准。
10. 下一次破坏性 API 版本升级时，删除 `skipped_forbidden_count/skipped_forbidden` 兼容字段。

## 9. 验收标准

1. 工票登记、撤销、批量导入、查询、日薪统计接口可调用。
2. 幂等重复提交不新增记录。
3. 幂等冲突返回 `WORKSHOP_IDEMPOTENCY_CONFLICT`。
4. 撤销超量返回 `WORKSHOP_REVERSAL_EXCEEDS_REGISTERED`。
5. 日薪公式可按测试数据复算。
6. 工票单价快照不受后续工价变更影响。
7. Job Card 同步失败可留痕并重试。
8. 权限、安全审计、操作审计、日志脱敏、request_id 规范沿用 BOM 模块标准。
9. 仅授权 `ITEM-B` 的用户登记 `ITEM-A` 工票必须返回 `AUTH_FORBIDDEN`，且不得写入工票、日薪和 Job Card 同步。
10. 批量导入中越权行必须失败并进入 `failed_items`，授权行可正常处理。
11. 仅授权 `ITEM-B` 的用户读取工价列表时不得返回 `ITEM-A` 工价。
12. 只有 Company 权限、没有 Item 权限的用户不得创建指定 `ITEM-Z` 的工价。
13. 批量导入遇到数据库、权限源、审计、ERPNext 服务或未知系统异常时，必须返回非 200 标准错误响应，不得进入 `failed_items`。
14. 批量导入只有行级业务失败时，才允许返回 `200/code=0` 并携带 `success_count/failed_count`。
15. 工票登记、撤销、批量导入本地事务提交前不得调用 ERPNext Job Card 更新。
16. 本地事务提交成功后必须存在 pending outbox，Worker 成功后 outbox 变为 `succeeded` 并写 sync_log。
17. Worker 重试不得导致 ERPNext Job Card 完成数量重复累加。
18. outbox `event_key` 不得因长 `job_card` 截断造成 digest 丢失或不同事件碰撞。
19. 普通 `Workshop Manager` 调用内部 Worker 接口必须返回 `AUTH_FORBIDDEN`，且不得修改 outbox 或调用 ERPNext。
20. 只有 `workshop:job_card_sync`、没有 `workshop:job_card_sync_worker` 的用户调用内部 Worker 接口必须返回 `AUTH_FORBIDDEN`。
21. 服务账号具备 `workshop:job_card_sync_worker` 时，才允许执行内部 Worker 并写操作审计。
22. 普通用户的 `/api/auth/actions?module=workshop` 结果不得包含 `workshop:job_card_sync_worker`。
23. 服务账号没有明确 Company 或 Item 权限时，不得处理任何对应 outbox。
24. 服务账号授权 `COMPANY-A + ITEM-B` 时，不得处理 `COMPANY-A + ITEM-A` 或 `COMPANY-B + ITEM-B` 的 outbox。
25. 服务账号资源越权时不得调用 ERPNext，不得把 outbox 标记为 `succeeded`。
26. 服务账号成功处理 outbox 的操作审计必须包含资源 scope。
27. 第 1 条 outbox 越权、第 2 条 outbox 有权、`batch_size=1` 时，Worker 必须处理第 2 条有权 outbox。
28. 主处理查询必须先按服务账号 scope 过滤，再执行 `order by/limit`。
29. 缺少 `company/item_code` 的 outbox 不得长期停留在 due pending 队头。
30. 服务账号空 scope 不得退回全量扫描。
31. 默认 `run-once` 不得重复扫描当前服务账号越权 outbox。
32. 同一越权拒绝事件在冷却期内不得重复写安全审计。
33. 越权诊断不得锁定 outbox、增加 attempts、更新 `next_retry_at` 或调用 ERPNext。
34. 授权 dry-run 成功必须写操作审计。
35. 生产环境 dry-run 默认禁用，未开启时必须返回 `WORKSHOP_DRY_RUN_DISABLED` 并写安全审计。
36. dry-run 不得改变 outbox 状态、attempts、锁字段、重试时间，不得调用 ERPNext。
37. 生产 dry-run 禁用判断必须在 ERPNext User Permission 和服务账号资源策略读取之前完成。
38. 生产 dry-run 禁用且 ERPNext 权限源超时时，必须仍返回 `WORKSHOP_DRY_RUN_DISABLED`。
39. 迁移后不得存在 `item_code IS NOT NULL AND company IS NULL AND status='active'` 的工价。
40. 工票登记计薪不得命中历史 `company=NULL` 款式工价。
41. 工价列表可见集合和计薪可命中集合必须在 `item_code/company` 口径上一致。
42. `company=NULL`、`company=''`、`company='   '` 必须全部按缺失 company 处理。
43. 迁移后不得存在 active `item_code IS NOT NULL AND (company IS NULL OR btrim(company)='')` 的工价。
44. 计薪遇到缺失 company 历史候选时必须返回 `WORKSHOP_WAGE_RATE_SCOPE_REQUIRED`。
45. `backfill_wage_rate_company_scope(dry_run=True)` 后 `session.new/session.dirty/session.deleted` 必须为空。
46. dry-run 后调用方执行 `session.commit()` 不得新增补数日志、不得修改工价、不得新增审计。
47. dry-run 报告行不得是已挂 session 的 ORM 对象。
48. `_resolve_backfill_companies()` 数据库读取失败必须返回 `DATABASE_READ_FAILED` 并中断补数计划。
49. 数据库读取失败不得被归类为 `WORKSHOP_WAGE_RATE_COMPANY_AMBIGUOUS` 或 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
50. execute 读取失败不得产生部分补数、部分日志或部分审计。
51. ERPNext Item 候选查询不可用时必须返回 `ERPNEXT_SERVICE_UNAVAILABLE` 或 `DATABASE_READ_FAILED`，不得返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
52. ERPNext Item 候选查询不可用时不得停用历史工价，不得写补数日志，不得返回成功 dry-run 报告。
53. ERPNext Item 查询成功且确认无候选时，仍可按业务规则返回 `WORKSHOP_WAGE_RATE_COMPANY_UNRESOLVED`。
54. 内部 Worker 响应必须将 `forbidden_diagnostic_count` 作为越权诊断唯一推荐字段。
55. `skipped_forbidden_count/skipped_forbidden` 如保留，必须标注 deprecated、diagnostics-only，且不得在接口文档中描述为主处理跳过数。
56. 默认主处理模式未启用诊断时，`forbidden_diagnostic_count/skipped_forbidden_count/skipped_forbidden` 必须全部为 0。
